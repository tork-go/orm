# Introduction

Tork is a Go ORM. You declare your tables as typed Go values, and the framework
uses them for two things: generating pure-SQL migrations, and (in a later release)
running typed queries against them.

A model is a struct that embeds `orm.Table` and declares its columns as typed
fields. There is no code generation step and no struct tags to keep in sync: the
column's Go type is the source of truth for its database type, and a pointer type
means the column is nullable.

```go
package models

import "github.com/tork-go/orm"

type UserModel struct {
	orm.Table
	ID       *orm.Column[int]
	Username *orm.Column[string]
	Email    *orm.Column[*string] // pointer type means nullable
}

var User = &UserModel{
	Table:    orm.NewTable("users"),
	ID:       orm.NewColumn[int]("id").PrimaryKey(),
	Username: orm.NewColumn[string]("username").Unique().NotNull().MaxLen(30),
	Email:    orm.NewColumn[*string]("email"),
}
```

That is a complete, valid model. From here, Tork can compare it against a live
database and write the migration that creates the `users` table.

## What you get

- **Typed columns.** `Column[int]`, `Column[string]`, `Column[*string]`, and so on.
  Your IDE knows the Go type of every column, and Tork derives the SQL type from
  it, so there is nothing to keep in sync by hand.
- **Migrations generated from your models.** Run `makemigrations` and Tork
  introspects the live database, diffs it against your models, and writes a
  plain `.sql` file with an `up` and a `down` section, in the same style as
  Alembic or Drizzle.
- **Migrations you can read and edit.** A migration is SQL, not a serialized
  operation log. You can open the file, read exactly what it does, and hand-edit
  it if you need to.
- **A programmatic apply, for your app's startup.** `migrate.Apply` connects,
  loads pending migrations, and brings the database up to date in one call,
  the same role as SQLAlchemy's `metadata.create_all(engine)` or Drizzle's
  `migrate(db, {...})`.
- **A CLI, built as a library.** Go cannot import an arbitrary package at
  runtime, so Tork does not ship a single global binary. Instead you write a
  four-line `main.go` that imports your models and Tork's CLI dispatcher, and
  compiling it gives you `migrate up`, `migrate down`, and `history` commands
  for your own project.

## What is not built yet

Tork is being built in stages. Model definitions and migrations are done.
A typed query API (`SELECT`, `INSERT`, `UPDATE`, joins, and loading
relationships like `HasMany`/`BelongsTo`) has not been built yet. This round of
documentation covers what exists today: defining models and generating and
running migrations against PostgreSQL.
