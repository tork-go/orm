# Quick Start

This walks through the whole loop: define models, generate a migration, apply
it, and confirm it worked. It assumes you have a Go module set up and a
PostgreSQL database reachable at the connection string from
[Installation](/docs/1.getting-started/2.installation).

## 1. Define your models

Create `models/models.go`. This declares two tables: `users`, and `posts` with a
foreign key back to `users`.

```go
package models

import "github.com/tork-go/orm"

// User is the row type: what a query returns, and what you construct when
// you have data to insert.
type User struct {
	ID       int
	Username string
	Email    *string
}

// UserModel describes the table. Its fields are columns, not values.
type UserModel struct {
	orm.Table[User]
	ID       *orm.IntColumn
	Username *orm.StringColumn
	Email    *orm.NullableStringColumn
	Posts    orm.HasMany[Post]
}

var Users = orm.DefineTable[User]("users", func(t *orm.TableBuilder[User]) *UserModel {
	return &UserModel{
		Table:    t.Table(),
		ID:       t.Int("id").PrimaryKey(),
		Username: t.String("username").Unique().NotNull().MaxLen(30),
		Email:    t.NullableString("email"),
	}
})

type Post struct {
	ID       int
	Title    string
	Content  string
	AuthorID int
}

type PostModel struct {
	orm.Table[Post]
	ID       *orm.IntColumn
	Title    *orm.StringColumn
	Content  *orm.StringColumn
	AuthorID *orm.IntColumn
	Author   orm.BelongsTo[User]
}

var Posts = orm.DefineTable[Post]("posts", func(t *orm.TableBuilder[Post]) *PostModel {
	return &PostModel{
		Table:    t.Table(),
		ID:       t.Int("id").PrimaryKey(),
		Title:    t.String("title").NotNull().MaxLen(100),
		Content:  t.String("content").NotNull(),
		AuthorID: t.Int("author_id").NotNull().References(Users.ID),
	}
})
```

Two types per table, and they do different jobs. `User` is a plain struct: one
row. `UserModel` describes the table, and its fields are columns you build
queries from. The singular name is the row, the plural is the table, so
`User{Username: "alice"}` and `Users.Username.Eq("alice")` never get confused
for each other.

`DefineTable` is what ties them together. It binds each column to the table and
matches it to the field of `User` it scans into, by snake-casing the field name
(`AuthorID` finds `author_id`) or by a `db:"..."` tag. A column with no matching
field is a mistake in your source, so it panics at startup rather than failing
on your first query.


`Posts` and `Author` are left uninitialized on purpose. That's what lets
`UserModel` mention `Post` even though `Posts` is declared after it; naming the
other table in the literal would make the two variables depend on each other,
which Go rejects outright.

Tork works out which foreign key joins the two tables by reading the schema, so
`Users.Posts.Relation()` reports that `users.id` matches `posts.author_id`.
Actually loading the related rows is part of the query API, which doesn't exist
yet. See [Relationships](/docs/2.fundamentals/5.relationships).

## 2. Write a migration entrypoint

Go can't import an arbitrary package at runtime, so Tork's CLI is a library, not
a standalone binary. A small `main.go` links your models into it. Create
`cmd/migrate/main.go`:

```go
package main

import (
	"os"

	"github.com/tork-go/orm/driver/postgres"
	"github.com/tork-go/orm/migrate/cli"

	"yourmodule/models"
)

func main() {
	os.Exit(cli.Run(postgres.Dialect{}, os.Getenv("DATABASE_URL"), "migrations",
		models.Users, models.Posts))
}
```

Replace `yourmodule/models` with your actual module path. Set `DATABASE_URL` to
the connection string from [Installation](/docs/1.getting-started/2.installation):

```sh
export DATABASE_URL="postgres://tork:tork@localhost:5432/tork_dev?sslmode=disable"
```

## 3. Generate a migration

```sh
go run ./cmd/migrate makemigrations -m create_users_and_posts
```

```
Wrote revision 2773adc3c811
```

Tork connected, introspected the (currently empty) database, compared it
against `User` and `Post`, and wrote
`migrations/2773adc3c811_create_users_and_posts.sql` (your revision id will be
different, it's random):

```sql
-- revision: 2773adc3c811
-- down_revision: none
-- migrate:up
CREATE TABLE "posts" (
    "id" INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    "title" VARCHAR(100) NOT NULL,
    "content" TEXT NOT NULL,
    "author_id" INTEGER NOT NULL
);

CREATE TABLE "users" (
    "id" INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    "username" VARCHAR(30) NOT NULL,
    "email" TEXT,
    CONSTRAINT "uq_users_username" UNIQUE ("username")
);

ALTER TABLE "posts" ADD CONSTRAINT "fk_posts_author_id" FOREIGN KEY ("author_id") REFERENCES "users" ("id");

-- migrate:down
ALTER TABLE "posts" DROP CONSTRAINT "fk_posts_author_id";

DROP TABLE "posts";

DROP TABLE "users";
```

This is plain, valid PostgreSQL DDL. Read it, and if you ever need to, edit it.
Notice the foreign key is added as its own statement after both tables exist,
not inlined into `CREATE TABLE posts`, so this works regardless of which table
you declared first.

## 4. Apply it

```sh
go run ./cmd/migrate migrate up head
```

```
OK
```

`head` means "every pending migration." Check what's applied:

```sh
go run ./cmd/migrate history
```

```
2773adc3c811  applied 2026-07-20 22:05:17
```

## 5. Roll it back, if you need to

```sh
go run ./cmd/migrate migrate down base
```

`base` means "everything," the inverse of `head`. `history` now shows the
revision as `pending` again, and the tables are gone.

## Applying migrations from your app

The CLI is one way to apply migrations. The other is a single function call from
your application's own startup code, `migrate.Apply`, which plays the same role
as SQLAlchemy's `metadata.create_all(engine)` or Drizzle's `migrate(db, {...})`:

```go
package main

import (
	"context"
	"log"
	"os"

	"github.com/tork-go/orm/driver/postgres"
	"github.com/tork-go/orm/migrate"
)

func main() {
	ctx := context.Background()
	dsn := os.Getenv("DATABASE_URL")

	if err := migrate.Apply(ctx, postgres.Dialect{}, dsn, "migrations"); err != nil {
		log.Fatal(err)
	}

	// ... start your application
}
```

`Apply` connects, loads whatever's in `migrations/`, and brings the database up
to `head`. Call it every time your app starts: if nothing is pending, it's a
no-op.
