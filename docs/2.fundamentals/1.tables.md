# Tables

Every model embeds `orm.Table`. It's the minimum a type needs to be a model: a
database table name.

```go
type UserModel struct {
	orm.Table
	// columns go here
}

var User = &UserModel{
	Table: orm.NewTable("users"),
}
```

`Table` is embedded by value, not by pointer, so a model struct stays a plain
composite literal. `NewTable` takes the table name and returns a `Table`:

```go
func NewTable(name string) Table
func (t Table) TableName() string
```

Because `Table` is embedded, `TableName()` is promoted onto your model:

```go
User.TableName() // "users"
```

## Table is a name, nothing else

`Table` doesn't know its own columns, and it doesn't register itself anywhere
when constructed. There's no global registry to worry about, and no import-order
dependency between your models. When something needs to know a model's columns
(migrations, eventually queries), it gets them by reflecting over the model's
own fields, covered in
[Model Reflection](/docs/2.fundamentals/6.model-reflection). `Table`'s only job
is to answer `TableName()`.

## The `Model` interface

Anything with a `TableName() string` method satisfies `orm.Model`:

```go
type Model interface {
	TableName() string
}
```

Embedding `Table` gives you this for free. Functions that work with models
generically, like `schema.ExtractSchema` or `migrate/cli`'s `Run`, take
`orm.Model` (or `...orm.Model`) rather than a concrete struct type, which is why
you can pass `User` and `Post` to the same call even though they're different
Go types.
