# Tables

Every model embeds `orm.Table[E]`, where `E` is the row type. That pairing is
the minimum a type needs to be a model: a database table name, and the struct
its rows scan into.

Declare one with `DefineTable`:

```go
type User struct {
	ID       int
	Username string
}

type UserModel struct {
	orm.Table[User]
	ID       *orm.IntColumn
	Username *orm.StringColumn
}

var Users = orm.DefineTable[User]("users", func(t *orm.TableBuilder[User]) *UserModel {
	return &UserModel{
		Table:    t.Table(),
		ID:       t.Int("id").PrimaryKey(),
		Username: t.String("username").Unique().NotNull(),
	}
})
```

Only `E` is written out. The model type is inferred from what the function
returns, so it can be a pointer or a value as you prefer.

`Table` is embedded by value, so a model struct stays a plain composite literal,
and `TableName()` is promoted onto your model:

```go
Users.TableName() // "users"
```

## Two types per table

The row type and the model are separate on purpose. `User` is one row: what a
query gives you back, and what you fill in to insert. `UserModel` describes the
table, and its fields are columns you build predicates from.

Name the row type in the singular and the table in the plural, and the two never
get confused at a call site:

```go
user := User{Username: "alice"}         // a row
Users.Username.Eq("alice")              // a column
```

## What `DefineTable` does

Three things a hand-built model doesn't get:

1. **Binds each column to the table**, so `OwnerTable()` reports it and
   generated SQL can qualify names.
2. **Records the columns in struct field order**, which fixes the order a
   `SELECT` lists them. Scanning is positional, and the driver exposes no
   column names, so that ordering is what maps a result row back to fields.
3. **Matches each column to a field of `E`.** A column takes the field tagged
   `db:"<column name>"` if there is one, otherwise the field whose name
   snake-cases to the column name: `AuthorID` finds `author_id`, `CreatedAt`
   finds `created_at`. Fields tagged `db:"-"` are skipped, and fields matching
   no column are left alone, which is what lets a row type carry related rows
   and computed values alongside its columns.

A column with no matching field, or one whose field has the wrong type, panics:

```
orm: table "users": column "email_address" has no field on models.User
    (looked for a field whose name snake-cases to "email_address", or one tagged `db:"email_address"`)
```

That's deliberate. `DefineTable` runs in a package-level `var`, where an error
return is something no caller could meaningfully check, so it follows
`regexp.MustCompile`: the mistake is in your source, the program can't run
correctly with it, and it surfaces at startup instead of on your first query.

## Tables without a row type

A model that only describes a table's shape, whether in a migration-only
program or a test, has no row struct to name. Say so with `NoEntity` and build it with
`NewTable`, which skips the entity mapping entirely:

```go
type LegacyModel struct {
	orm.Table[orm.NoEntity]
	ID *orm.IntColumn
}

var Legacy = &LegacyModel{
	Table: orm.NewTable[orm.NoEntity]("legacy"),
	ID:    orm.NewIntColumn("id"),
}
```

Such a model works for migrations but can't back a query: there's no type for
results to scan into, and its columns are never bound to the table.

## Table is a name, nothing else

`Table` doesn't register itself in any global registry, so there's no
import-order dependency between your models. When something needs a model's
columns (migrations, queries) it gets them by reflecting over the model's own
fields, covered in
[Model Reflection](/docs/2.fundamentals/6.model-reflection).

## The `Model` interface

Anything with a `TableName() string` method satisfies `orm.Model`:

```go
type Model interface {
	TableName() string
}
```

Embedding `Table` gives you this for free. Functions that work with models
generically, like `schema.ExtractSchema` or `migrate/cli`'s `Run`, take
`orm.Model` (or `...orm.Model`) rather than a concrete struct type, which is why
you can pass `Users` and `Posts` to the same call even though they're different
Go types.
