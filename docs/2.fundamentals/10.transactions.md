# Transactions

```go
err := db.Transaction(ctx, func(tx *orm.DB) error {
    if err := Users.With(tx).Insert(ctx, user); err != nil {
        return err
    }
    post.AuthorID = user.ID
    return Posts.With(tx).Insert(ctx, post)
})
```

Return `nil` and the transaction commits. Return an error and it rolls back, and
that error is what `Transaction` returns, unwrapped, so `errors.Is` against your own
sentinel works.

## The handle is an ordinary `*orm.DB`

`tx` is not a special type with a reduced API. It is a `*orm.DB` like the one you
started with, so everything works inside unchanged:

```go
err := db.Transaction(ctx, func(tx *orm.DB) error {
    recent, err := Posts.With(tx).Where(Posts.CreatedAt.GreaterThan(cutoff)).All(ctx)
    if err != nil {
        return err
    }

    if err := Users.With(tx).InsertMany(ctx, users...); err != nil {
        return err
    }

    _, err = Posts.With(tx).Where(Posts.Draft.Equals(true)).DeleteAll(ctx)
    return err
})
```

That works because the statement surface a `DB` holds is an `Execer`, which both a
connection and an open transaction satisfy. Moving existing code inside a
transaction is a matter of passing `tx` where you passed `db`.

## A panic rolls back too

```go
err := db.Transaction(ctx, func(tx *orm.DB) error {
    mustNotPanic(tx)   // but does
    return nil
})
```

The transaction is rolled back and the panic continues unwinding to your own
recover, if you have one. Letting the panic out with the transaction still open
would hold whatever locks its statements had taken until the connection closed.

## Nested calls join rather than nest

A handle that is already a transaction has no connection to start a second one from,
so an inner `Transaction` runs its callback against the existing one:

```go
err := db.Transaction(ctx, func(tx *orm.DB) error {
    return doWork(ctx, tx)   // may itself call tx.Transaction
})
```

This is what makes composition safe: a bulk write opens a transaction of its own
(see [Bulk Operations](/docs/2.fundamentals/9.bulk-operations)), and calling one
inside your transaction must not deadlock or commit halfway.

::alert{type="warning"}
The consequence is that an inner call returning an error rolls back **the whole
outer transaction**, not just its own part. Real nesting means savepoints, which no
driver method exposes yet.
::

## Bulk operations are already atomic

You do not need to wrap a bulk operation yourself. One that runs more than one
statement opens a transaction of its own:

```go
// already all-or-nothing, however many statements it takes
err := Users.With(db).InsertMany(ctx, tenThousandUsers...)
```

Wrap them when you want *several* operations to succeed or fail together.

## Locking the rows you read

A read followed by a write is two statements, and something else can happen in
between. `ForUpdate` closes that window: it holds the rows the read returned until
the transaction ends.

```go
err := db.Transaction(ctx, func(tx *orm.DB) error {
    account, err := Accounts.With(tx).Where(Accounts.ID.Equals(id)).ForUpdate().First(ctx)
    if err != nil {
        return err
    }
    account.Balance -= amount
    return Accounts.With(tx).Update(ctx, account)
})
```

```sql
SELECT "id", "balance" FROM "accounts" WHERE "id" = $1 LIMIT 1 FOR UPDATE
```

Without it, two transfers reading the same balance both see the old number and the
second write silently overwrites the first. With it, the second reader waits until
the first transaction ends and then reads what it did.

The lock is held **until the transaction ends**, which is the point — and the reason
a lock outside one buys nothing, since a lone statement is already a transaction and
it is over by the time the rows come back.

### `SkipLocked`: turning a table into a work queue

```go
jobs, err := Jobs.With(tx).
    Where(Jobs.State.Equals("queued")).
    OrderBy(Jobs.CreatedAt.Asc()).
    ForUpdate().SkipLocked().
    Limit(10).
    All(ctx)
```

```sql
SELECT ... FROM "jobs" WHERE "state" = $1 ORDER BY "created_at" ASC
    LIMIT 10 FOR UPDATE SKIP LOCKED
```

Ten workers running exactly this get ten *different* batches. The rows another
transaction is holding are passed over rather than waited for, so nobody queues
behind anybody and nothing has to coordinate.

A query asking for ten may return fewer. That is the answer, not a shortfall: the
rows it skipped belong to someone else right now.

### The four calls

| | |
| --- | --- |
| `ForUpdate()` | exclusive — blocks every other reader and writer |
| `ForShare()` | shared — other readers may take the same lock, writers may not |
| `SkipLocked()` | pass over rows someone else holds |
| `NoWait()` | fail rather than wait for them |

`ForShare` is for a reader that needs a row to stay as it is without intending to
change it. `NoWait` is for the caller who would rather report contention than sit
behind it, where `SkipLocked` is for the one who would rather take different rows.

The two halves are one clause and compose in either order. `SkipLocked` or `NoWait`
without a mode is an error — SQL has no way to say what to do about a locked row on
a read that is not locking any.

### Where a lock is refused

Anywhere the clause would have to be dropped, Tork refuses instead rather than
running an unlocked statement where you wrote a locked one:

```go
Users.With(tx).Where(...).ForUpdate().Count(ctx)      // error: nothing left to lock
Users.With(tx).Distinct().ForUpdate().All(ctx)        // error: duplicates collapsed
Users.With(tx).Where(...).ForUpdate().DeleteAll(ctx)  // error: a write locks by writing
```

`Count`, the aggregates and the grouped forms collapse the rows into a value, so
there is no row left to lock — lock the read that returns the rows instead. PostgreSQL
rejects `FOR UPDATE` beside `DISTINCT` outright, and Tork says so first, naming what
you wrote rather than what the statement became.

SQLite locks whole databases rather than rows. A driver with no clause to give
returns an error naming the operation.

### The optimistic alternative

A lock is the pessimistic answer: hold the row so nobody else can change it.
`UpdateIf` is the optimistic one — do not hold anything, but say what the row
looked like when it was read and fail if that is no longer true.

```go
seen := account.Version
account.Version++
account.Balance -= amount

err := Accounts.With(db).UpdateIf(ctx, account, Accounts.Version.Equals(seen))
if errors.Is(err, orm.ErrNoRows) {
    // re-read and try again
}
```

It costs nothing while nothing contends, needs no transaction, and does not hold
a row across a round trip. It costs a retry when something does. Reach for a lock
when contention is the norm and for `UpdateIf` when it is the exception. See
[Writing only if the row has not
changed](/docs/2.fundamentals/8.querying#writing-only-if-the-row-has-not-changed).

## Doing it yourself

`Transaction` is the shape to reach for. If you need the transaction to outlive a
single function, `Conn` exposes the underlying primitives:

```go
type Conn interface {
    Execer
    Begin(ctx context.Context) (Tx, error)
    Close(ctx context.Context) error
}

type Tx interface {
    Execer
    Commit(ctx context.Context) error
    Rollback(ctx context.Context) error
}
```

You are then responsible for committing or rolling back on every path, including
panics, which is the work `Transaction` exists to do for you.

## Shorthand for your own project

Tork has no code generation step, so it cannot give you a `tx.Users` field: `Users`
is a variable in *your* package that Tork has never seen. If you want that shape,
it is a few lines written once in the package your models live in:

```go
package models

import (
    "context"

    "github.com/tork-go/orm"
)

// Tx is your models, bound to one transaction.
type Tx struct{ DB *orm.DB }

func (t Tx) Users() *orm.Query[User] { return Users.With(t.DB) }
func (t Tx) Posts() *orm.Query[Post] { return Posts.With(t.DB) }

// Transaction runs fn against every model at once.
func Transaction(ctx context.Context, db *orm.DB, fn func(Tx) error) error {
    return db.Transaction(ctx, func(d *orm.DB) error { return fn(Tx{DB: d}) })
}
```

Which reads as:

```go
err := models.Transaction(ctx, db, func(tx models.Tx) error {
    if err := tx.Users().Insert(ctx, user); err != nil {
        return err
    }
    post.AuthorID = user.ID
    return tx.Posts().Insert(ctx, post)
})
```

Adding a model is one more line, and it stays fully typed: `tx.Users()` is a
`*orm.Query[User]`, so the compiler still rejects `tx.Users().Insert(ctx, post)`.
