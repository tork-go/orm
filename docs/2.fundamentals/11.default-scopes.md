# Default Scopes

A model can declare a predicate that every query against its table carries
automatically, on top of whatever `Where` a caller adds:

```go
func (m *PostModel) DefaultScope() orm.Predicate {
    return m.Published.Eq(true)
}
```

```go
Posts.With(db).All(ctx)
// SELECT ... FROM "posts" WHERE "published" = $1
```

Implementing `DefaultScope() orm.Predicate` is what opts a model in — there is
no method to call to enable it, the same way `Indexes()` and `Checks()` are
detected rather than registered. See [Model
Reflection](/docs/2.fundamentals/6.model-reflection).

## What it applies to

Everything that reads or writes rows through a condition:

```go
Posts.With(db).Count(ctx)                                    // scoped
Posts.With(db).Exists(ctx)                                    // scoped
Posts.With(db).Where(Posts.Title.Contains("go")).All(ctx)     // scoped, ANDed with Where
Posts.With(db).UpdateAll(ctx, Posts.Title.Set("x"))            // scoped
Posts.With(db).DeleteAll(ctx)                                  // scoped
```

The scope is always **ANDed** with whatever conditions a query already
carries, never replacing them:

```go
Posts.With(db).Where(Posts.Title.Contains("go")).SQL()
// WHERE ("title" LIKE $1 AND "published" = $2)
```

It is not consulted for the entity-identified operations on `Query[E]` —
`Insert`, `Update`, `Delete`, `Save` — which act on a `*E` you are already
holding by primary key. Those have nothing to scope: the row is named
directly, not found by a condition.

## Dropping it: `Unscoped`

```go
Posts.With(db).Unscoped().All(ctx)
// SELECT ... FROM "posts"   — no WHERE at all
```

`Unscoped` disables the default scope for one query, and for everything that
query reaches into — see below. It does not disable a `Where` you wrote
yourself:

```go
Posts.With(db).Where(Posts.Title.Contains("go")).Unscoped().SQL()
// WHERE "title" LIKE $1
```

## It reaches into `Load` and `Has`

A related table's default scope applies the same way whether you query it
directly, load it, or ask about it with `Has`:

```go
authors, err := Authors.With(db).Load(Authors.Posts).All(ctx)
// the second statement carries Posts' own default scope:
// SELECT ... FROM "posts" WHERE ("author_id" IN (...) AND "published" = $1)

Authors.With(db).Where(orm.Has(Authors.Posts)).All(ctx)
// EXISTS (SELECT 1 FROM "posts"
//         WHERE "posts"."author_id" = "authors"."id" AND "posts"."published" = $1)
```

That last one applies even when `Has` was given no conditions of its own —
the scope alone is reason enough to add the check. `Unscoped` on the outer
query reaches into both the same way:

```go
Authors.With(db).Unscoped().Load(Authors.Posts).All(ctx)
Authors.With(db).Unscoped().Where(orm.Has(Authors.Posts)).All(ctx)
// neither carries Posts' default scope
```

::alert{type="info"}
**A many-to-many's join table carries no scope of its own.** `Has` over a
`ManyToMany` becomes two `EXISTS`, one for the join table and one for the far
table (see [Filtering by related
rows](/docs/2.fundamentals/8.querying#filtering-by-related-rows)); only the
far table's default scope reaches in. A join table declaring its own
`DefaultScope` would be unusual, and nothing here supports it.
::

## The guard against writing a whole table stays honest

`UpdateAll` and `DeleteAll` refuse a `Where` that compiled to nothing — see
[No `Where` means every
row](/docs/2.fundamentals/9.bulk-operations#no-where-means-every-row). That
check looks only at what you wrote:

```go
conds := buildFilters(req)                          // came back empty
Posts.With(db).Where(conds...).DeleteAll(ctx)
```

```
orm: table "posts": Where added no condition, so DeleteAll would delete
    every row; if that is what you meant, call DeleteAll without a Where
```

A table's own default scope is **never** allowed to satisfy this guard on
your behalf. Without that rule, a scoped table would make the empty-filter
mistake silently pass — the compiled statement would have a `WHERE` in it,
just not the one you meant to write — which is exactly backwards: the tables
worth guarding hardest are the ones with a scope, since a soft-delete table
(see next) is what `DeleteAll` usually means to touch.
