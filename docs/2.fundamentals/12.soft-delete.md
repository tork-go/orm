# Soft Delete

A nullable timestamp column can be marked as a table's soft-delete marker:

```go
type PostModel struct {
    orm.Table[Post]
    ID        *orm.IntColumn
    DeletedAt *orm.NullableTimeColumn
}

var Posts = orm.DefineTable[Post]("posts", func(t *orm.TableBuilder[Post]) *PostModel {
    return &PostModel{
        Table:     t.Table(),
        ID:        t.Int("id").PrimaryKey(),
        DeletedAt: t.NullableTime("deleted_at").SoftDelete(),
    }
})
```

`SoftDelete` exists only on `NullableTimeColumn` — the same restriction that
keeps `MaxLen` off an int column keeps this off anything that is not a
nullable timestamp, so there is nothing to validate at declaration time. It
is an ordinary column otherwise: `deleted_at` is a plain nullable `TIMESTAMP`
as far as migrations and introspection are concerned, not a new kind of DDL.

A table may declare at most one. Two is ambiguous — nothing says which one
`Delete` should stamp — and `DefineTable` panics naming both:

```
orm: table "posts": declares 2 soft-delete columns ("deleted_at", "removed_at");
    only one is allowed
```

## `Delete` becomes an `UPDATE`

```go
err := Posts.With(db).Delete(ctx, post)
// UPDATE "posts" SET "deleted_at" = $1 WHERE "id" = $2
```

The stamped value is `time.Now()`, bound as an ordinary parameter — not a SQL
`NOW()` — the same way every other value in the query API is. `DeleteIf`,
`DeleteAll` and `DeleteMany` all rewrite the same way:

```go
n, err := Posts.With(db).Where(Posts.AuthorID.Equals(7)).DeleteAll(ctx)
// UPDATE "posts" SET "deleted_at" = $1 WHERE ("author_id" = $2 AND "deleted_at" IS NULL)

n, err := Posts.With(db).DeleteMany(ctx, posts...)
// UPDATE "posts" SET "deleted_at" = $1 WHERE "id" IN (...)
```

Hooks fire exactly as they do for an ordinary delete — `BeforeDelete` and
`AfterDelete` on `Delete`/`DeleteIf`/`DeleteMany`, nothing on `DeleteAll` —
see [Hooks](/docs/2.fundamentals/8.querying#hooks). Only the SQL changes.

## Marked rows are hidden by default

A soft-delete column contributes an implicit [default
scope](/docs/2.fundamentals/11.default-scopes): every read excludes a row
where it is set, the same as a `Scoper`-declared one would, and the two
combine when a model has both:

```go
Posts.With(db).All(ctx)
// WHERE "deleted_at" IS NULL

Posts.With(db).Unscoped().All(ctx)
// no WHERE at all — deleted rows included
```

That implicit scope is why `DeleteAll` above carried `"deleted_at" IS NULL`
in its `WHERE` alongside the caller's own condition: a soft delete never
touches a row that is already marked, the same way a read never returns one.

## `ForceDelete` for when you mean it

```go
err := Posts.With(db).ForceDelete(ctx, post)
// DELETE FROM "posts" WHERE "id" = $1
```

`ForceDeleteAll` and `ForceDeleteMany` are the same escape hatch for the
batch forms, and `ForceDeleteAllReturning` reads the removed rows back. All
three fire the same hooks their ordinary counterparts do.

::alert{type="warning"}
**`Force` changes the verb, not the visibility.** `ForceDeleteAll` alone
still only reaches rows the default scope shows you — a row already marked
by an earlier `Delete` stays untouched, because the implicit
`"deleted_at" IS NULL` is still in its `WHERE`:

```go
Posts.With(db).ForceDeleteAll(ctx)
// DELETE FROM "posts" WHERE "deleted_at" IS NULL
```

To purge everything, including rows already soft-deleted, chain `Unscoped`
in front of it:

```go
Posts.With(db).Unscoped().ForceDeleteAll(ctx)
// DELETE FROM "posts"   — no WHERE at all
```
::
