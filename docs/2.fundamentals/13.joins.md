# Joins

```go
users, err := Users.With(db).
    Join(Users.Posts).
    Where(Posts.Title.Contains("go")).
    All(ctx)
```

```sql
SELECT "users"."id", "users"."username", ... FROM "users"
JOIN "posts" ON "posts"."author_id" = "users"."id"
WHERE "posts"."title" LIKE $1
```

`Join` correlates on the relationship's own foreign key — the same correlation
[`Has`](/docs/2.fundamentals/8.querying#filtering-by-related-rows) already uses,
rendered as a real `JOIN` instead of a correlated `EXISTS`. `LeftJoin` is the same,
keeping a primary row with no matching related row:

```go
Users.With(db).LeftJoin(Users.Posts).Where(Posts.ID.IsNull()).All(ctx)
// users with no posts
```

## Why this exists beside `Has` and `Load`

`Has` and `HasNone` answer *whether* a related row is there, without reading
anything about it. `Load` reads the related rows, in a statement of their own,
after the parent rows are already in hand. Neither can put a column of the
related table into a filter, an order, or a projection that reads both tables at
once — "each author's book count" needs one statement covering both tables, and
`Join` is what gives the statement that.

## `JoinOn` and `LeftJoinOn`: conditions on the join, not the row

```go
Users.With(db).
    JoinOn(Users.Posts, Posts.Published.Equals(true)).
    All(ctx)
```

```sql
JOIN "posts" ON "posts"."author_id" = "users"."id" AND "posts"."published" = $1
```

The difference from `Where` matters most with a left join, which is what
`LeftJoinOn` is for: a condition in `Where` runs *after* the join, so it drops
the very rows a left join exists to keep. A condition ANDed onto `ON` instead is
checked as part of matching related rows, so every user comes back, matched
against only their published posts, or none:

```go
Users.With(db).
    LeftJoinOn(Users.Posts, Posts.Published.Equals(true)).
    Where(Posts.ID.IsNull()).
    All(ctx)
// users with no published post — including one with only drafts
```

## One table, named once

A statement cannot name the same table twice, which is what a self-referencing
relationship would do:

```go
Employees.With(db).Join(Employees.Manager).All(ctx)
```

```
orm: table "employees": this join names table "employees", which the statement
    already reads from; give one side a name of its own with orm.Alias and join
    it with JoinAs
```

Give the far side a name and the join is ordinary again:

```go
mgr := orm.Alias(Employees, "mgr")
Employees.With(db).JoinAs(Employees.Manager, mgr).Where(mgr.Name.Equals("Ada"))
```

```sql
JOIN "employees" AS "mgr" ON "mgr"."id" = "employees"."manager_id"
WHERE "mgr"."name" = $1
```

`JoinAs`, `LeftJoinAs`, `JoinOnAs` and `LeftJoinOnAs` are the four forms above,
each taking a name for the table it joins. See
[Aliases](/docs/2.fundamentals/18.aliases).

## `JoinTo`: no relationship, just conditions

```go
Users.With(db).
    JoinTo(Logins, Logins.UserID.Value().Equals(Users.ID)).
    Where(Logins.Failed.Equals(true))
```

```sql
JOIN "logins" ON "logins"."user_id" = "users"."id" WHERE "logins"."failed" = $1
```

Every form above takes a relationship, which supplies the keys to match on.
`JoinTo` and `LeftJoinTo` take the conditions instead, for two tables with no
relationship declared between them. At least one condition is required: a join
with none is a cross join, which this package does not offer.

## Once you join, everything is qualified

A statement with no `Join` names its columns bare — `"id"`, `"username"`. Once a
`Join` is present, every column is qualified, including the primary table's own,
since a column name can collide across joined tables:

```go
Books.With(db).Join(Books.Author).SQL()
```

```sql
SELECT "books"."id", "books"."author_id", "books"."title" FROM "books"
JOIN "authors" ON "authors"."id" = "books"."author_id"
```

Refer to the joined table's columns through its own model, the same way any
other column is written:

```go
Books.With(db).Join(Books.Author).Where(Authors.Name.Equals("Le Guin")).OrderBy(Authors.Name.Asc())
```

## `Select` stays primary-table-only

```go
Users.With(db).Join(Users.Posts).Select(Posts.Title).All(ctx)
```

```
orm: table "users": Select column 0 belongs to table "posts", which this
    statement does not select from; read a joined table's columns with
    SelectAs instead
```

The rows a `Join`ed read scans into are still `*User`, and `User`'s fields are
mapped from the `users` table's own columns only — there is no field on `User`
a `posts` column could fill. Reading a joined table's columns, alone or
alongside the primary table's, needs
[`SelectAs`](/docs/2.fundamentals/14.projections), which reads into a type of
your own choosing rather than `*E`.

## `Count` after a `Join` counts joined rows

```go
Users.With(db).Join(Users.Posts).Count(ctx)
```

```sql
SELECT COUNT(*) FROM "users" JOIN "posts" ON "posts"."author_id" = "users"."id"
```

A user with three matching posts is counted three times — exactly what raw SQL
would count, and not a bug this package works around. Adding a silent `DISTINCT`
would be wrong exactly as often as it would be right. `Has`/`HasNone`, built on
`EXISTS`, are what to reach for when the question is how many primary rows have
a match, not how many matches there are.

## No many-to-many

```go
Books.With(db).Join(Books.Tags).All(ctx)
```

```
orm: table "books": Join does not support a many to many relationship, which
    needs two joins; use Has or HasNone instead
```

A many-to-many needs two joins through a join table, multiplying rows in a way
`Has`/`HasNone`, built on two nested `EXISTS`, never do — and this package
already has an answer for the question a many-to-many join usually means to ask.

## What a `Join` cannot carry

`UpdateAll`, `DeleteAll`, `Scalars` (`orm.Select`), `Grouped` (`orm.CountBy` and
friends) and the scalar aggregates (`orm.Sum`, `orm.Avg`, `orm.Min`, `orm.Max`)
all reject a query carrying a `Join`:

```go
orm.Select(Users.With(db).Join(Users.Posts), Users.Username)
// error: Select cannot run over a query carrying a Join or LeftJoin
```

No dialect Tork targets writes a portable `UPDATE` or `DELETE` with a `JOIN` in
it, and the aggregate functions collapse rows into a single value, where a join
would first have multiplied them. `Filtered`'s own reads (`All`, `First`,
`Count`) and `SelectAs` are what support a `Join`.
