# Projections

```go
type AuthorReport struct {
    Name      string
    BookCount int64
}

reports, err := orm.SelectAs[AuthorReport](
    Authors.With(db).LeftJoin(Authors.Books),
    Authors.Name,
    orm.CountOf(Books.ID),
).GroupBy(Authors.Name).All(ctx)
```

```sql
SELECT "authors"."name", COUNT("books"."id") FROM "authors"
LEFT JOIN "books" ON "books"."author_id" = "authors"."id"
GROUP BY "authors"."name"
```

This is the question [Joins](/docs/2.fundamentals/13.joins) exists to make
answerable in one round trip: each author, with how many books they have —
neither a plain read (one table) nor
[`orm.CountBy`](/docs/2.fundamentals/8.querying#grouping) (one aggregate, no
joined column) can express it.

## Fields are matched positionally, not by name

`AuthorReport`'s exported fields, in declaration order, line up one for one
with the expressions passed to `SelectAs` — `Name` with `Authors.Name`,
`BookCount` with `orm.CountOf(Books.ID)`. There is no string aliasing, which
matches every other scan in this package: `driver.Rows` exposes no column
names, so nothing here is ever matched by name.

A count or type mismatch is reported when the statement compiles, not as a
scan failure at run time:

```go
type Wrong struct{ Name string }
orm.SelectAs[Wrong](Authors.With(db), Authors.Name, orm.CountOf(Books.ID))
```
```
orm: SelectAs: models.Wrong has 1 exported field(s) but 2 expression(s)
    were given; they are matched one for one, in order
```

## A plain column needs no wrapper

`SelectExpr`, what `SelectAs` takes a list of, is satisfied by any typed
column already — `Authors.Name` is one on its own, with no `orm.ColumnOf(...)`
or similar needed. So is an [expression](/docs/2.fundamentals/16.expressions),
a `CASE`, and a subquery yielding one column. Only an aggregate or a window
function is wrapped, since those are shapes a column type alone cannot
describe.

## Computed columns

Anything the database can compute reads into a field of its own:

```go
type Line struct {
    Name  string
    Total int
}

rows, err := orm.SelectAs[Line](
    Items.With(db),
    Items.Name,
    Items.Price.Times(Items.Qty),
).All(ctx)
```
```sql
SELECT "name", ("price" * "qty") FROM "items"
```

The expression's own Go type is what the field is checked against, so a
`float64` computation read into an `int` field is reported before the
statement runs, exactly as a plain column would be. A `CASE` works the same
way:

```go
type Verdict struct {
    Name    string
    Outcome string
}

orm.SelectAs[Verdict](Items.With(db), Items.Name,
    orm.Case[string]().
        When(Items.Price.Value().GreaterThan(Items.Cost), "profit").
        Else("loss"),
)
```

See [Expressions](/docs/2.fundamentals/16.expressions) for the full
vocabulary.

## A subquery as a column

A query that yields one column reads as a column of the enclosing
projection — the same value you would otherwise hand to `InQuery`:

```go
latestBook := orm.Select(
    Books.With(db).
        Where(Books.AuthorID.Value().Equals(Authors.ID)).
        OrderBy(Books.ID.Desc()).
        Limit(1),
    Books.Title,
)

type Author struct {
    Name       string
    LatestBook string
}
rows, err := orm.SelectAs[Author](Authors.With(db), Authors.Name, latestBook).All(ctx)
```
```sql
SELECT "name", (SELECT "books"."title" FROM "books"
                WHERE "books"."author_id" = "authors"."id"
                ORDER BY "books"."id" DESC LIMIT 1)
FROM "authors"
```

That is a **correlated** subquery: its condition names the outer statement's
column, so it is answered per row of the outer query. Writing the correlation
at all needs comparing a column against a column, which is what
[`Value()`](/docs/2.fundamentals/16.expressions#comparing-a-column-against-a-column)
is for.

The subquery brings its own conditions, ordering and limit, and its
placeholders number in with the rest of the statement. `orm.NonNull` is a
subquery too, so it reads as a column on the same terms. A
[`CTE`](/docs/2.fundamentals/15.ctes) reference does not, only because
`SelectAs` will not run over a query carrying a `With` in the first place, so
there would be nothing defining the name it refers to.

::alert{type="warning"}
**A correlated subquery matching no row yields `NULL`.** Read it into a
pointer field, or select a nullable column, unless the query guarantees a
match — otherwise the scan fails on the row that has none.
::

## Aggregates

```go
orm.CountOf(Books.ID)   // COUNT("books"."id")
orm.CountAll()          // COUNT(*)
orm.SumOf(Users.Age)    // SUM("age"), typed by the column
orm.AvgOf(Users.Age)    // AVG("age"), always float64
orm.MinOf(Users.Age)    // MIN("age"), typed by the column
orm.MaxOf(Users.Age)    // MAX("age"), typed by the column
```

`SumOf`, `MinOf` and `MaxOf` infer their result type from the column, the same
way the free functions [`orm.Sum`, `orm.Min` and
`orm.Max`](/docs/2.fundamentals/8.querying#aggregates) already do. `AvgOf` is
always a `float64` whatever the column holds, for the reason `orm.Avg` gives:
a mean of integers is not an integer.

Each has a twin taking an
[expression](/docs/2.fundamentals/16.expressions#aggregating-one) rather than
a column — `CountOfExpr`, `SumOfExpr`, `AvgOfExpr`, `MinOfExpr`, `MaxOfExpr`
— which is how a conditional tally is written:

```go
orm.SumOfExpr(orm.Case[int]().When(Users.Active.Equals(true), 1).Else(0))
// SUM(CASE WHEN "active" = $1 THEN $2 ELSE $3 END)
```

### An aggregate is an expression

`COUNT`, `SUM` and the rest are [expressions](/docs/2.fundamentals/16.expressions)
like any other, so everything an expression does, they do:

```go
orm.SumOf(Sales.Revenue).DividedBy(orm.CountAll())  // SUM("revenue") / COUNT(*)
orm.Round(orm.AvgOf(Users.Age), 2)                  // ROUND(AVG("age"), $1)
orm.CountOf(Books.AuthorID).Distinct()              // COUNT(DISTINCT "author_id")
```

`Distinct` belongs to an aggregate and to nothing else — `LOWER(DISTINCT x)` is
not a thing SQL has — so calling it on an ordinary function is reported when the
statement builds, as is `COUNT(*).Distinct()`, whose argument is every row
rather than a value to take the distinct ones of.

### `GroupBy` and `Having`

```go
orm.SelectAs[AuthorReport](
    Authors.With(db).LeftJoin(Authors.Books),
    Authors.Name, orm.CountOf(Books.ID),
).GroupBy(Authors.Name).Having(orm.CountOf(Books.ID).GreaterOrEqual(int64(3))).All(ctx)
```

```sql
... GROUP BY "authors"."name" HAVING COUNT("books"."id") >= $1
```

`Having` takes predicates, exactly as `Where` does, so everything that composes
a condition composes here — several accumulate and are joined with `AND`,
`orm.Or` nests alternatives, and both sides may be aggregates:

```go
.Having(orm.SumOf(Sales.Revenue).GreaterThan(orm.SumOf(Sales.Cost)))
.Having(orm.Or(orm.CountAll().GreaterThan(int64(10)), orm.AvgOf(Items.Price).LessThan(5.0)))
```

A condition need not name an aggregate already in the `SELECT` list — ordinary
SQL allows `HAVING COUNT(*) > 5` with no `COUNT(*)` selected, and this follows
it. The difference from `Where` is which rows each is asked about: `Where`
filters rows before they are grouped, `Having` filters the groups afterwards,
and only `Having` may name an aggregate at all.

### Grouping by a computed value

`GroupBy` takes anything the `SELECT` list can carry, which is what the monthly
report needs:

```go
month := orm.Fn[time.Time]("date_trunc", "month", Orders.CreatedAt)

orm.SelectAs[Monthly](Orders.With(db), month, orm.SumOf(Orders.Total)).
    GroupBy(month).
    Having(orm.SumOf(Orders.Total).GreaterThan(1000)).
    OrderBy(month.Asc()).
    All(ctx)
```

```sql
SELECT date_trunc(CAST($1 AS TEXT), "created_at"), SUM("total") FROM "orders"
GROUP BY date_trunc(CAST($2 AS TEXT), "created_at")
HAVING SUM("total") > $3
ORDER BY date_trunc(CAST($4 AS TEXT), "created_at") ASC
```

The expression is written out in each clause rather than referred to by its
position in the `SELECT` list. An output column's ordinal is legal in most
databases and reads as a magic number in all of them — which is also why its
own bound argument appears once per clause.

## Window functions

```go
type RankedUser struct {
    Username string
    Country  string
    Rank     int64
}

rows, err := orm.SelectAs[RankedUser](
    Users.With(db), Users.Username, Users.Country,
    orm.RowNumber().PartitionBy(Users.Country).OrderBy(Users.Age.Desc()),
).All(ctx)
```

```sql
SELECT "username", "country",
    ROW_NUMBER() OVER (PARTITION BY "country" ORDER BY "age" DESC)
FROM "users"
```

A window function is an ordinary [expression](/docs/2.fundamentals/16.expressions),
so it sits in a projection beside plain columns and aggregates. `PartitionBy`
restarts it within each group of rows sharing those columns' values — without
it, the whole result is one window — and `OrderBy` decides what "first" means
inside one.

`RowNumber` numbers every row distinctly, breaking ties arbitrarily. `Rank`
gives tied rows the same number and skips ahead by the tie's size afterward;
`DenseRank` is the same without the skip.

The vocabulary is larger than those three: `NTile`, `Lag`, `Lead`,
`FirstValue`, `LastValue`, any aggregate through `Over` — a running total is
`orm.SumOf(x).Over().OrderBy(...)` — and frames through `Rows` and `Range`. See
[Window functions](/docs/2.fundamentals/19.window-functions).

::alert{type="warning"}
**To filter on a window function's own result**, wrap the projection as a
[derived table](/docs/2.fundamentals/17.derived-tables). `WHERE rank <= 3` —
the "top N per group" report — cannot be written against the projection
itself, because the rank does not exist until the projection has produced it.
::

## What `SelectAs` cannot carry

A lock (`ForUpdate`, `ForShare`) is rejected — locking names rows of a table,
and a projection that mixes columns with aggregates has nothing in particular
left to lock:

```go
orm.SelectAs[AuthorReport](Authors.With(db).Where(...).ForUpdate(), ...)
// error: SelectAs collapses the rows into a value, so there is nothing left
//        for ForUpdate or ForShare to lock
```

Every column and every aggregate's column must belong to a table the source
query actually reads from — its own table, or one it `Join`ed onto — checked
the same way `Where` already checks a condition's column:

```go
orm.SelectAs[AuthorReport](Authors.With(db), Books.Title)
// error: table "authors": column "title" belongs to table "books", which
//        this statement does not select from
```

## Document columns are not a `SelectExpr` yet

`SelectAs` scans positionally through each exported field's own destination,
which does not run a JSON column's codec. Selecting one through `SelectAs`
today reads its raw encoded bytes rather than the decoded Go value — read a
document column through an ordinary row or `orm.Select` instead.
