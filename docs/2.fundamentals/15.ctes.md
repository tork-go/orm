# CTEs

```go
recent := orm.Select(Posts.With(db).Where(Posts.CreatedAt.GreaterThan(cutoff)), Posts.AuthorID)

users, err := Users.With(db).
    With("recent_authors", recent).
    Where(Users.ID.InQuery(orm.CTE[int]("recent_authors"))).
    All(ctx)
```

```sql
WITH "recent_authors" AS (SELECT "author_id" FROM "posts" WHERE "created_at" > $1)
SELECT ... FROM "users" WHERE "id" IN (SELECT * FROM "recent_authors")
```

`With` attaches a named `WITH` clause to a query, defined by the same shape
[`InQuery` and `NotInQuery`](/docs/2.fundamentals/8.querying#matching-against-another-query)
already accept — what `orm.Select` returns. `CTE[T]` refers back to it from a
condition elsewhere in the same query.

## Placeholders number in the order they appear in the statement, not the order you wrote them

The CTE's own definition is textually first in the finished SQL — `WITH ... AS
(...) SELECT ...` — so its placeholders are numbered first too, whatever order
`With` and `Where` were called in:

```go
Authors.With(db).
    With("book_authors", recentBooks).           // its own $1
    Where(Authors.Name.Equals("Pratchett")).          // becomes $2
    Where(Authors.ID.InQuery(orm.CTE[int]("book_authors"))).
    SQL()
```

## More than one

Calls accumulate, each rendered in the order `With` was called:

```go
Users.With(db).
    With("a", subqueryA).
    With("b", subqueryB).
    SQL()
// WITH "a" AS (...), "b" AS (...) SELECT ...
```

## Only a plain read supports one, for now

`Scalars` (`orm.Select`), `Grouped` (`orm.CountBy` and friends), the scalar
aggregates, `SelectAs`, a combined query (`Union` and friends) and a set
operation (`UpdateAll`, `DeleteAll`) all reject a query carrying a `With`
rather than silently dropping its definition while a condition still refers to
it by name:

```go
orm.Select(Users.With(db).With("x", recent), Users.Username)
// error: Select cannot run over a query carrying a With; only a plain read
//        supports a CTE for now
```

`Filtered`'s own reads — `All`, `First`, `Count`, `SQL` — are what support a
`With` today. A **recursive** CTE is not subject to this: it defines the table
being read rather than one a condition refers to by name, so a projection over
one carries its definition along. See below.

## Recursive CTEs

A recursive CTE reads its own output. It starts from an **anchor** — the rows
it can find without recursing — then runs a **step** that joins against
everything found so far, over and over, until a pass finds nothing new. It is
the one query shape no amount of ordinary SQL replaces: an org chart to any
depth, a category tree, the parts of a part, a graph walked from a node.

The columns are named by a [derived table](/docs/2.fundamentals/17.derived-tables)
model, exactly as a non-recursive one names them:

```go
type Report struct {
    ID       int
    Name     string
    Depth    int
}

var Reports = orm.DefineDerived[Report]("reports", func(t *orm.TableBuilder[Report]) *ReportModel {
    return &ReportModel{
        DerivedTable: t.Derived(),
        ID:           t.Int("id"),
        Name:         t.String("name"),
        Depth:        t.Int("depth"),
    }
})

tree, err := Reports.Recursive(
    // the anchor: the top of the chart, at depth 0
    orm.SelectAs[Report](
        Employees.With(db).Where(Employees.ManagerID.IsNull()),
        Employees.ID, Employees.Name, orm.Lit(0),
    ),
    // the step: everyone reporting to somebody already found, one deeper
    orm.SelectAs[Report](
        Employees.With(db).JoinTo(Reports, Reports.ID.Value().Equals(Employees.ManagerID)),
        Employees.ID, Employees.Name, Reports.Depth.Plus(1),
    ),
).OrderBy(Reports.Depth.Asc()).All(ctx)
```

```sql
WITH RECURSIVE "reports"("id", "name", "depth") AS (
    SELECT "id", "name", CAST($1 AS INTEGER) FROM "employees" WHERE "manager_id" IS NULL
    UNION ALL
    SELECT "employees"."id", "employees"."name", ("reports"."depth" + $2)
    FROM "employees" JOIN "reports" ON "reports"."id" = "employees"."manager_id"
)
SELECT "id", "name", "depth" FROM "reports" ORDER BY "depth" ASC
```

**The step names the model itself**, and that is what makes it recursive:
inside the recursion the model is an ordinary table reference holding every
row found so far. Join it, filter on its columns, compute from them — as the
depth counter above does.

Both halves have to yield exactly what the model declares, one for one and in
order. That is checked when the statement builds, and the error names which
half was wrong rather than leaving it to the database.

The result is a `Filtered` over the finished pool, so everything a read does
it does: `Where`, `OrderBy`, `Limit`, a projection over it, a join onto it.

### Recursion that ends

Rows are pooled with `UNION ALL`, so a step that keeps finding rows keeps
running. Over a tree that is what you want, and it always ends, because a tree
runs out of children. Over a graph that may contain a cycle it does not: A
points at B, B points back at A, and the two feed each other forever.

`RecursiveDistinct` is the answer there. It pools with `UNION`, which drops a
row already found, so a cycle closes instead of looping:

```go
Reports.RecursiveDistinct(anchor, step).All(ctx)
```

It costs what dropping duplicates always costs — every row found is compared
against every row already pooled — so `Recursive` is the one to reach for over
a tree, where there is nothing to compare.

A depth column is the other guard, and worth carrying anyway when the question
is "how far" and not merely "who": `orm.Lit(0)` in the anchor, that column plus
one in the step, and a `Where` on it afterwards.
