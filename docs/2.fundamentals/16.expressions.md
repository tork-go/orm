# Expressions

An expression is a value the **database** computes, rather than one you send
it. It reads left to right from the column it starts at:

```go
Items.Price.Times(Items.Qty)          // "price" * "qty"
Items.Price.Times(Items.Qty).Plus(10) // ("price" * "qty") + $1
```

The same value goes in four places, and means the same thing in each:

```go
// read it
orm.SelectAs[Row](Items.With(db), Items.Name, Items.Price.Times(Items.Qty))

// filter on it
Items.With(db).Where(Items.Price.Times(Items.Qty).GreaterThan(100))

// sort by it
Items.With(db).OrderBy(Items.Price.Times(Items.Qty).Desc())

// write it
Items.With(db).Where(...).UpdateAll(ctx, Items.Qty.SetExpr(Items.Qty.Minus(1)))
```

## The four operators

```go
Items.Price.Plus(Items.Shipping)     // "price" + "shipping"
Users.Balance.Minus(amount)          // "balance" - $1
Items.Price.Times(Items.Qty)         // "price" * "qty"
Orders.Total.DividedBy(Orders.Count) // "total" / "count"
```

They are methods rather than functions on purpose. `T` comes from the
receiver, so typing `Items.Price.` in an editor offers the whole vocabulary —
a free function taking two operands could not, since there is nothing to
complete from. They also chain, so a longer computation reads in the order it
happens instead of growing inside-out.

Each operand accepts **a column, another expression, or a plain value**:

```go
Items.Price.Times(Items.Qty)              // a column
Items.Price.Times(Items.Qty.Plus(1))      // an expression
Items.Price.Times(2)                      // a value
```

Only the six numeric column kinds carry arithmetic — `Int`, `Int32`,
`BigInt`, `Float`, `Double` and `Decimal`. `Users.Username.Times(2)` does not
compile, the same way `Users.Active.GreaterThan(false)` does not.

### Everything is parenthesised

```go
Items.Price.Minus(Items.Cost).Times(Items.Qty)
// (("price" - "cost") * "qty")
```

Tork does not reimplement SQL's precedence rules to work out when brackets
are needed. Getting that subtly wrong would change *what a statement
computes* rather than merely how it reads — `(price - cost) * qty` and
`price - (cost * qty)` are different numbers — so every level is bracketed
and the Go you wrote is the grouping you get.

## Comparing a column against a column

A column's own `Equals` and `GreaterThan` take **a value** of that column's
type. That is what makes `Users.Age.GreaterThan("x")` a compile error, and it
is worth keeping. It also means they cannot accept a column: Go has no
overloading, and a second method of the same name would make the selector
ambiguous.

`Value()` is the way across. It lifts a bare column into an expression, and
an expression's comparisons accept a column, an expression, or a value:

```go
Items.With(db).Where(Items.Price.Value().GreaterThan(Items.Cost))
// WHERE "price" > "cost"
```

Arithmetic needs no lift, because it already produces an expression:

```go
Items.Price.Times(Items.Qty).GreaterThan(100)   // no Value()
Items.Price.Value().GreaterThan(Items.Cost)     // Value() needed
Users.Age.GreaterThan(18)                       // unchanged, still checked
```

An expression carries the same six comparisons a column does — `Equals`,
`NotEquals`, `GreaterThan`, `GreaterOrEqual`, `LessThan`, `LessOrEqual` — so
there is no second vocabulary to learn.

This is also what makes a [correlated
subquery](/docs/2.fundamentals/14.projections#a-subquery-as-a-column)
expressible, since correlating means comparing the inner table's column
against the outer's.

## A wrongly typed value is caught when the statement builds

Because an operand accepts a column *or* an expression *or* a value, it is
typed `any`, so the Go compiler cannot check the value. Tork checks it
instead, when the statement is built:

```go
Users.With(db).Where(Users.Age.Value().GreaterThan("x")).All(ctx)
```
```
orm: table "users": this expression is int but the value combined with it
    is string
```

That is the same treatment `Find` already gives a wrongly typed key: named,
before anything runs, rather than left to come back as a database error.

::alert{type="info"}
**The result stays typed.** Only the operand is unchecked. `Items.Price.Times(2)`
is an `Expr[float64]` whatever you passed, so assigning it to an `int` column
with `SetExpr`, or reading it into a `string` struct field with `SelectAs`, is
still a compile error.
::

## `CASE`

```go
orm.Case[string]().
    When(Items.Price.Value().GreaterThan(Items.Cost), "profit").
    When(Items.Price.Value().Equals(Items.Cost), "break-even").
    Else("loss")
```
```sql
CASE WHEN "price" > "cost" THEN $1
     WHEN "price" = "cost" THEN $2
     ELSE $3 END
```

Arms are tested in the order given, and the first whose condition holds
decides the value. `Case` names its type because nothing else in the
expression fixes it — an untyped `1` would infer `int` where the column it
sits beside is a `float64`.

It is an ordinary expression, so it sorts and filters as readily as it reads:

```go
// put admins first, everyone else after
Users.With(db).OrderBy(
    orm.Case[int]().When(Users.Role.Equals("admin"), 0).Else(1).Asc(),
)
```

### `Else` is required

A `CASE` with no `ELSE` yields `NULL` when nothing matches, and a `NULL`
cannot be scanned into a `T` that is not a pointer. Requiring `Else` means
`T` never has to stand for absence, and the builder only becomes an
expression once its value is total.

Where `NULL` genuinely is the answer, say so in the type:

```go
orm.Case[*int]().When(cond, &n).Else(nil)
```

::alert{type="warning"}
**A `CASE` cannot be aggregated yet.** `SUM(CASE WHEN … THEN 1 ELSE 0 END)`,
the usual way to write a conditional tally, needs an aggregate over an
expression, and `CountOf`, `SumOf` and the rest take a column. Count in Go
after reading, or reach for
[`RawQuery`](/docs/2.fundamentals/8.querying#raw-queries-typed).
::

## What will not take an expression

**Cursor paging.** `Cursor` and `After` seek by comparing the ordering's own
columns against values read back out of a row. A computed ordering has no
field to read from, so it is refused rather than paged from a zero value that
would silently start from the wrong place:

```
orm: table "users": Cursor cannot page an ordering computed by the database,
    at position 0; order by a column, or page with Limit and Offset instead
```

**A grouped query.** `orm.CountBy` and its siblings order only by their key or
by their aggregate, since every other column has been collapsed into the
group; a computed ordering is neither.
