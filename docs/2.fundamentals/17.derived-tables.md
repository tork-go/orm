# Derived tables

Some questions cannot be asked of a table directly, only of another query's
result. "The top three scorers in each team" needs the ranking to exist
before anything can filter on it. A derived table is that: a query standing
in for a table.

```go
type Ranked struct {
    Username string
    Rank     int64
}

type RankedModel struct {
    orm.DerivedTable[Ranked]
    Username *orm.StringColumn
    Rank     *orm.BigIntColumn
}

var RankedT = orm.DefineDerived[Ranked]("ranked",
    func(t *orm.TableBuilder[Ranked]) *RankedModel {
        return &RankedModel{
            DerivedTable: t.Derived(),
            Username:     t.String("username"),
            Rank:         t.BigInt("rank"),
        }
    })

inner := orm.SelectAs[Ranked](
    Users.With(db),
    Users.Username,
    orm.RowNumber().PartitionBy(Users.Country).OrderBy(Users.Age.Desc()),
)

top3, err := RankedT.From(inner).Where(RankedT.Rank.LessOrEqual(3)).All(ctx)
```
```sql
SELECT "username", "rank" FROM (
    SELECT "username" AS "username",
           ROW_NUMBER() OVER (PARTITION BY "country" ORDER BY "age" DESC) AS "rank"
    FROM "users"
) AS "ranked" WHERE "rank" <= $1
```

## Why it is declared as a model

A derived table's columns have to be named by something, and the alternatives
are worse. Naming them by string (`Field[int64](sub, "Rank")`) or by position
(`At[int64](sub, 1)`) would give up compile-time checking on the one table
whose shape is least obvious — the one that does not exist until a query
produces it.

Declaring it is the same declaration every other table gets, so `RankedT.Rank`
is an ordinary typed column. Everything a query does with a stored table's
columns it does with these: filtering, ordering, paging, counting, reading one
column, grouping.

`DerivedTable` and `Table` are different types on purpose. `With` belongs to a
stored table and means nothing for a derived one, which has no rows until it
is given a source; `From` is the reverse. Each of those is a compile error
rather than something reported at run time.

The shape is checked when `From` is called — one output per declared column,
each assignable to its column's type:

```
orm: derived table "ranked": column 1, "rank", is int64 but the source's
    expression 1 is string
```

::alert{type="info"}
**It takes no part in migrations.** Nothing extra is needed for that: pass
your stored models to `schema.ExtractSchema` as you already do, and a derived
one is simply never among them. It is not registered against its row type
either, so declaring a derived table over the same struct as a stored one does
not disturb relationships.
::

## What can be a source

**A projection**, which is the common case and the only one that can compute
new columns:

```go
orm.SelectAs[Ranked](Users.With(db), Users.Username, orm.RowNumber()...)
```

**A combined query**, which is how a `UNION` gets filtered — a combined result
has no `WHERE` of its own:

```go
staff := orm.Union(admins, moderators)
recent, err := StaffT.From(staff).Where(StaffT.Role.Equals("admin")).All(ctx)
```

**A plain read**, the simplest way to wrap a result so it can be narrowed
again:

```go
NamedUsers.From(Users.With(db).Select(Users.ID, Users.Username).Where(...))
```

The source's own columns are aliased to the names the derived model declares,
which is what lets the outer query refer to a computed column at all. For a
combined query only the left operand is aliased, since SQL takes a combined
result's column names from its first operand.

## The three shapes it unlocks

**Top N per group**, shown above: a window function's result is not something
the query producing it can filter on.

**An aggregate of an aggregate.** Grouping produces one row per group; to
aggregate *those*, the grouped result has to be a table first:

```go
perTeam := orm.SelectAs[TeamTotal](
    Scores.With(db), Scores.Team, orm.SumOf(Scores.Points),
).GroupBy(Scores.Team)

mean, err := orm.Avg(ctx, TeamTotals.From(perTeam), TeamTotals.Total)
```

**Re-filtering a combined query**, since `Union` and its siblings carry only
an `OrderBy` and a `Limit`.

## What a derived query will not take

```go
RankedT.From(inner).Load(...)        // no relationships to load
RankedT.From(inner).Join(...)        // same
RankedT.From(inner).UpdateAll(...)   // nothing to write back to
RankedT.From(inner).DeleteAll(ctx)   // same
```

Each says so by name. A derived table has no foreign keys, so a relationship
has nothing to resolve through, and its rows are a query's output rather than
storage. The entity operations — `Insert`, `Update`, `Delete` — are not
offered at all: `From` returns a `Filtered`, which never had them.

The **source** has limits of its own. A lock is refused, because no dialect
Tork targets locks rows through a derived table, and a `Preload` is refused
because it runs a query against rows the wrapping statement never has on their
own.
