# Aliases and self-joins

A table has one name, and every column carries it. That is enough until a
statement has to reach the same table twice — an employee and their manager are
both rows of `employees`, and `JOIN "employees" ON ...` leaves the database with
no way to tell which one a condition means. An alias is a second name for one
table, so the statement can.

```go
mgr := orm.Alias(Employees, "mgr")

Employees.With(db).
    JoinAs(Employees.Manager, mgr).
    Where(mgr.Name.Equals("Ada")).
    All(ctx)
```

```sql
SELECT "employees"."id", "employees"."name", ... FROM "employees"
JOIN "employees" AS "mgr" ON "mgr"."id" = "employees"."manager_id"
WHERE "mgr"."name" = $1
```

## What `orm.Alias` gives back

The same model type, with the same typed columns, each bound to the new name
instead of the stored one. `mgr.Name` is an ordinary `*StringColumn`, and
`mgr.Name.Equals("Ada")` an ordinary predicate; what differs is only that it
qualifies as `"mgr"."name"`. Everything a query can do with a table's own
columns — filter, order, group, project, aggregate — it can do with an alias's,
because they are columns in exactly the same sense.

```go
mgr := orm.Alias(Employees, "mgr")
mgr.TableName()          // "mgr"
mgr.Name.OwnerTable()    // "mgr"
Employees.Name.OwnerTable() // "employees" — the table it was made from is untouched
```

The model is declared a second time, by re-running the very build function
`DefineTable` was given. Nothing is copied field by field, which is what keeps an
alias's columns as real as the originals rather than a thinner imitation of them.

## Joining a relationship under an alias

`JoinAs`, `LeftJoinAs`, `JoinOnAs` and `LeftJoinOnAs` are `Join`, `LeftJoin`,
`JoinOn` and `LeftJoinOn` with a name for the far side. The relationship still
supplies the keys the join matches on; the alias supplies the name that side is
rendered under, and therefore which columns your conditions name.

```go
Employees.With(db).LeftJoinOnAs(Employees.Manager, mgr, mgr.Active.Equals(true))
```

```sql
LEFT JOIN "employees" AS "mgr"
    ON "mgr"."id" = "employees"."manager_id" AND "mgr"."active" = $1
```

The alias must be of the table the relationship names. One of some other table is
reported rather than joined, however similar its columns happen to look.

## Joining the same table twice

Every join needs its own name, and each renders in the order it was called:

```go
mgr := orm.Alias(Employees, "mgr")
report := orm.Alias(Employees, "report")

Employees.With(db).
    JoinAs(Employees.Manager, mgr).
    JoinAs(Employees.Reports, report)
```

```sql
FROM "employees"
JOIN "employees" AS "mgr"    ON "mgr"."id" = "employees"."manager_id"
JOIN "employees" AS "report" ON "report"."manager_id" = "employees"."id"
```

## `JoinTo`: two tables with no relationship

`Join` and its siblings take a relationship, which supplies the keys to match on.
`JoinTo` takes the conditions instead, for the tables between which no
relationship is declared: a log nobody wanted a foreign key on, a join on
something other than a key, or a table whose relationship exists but is not the
one this statement means to follow.

```go
Users.With(db).
    JoinTo(Logins, Logins.UserID.Value().Equals(Users.ID)).
    Where(Logins.Failed.Equals(true)).
    All(ctx)
```

```sql
SELECT "users"."id", ... FROM "users"
JOIN "logins" ON "logins"."user_id" = "users"."id"
WHERE "logins"."failed" = $1
```

Comparing two columns needs no vocabulary of its own.
[`Value`](/docs/2.fundamentals/16.expressions) lifts a column into an expression,
whose comparisons accept a column as readily as a literal.

The conditions are the whole of the `ON` clause, so at least one is required:

```go
Users.With(db).JoinTo(Logins)
```

```
orm: table "users": JoinTo was given no conditions to join "logins" on; a join
    with none is a cross join, which this package does not offer
```

`JoinTo` takes an alias as readily as a table, which is what joining one table to
itself on something other than a declared key needs.

## Naming one table twice is refused

```go
Employees.With(db).Join(Employees.Manager).All(ctx)
```

```
orm: table "employees": this join names table "employees", which the statement
    already reads from; give one side a name of its own with orm.Alias and join
    it with JoinAs
```

The same guard catches a table joined twice for two different reasons, not only a
table joined to itself. It is reported before the statement is sent, rather than
by the database once it is.

## Reading through an alias

An alias is a handle on the same rows, so a query through one reads exactly what
the stored table holds:

```go
mgr.With(db).Where(mgr.Active.Equals(true)).All(ctx)
```

```sql
SELECT "id", "name", "active", "manager_id" FROM "employees" AS "mgr"
WHERE "active" = $1
```

Aliasing an alias names the stored table, so a chain of them never grows a chain
of names.

## What an alias cannot do

It cannot be written through. An `INSERT`, `UPDATE` or `DELETE` names a stored
table, and an alias exists only inside the statement that introduces it:

```go
mgr.With(db).Insert(ctx, &employee)
```

```
orm: table "mgr": a write cannot run over an alias of "employees", which names no
    stored table to write to; write through the model the alias was made from
```

It does not join the registry. The registry maps a row type to the table declared
for it, which is how a relationship finds its far side; an alias registering
itself would shadow the real table for every relationship naming that row type.

It takes no part in migrations. `schema.ExtractSchema` is handed its models
explicitly, so an alias is simply never passed to it.

A model built with `NewTable` cannot be aliased, and neither can a
[derived table](/docs/2.fundamentals/17.derived-tables) — whose name is already
the alias its subquery is wrapped under. Both are panics, for the reason
`DefineTable`'s are: an alias is declared from a model and a name, neither of
which can vary at run time.
