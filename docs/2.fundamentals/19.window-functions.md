# Window functions

An aggregate collapses a group into one number. A window function leaves every
row where it is and gives each one a value computed from the rows around it:

```go
orm.SumOf(Sales.Total)                                   // SUM("total") — one number
orm.SumOf(Sales.Total).Over().OrderBy(Sales.Day.Asc())   // a running total, per row
```

```sql
SUM("total") OVER (ORDER BY "day" ASC)
```

They are the same call with an `OVER` clause between them, which is why every
aggregate here can be windowed and why a window function is an ordinary
[expression](/docs/2.fundamentals/16.expressions) — it nests, compares and
sorts like any other.

## The clauses

```go
orm.SumOf(Sales.Total).
    PartitionBy(Sales.Region).       // restart per region
    OrderBy(Sales.Day.Asc()).        // count in this order
    Rows(orm.UnboundedPreceding(), orm.CurrentRow())   // see this much of it
```

```sql
SUM("total") OVER (
    PARTITION BY "region" ORDER BY "day" ASC
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
```

`PartitionBy` divides the result the way `GroupBy` divides a grouped read —
without it, the whole result is one window. `OrderBy` decides what "so far"
means; without it the database counts rows in whatever order it visits them,
which is rarely what a rank or a running total is meant to say. Both accumulate
across calls, and both imply the window, so `Over` is only needed when neither
follows:

```go
orm.SumOf(Sales.Total).Over()   // a grand total beside every row
```

The window's ordering is its own. A read may sort its rows one way and count
them another, and the two never interfere.

## Ranking

```go
orm.RowNumber()   // 1, 2, 3, 4 — ties broken arbitrarily
orm.Rank()        // 1, 2, 2, 4 — ties share, then skip
orm.DenseRank()   // 1, 2, 2, 3 — ties share, no skip
orm.NTile(4)      // which quarter of the window this row falls in
```

These exist only as windows, so they carry an `OVER` clause from the start.

To **filter on a rank** — the "top three per group" report — wrap the
projection as a [derived table](/docs/2.fundamentals/17.derived-tables). No
statement can name a window function in its own `WHERE`, because the rank does
not exist until the rows have been produced.

## Neighbours

```go
orm.Lag(Sales.Total).OrderBy(Sales.Day.Asc())    // the previous row's value
orm.Lead(Sales.Total).OrderBy(Sales.Day.Asc())   // the next row's
```

Both are typed as pointers: the first row of a window has no row before it and
the last none after, so the result is genuinely absent there. Where a zero
reads better than a nil, [`Coalesce`](/docs/2.fundamentals/16.expressions#the-named-ones)
says so:

```go
orm.Coalesce[int](orm.Lag(Sales.Total).OrderBy(Sales.Day.Asc()), 0)
```

`FirstValue` and `LastValue` read the ends of the window rather than the
neighbours, and are not pointers — a window always holds at least the row being
computed. `LastValue` is the one to watch: its default frame stops at the
current row, so without a frame it reads the current row rather than the
window's final one. That is SQL's own rule, and the reason to write the frame
out:

```go
orm.LastValue(Sales.Total).
    OrderBy(Sales.Day.Asc()).
    Rows(orm.UnboundedPreceding(), orm.UnboundedFollowing())
```

## Frames

A frame says how much of the window each row sees:

```go
orm.UnboundedPreceding()   // from the start of the window
orm.Preceding(2)           // two back
orm.CurrentRow()
orm.Following(3)           // three forward
orm.UnboundedFollowing()   // to the end
```

`Rows` counts rows; `Range` counts values, so every row sorted equal to the
current one is in the frame together. A window with an `ORDER BY` and no frame
at all is `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, which is why a
running total over a column with ties adds every tied row at once — `Rows` is
what counts them one at a time.

The frame earns its place where it differs from that default:

```go
orm.AvgOf(Sales.Total).OrderBy(Sales.Day.Asc()).Rows(orm.Preceding(2), orm.CurrentRow())
// a trailing average over the last three rows
```

A frame that runs backwards is reported when the statement builds rather than
sent:

```
orm: table "sales": SUM: this window's frame starts at CURRENT ROW and ends at
    1 PRECEDING, which is before it starts
```

## Only a call can be windowed

`OVER` belongs to a function call. Arithmetic and a `CASE` are windowed by
putting them inside one:

```go
orm.SumOfExpr(Items.Price.Times(Items.Qty)).Over().OrderBy(Items.ID.Asc())
```

Asking arithmetic itself for a window is reported instead:

```
orm: table "items": this expression is not a function call, so it cannot be
    given a window; OVER belongs to a call, and arithmetic or a CASE is
    windowed by putting it inside one
```
