# Columns

A column is a `*Column[T]` field, created with `NewColumn[T]` and configured
with a chain of builder methods:

```go
Username *orm.Column[string]

Username: orm.NewColumn[string]("username").Unique().NotNull().MaxLen(30),
```

`T` is the column's Go value type. Tork uses it to pick the SQL column type when
generating migrations: `int` becomes `INTEGER`, `int64` becomes `BIGINT`,
`string` becomes `TEXT` (or `VARCHAR(n)`, see below), `bool` becomes `BOOLEAN`,
`float64` becomes `DOUBLE PRECISION`, `decimal.Decimal` becomes `NUMERIC`, a
slice type like `[]string` becomes an array (`TEXT[]`), and so on. You don't
write the SQL type anywhere; it's derived from `T`.

## Nullable columns

A pointer type marks a column nullable, the same convention as `Optional[T]` in
other languages:

```go
Email *orm.Column[*string] // nullable
ID    *orm.Column[int]     // not nullable
```

`Column[*string]` and `Column[string]` are different instantiations of the same
generic type. Any number of leading pointers works (`**int` behaves like `*int`
for this purpose), though a single pointer is what you'll normally reach for.

## Builder methods

Each of these returns `*Column[T]`, so they chain:

| Method | Effect |
|---|---|
| `.PrimaryKey()` | Marks the column as (part of) the table's primary key. |
| `.Unique()` | Adds a unique constraint on this column. |
| `.NotNull()` | Marks the column `NOT NULL`, independent of whether `T` is a pointer. |
| `.MaxLen(n int)` | Sets a maximum length. Only meaningful on a string column; see below. |
| `.Index()` | Adds a plain (non-unique) index on this column. See below. |
| `.ServerDefault(expr string)` | Gives the column a `DEFAULT` computed by the database. See below. |
| `.GeneratedByClient(gen func() T)` | Declares a Go-side value generator. See below. |
| `.Numeric(precision, scale int)` | Sets explicit precision/scale on a `decimal.Decimal` column. See below. |
| `.Enum(typeName string, values ...string)` | Declares the column as a native Postgres enum. See below. |
| `.JSON()` / `.JSONB()` | Stores the column as `JSON`/`JSONB`. See below. |
| `.Serialize(marshal, unmarshal)` | Overrides the default `encoding/json` marshaling for a JSON/JSONB column. See below. |

```go
ID:       orm.NewColumn[int]("id").PrimaryKey(),
Username: orm.NewColumn[string]("username").Unique().NotNull().MaxLen(30),
Email:    orm.NewColumn[*string]("email"),
```

A single-column integer or bigint primary key is automatically an
auto-incrementing identity column when Tork generates its `CREATE TABLE`;
that's covered in [Constraints](/docs/2.fundamentals/4.constraints).

## `NotNull` and nullability are two different things

`IsNullable()` (below) reflects `T`: it's `true` for `Column[*string]`, `false`
for `Column[string]`, and that's fixed at compile time by the type you wrote.
`NotNull()` is a separate, explicit call that only matters when `T` is a
pointer, if you want a column that's nullable in Go but required in the
database. Calling `.NotNull()` on a `Column[string]` (already non-nullable) has
no additional effect.

## `MaxLen` is only for strings

`MaxLen` exists on every `Column[T]`, regardless of `T`, because Go generics
can't add a method to only some instantiations of a generic type. Calling it on
a non-string column, or with a value that isn't positive, is caught when a
migration is generated, not silently ignored. See
[Generating Migrations](/docs/3.migrations/3.generating-migrations) for the
exact rule. A string column with no `MaxLen` call becomes `TEXT` (unbounded);
one with `MaxLen(n)` becomes `VARCHAR(n)`.

## A plain index with `.Index()`

`.Unique()` gives you a unique constraint (which is also an index). For an
index that doesn't need to be unique, use `.Index()`:

```go
AuthorID: t.Int("author_id").References(Users.ID).Index(),
```

If a column has both `.Unique()` and `.Index()`, you get one unique
constraint, not a redundant separate index, a unique constraint already
provides an index in Postgres. For an index spanning more than one column,
see [Constraints](/docs/2.fundamentals/4.constraints).

## `.ServerDefault` for values the database computes

Some columns should be filled in by the database itself if you don't supply a
value: a timestamp defaulting to `now()`, a boolean defaulting to `false`, or a
UUID primary key defaulting to `gen_random_uuid()`. `.ServerDefault` takes a
raw SQL expression and renders it as a `DEFAULT` clause:

```go
CreatedAt: orm.NewColumn[string]("created_at").ServerDefault("now()").NotNull(),
```

A UUID primary key is a common use for this, since Postgres can generate one
for you:

```go
import "github.com/google/uuid"

ID: orm.NewColumn[uuid.UUID]("id").PrimaryKey().ServerDefault("gen_random_uuid()"),
```

`ServerDefault` on a *single-column integer or bigint* primary key is
rejected when a migration is generated: that column is already an
auto-incrementing identity column (see
[Constraints](/docs/2.fundamentals/4.constraints)), and Postgres doesn't allow
both `GENERATED ALWAYS AS IDENTITY` and an explicit `DEFAULT` on the same
column. A UUID primary key doesn't hit this, only integer/bigint ones do.

`ServerDefault` only affects the DDL Tork generates when creating or adding
the column. It isn't compared when diffing an existing column, and isn't read
back from a live database, changing it on an existing column and running
`makemigrations` again won't produce an `ALTER` for it. This is a known,
documented limitation, not an oversight.

## `.GeneratedByClient` for values Go computes

The other direction: a value your Go code computes before an insert, rather
than the database. `uuid.New` is the common case, if you'd rather generate
UUIDs in Go than ask Postgres to:

```go
ID: orm.NewColumn[uuid.UUID]("id").PrimaryKey().GeneratedByClient(uuid.New),
```

This stores the function as metadata on the column. It has no effect on
migrations or any SQL Tork generates today, there's no query or insert API
yet for it to plug into. Declaring it now means your model won't need to
change once that API exists.

## Arbitrary-precision numbers with `.Numeric`

`float64`/`float32` are binary floating point, not exact, the wrong choice
for money or anything else that needs exact decimal arithmetic. For that,
use [`decimal.Decimal`](https://pkg.go.dev/github.com/shopspring/decimal):

```go
import "github.com/shopspring/decimal"

Amount: orm.NewColumn[decimal.Decimal]("amount"),
```

On its own this renders a bare `NUMERIC` (Postgres's own arbitrary-precision
type, no fixed size), the same relationship a bare `string` column has to
`TEXT`. Call `.Numeric(precision, scale)` for an explicit `NUMERIC(p,s)`:

```go
Amount: orm.NewColumn[decimal.Decimal]("amount").Numeric(10, 2), // NUMERIC(10,2)
```

`.Numeric` is only valid on a `decimal.Decimal` column; calling it elsewhere
is rejected the same way an invalid `MaxLen` is, when a migration is
generated.

## Native enums with `.Enum`

`.Enum` declares the column as a Postgres native `ENUM` type, creating (and
managing the lifecycle of) that type alongside your table:

```go
type OrderStatus string

Status: orm.NewColumn[OrderStatus]("status").Enum("order_status", "pending", "paid", "done"),
```

The first argument is the enum type's own name (Postgres object, shared
schema-wide, not per-table); the rest are its values, in order. `T` must be a
string-kind type once pointer nullability is unwrapped (a plain `string`
works too, `OrderStatus` above is just documentation). If two columns across
your models declare the same type name, their value lists must match
exactly, checked when a migration is generated.

Enum values can only be **added**, never removed or reordered, without
Postgres recreating the type entirely, something Tork doesn't attempt. See
[Generating Migrations](/docs/3.migrations/3.generating-migrations) for what
that means for `makemigrations`.

## JSON and JSONB

`.JSON()` and `.JSONB()` mark a column as stored as `JSON`/`JSONB`, using
`encoding/json` to marshal whatever `T` is:

```go
Metadata: orm.NewColumn[map[string]any]("metadata").JSONB(),
```

Postgres itself recommends `jsonb` over `json` in virtually all cases (binary
storage, indexable, supports more operators); reach for `.JSON()` only when
you specifically need the original text preserved byte-for-byte.

### Custom serialization with `.Serialize`

`.Serialize` overrides the default `encoding/json` marshal/unmarshal pair
with your own, while keeping the JSON/JSONB storage kind:

```go
Metadata: orm.NewColumn[Metadata]("metadata").Serialize(
    func(m Metadata) ([]byte, error) { return m.MarshalBinary() },
    func(b []byte) (Metadata, error) { var m Metadata; err := m.UnmarshalBinary(b); return m, err },
),
```

Calling `.Serialize` alone, without a preceding `.JSON()`/`.JSONB()`, implies
`.JSONB()`. Like `.GeneratedByClient`, this is pure metadata today: there's
no query or insert API yet to call `Serialize`'s functions.

## Array columns

A slice-typed `Column[[]T]` becomes a Postgres array, no builder call
needed, the same way a pointer type means nullable without a builder call:

```go
Tags: orm.NewColumn[[]string]("tags"), // TEXT[]
```

`.MaxLen` and `.Numeric` apply to the *element* type, not the array itself:

```go
Tags:    orm.NewColumn[[]string]("tags").MaxLen(20),               // VARCHAR(20)[]
Amounts: orm.NewColumn[[]decimal.Decimal]("amounts").Numeric(10, 2), // NUMERIC(10,2)[]
```

Multi-dimensional arrays (`[][]string`) aren't supported. An array of a named
string type declared without `.Enum()` isn't supported either (`.Enum()`
itself requires a non-slice string-kind column); use `.JSONB()` for a
column that needs to hold a list of enum-like values.

## Reading a column's metadata back

These are the read side, used by migrations (and later, queries) to see what
you declared:

```go
Name() string
OwnerTable() string // the table this column belongs to, "" if unbound
IsPrimaryKey() bool
IsUnique() bool
HasNotNull() bool
MaxLength() (n int, ok bool) // ok is false if MaxLen was never called
GoType() reflect.Type
IsNullable() bool
IsIndexed() bool
ServerDefaultExpr() (expr string, ok bool) // ok is false if ServerDefault was never called
IsClientGenerated() bool
Generator() (gen func() T, ok bool) // the function passed to GeneratedByClient
NumericPrecisionScale() (precision, scale int, ok bool) // ok is false if Numeric was never called
IsJSON() bool
IsJSONB() bool // true if JSONB() was called, or Serialize() was called without JSON()
IsSerialized() bool
Serializer() (marshal func(T) ([]byte, error), unmarshal func([]byte) (T, error), ok bool)
EnumSpec() (typeName string, values []string, ok bool)

ReferencedTable() string   // "" unless the column references another
ReferencedColumn() string  // ""
OnDeleteAction() ForeignKeyAction
OnUpdateAction() ForeignKeyAction
```

`MaxLength`, `ServerDefaultExpr`, and `NumericPrecisionScale` return
`ok = false` if the corresponding builder was never called, which is
different from having been called with a zero value or an empty string.
`Generator` and `Serializer` are accessors that aren't on `ColumnMeta` (see
[Model Reflection](/docs/2.fundamentals/6.model-reflection)), since both
return a function mentioning `T`, which isn't known generically. You won't
usually call any of these directly; they exist mainly for code (like the
migrations system) that needs to inspect a column generically.

## Typed column types

`Column[T]` accepts every builder for every `T`, because Go generics can't add
a method to only some instantiations of a generic type. That's why `MaxLen`
exists on an `int` column and is only rejected later, when a migration is
generated.

The typed column types fix that by moving the decision up a level. Each one is
a distinct type exposing only the builders and operations its kind supports, so
a wrong call is a compile error at the point you write it:

```go
orm.NewStringColumn("username").Unique().NotNull().MaxLen(30)
orm.NewIntColumn("id").PrimaryKey()
orm.NewNullableStringColumn("email")
```

```go
orm.NewIntColumn("id").MaxLen(3)      // compile error: no method MaxLen
orm.NewStringColumn("s").Numeric(10, 2) // compile error: no method Numeric
```

There is a nullable and a non-nullable form of each kind:

| Go type | Non-nullable | Nullable |
| --- | --- | --- |
| `bool` | `BoolColumn` | `NullableBoolColumn` |
| `int` | `IntColumn` | `NullableIntColumn` |
| `int32` | `Int32Column` | `NullableInt32Column` |
| `int64` | `BigIntColumn` | `NullableBigIntColumn` |
| `float32` | `FloatColumn` | `NullableFloatColumn` |
| `float64` | `DoubleColumn` | `NullableDoubleColumn` |
| `decimal.Decimal` | `DecimalColumn` | `NullableDecimalColumn` |
| `string` | `StringColumn` | `NullableStringColumn` |
| `time.Time` | `TimeColumn` | `NullableTimeColumn` |
| `uuid.UUID` | `UUIDColumn` | `NullableUUIDColumn` |
| `string` (native enum) | `EnumColumn` | `NullableEnumColumn` |
| `T` (JSON document) | `JSONColumn[T]` | `NullableJSONColumn[T]` |
| `[]T` | `ArrayColumn[T]` | `NullableArrayColumn[T]` |

Every one of them is a `ColumnMeta`, so schema extraction, diffing, and
migration generation treat them exactly like a `Column[T]`. Nothing downstream
changes.

Two pairs are worth reading carefully, because the obvious name isn't the
obvious type:

- **`FloatColumn` is `float32`** (`REAL`); `float64` is `DoubleColumn`
  (`DOUBLE PRECISION`). The types are named for the SQL kind they produce, not
  for Go's habit of calling `float64` a float.
- **`IntColumn` is `int`, `Int32Column` is `int32`.** Both become `INTEGER`.
  Both exist because a column's Go type must match its entity field exactly, so
  an `int32` field needs an `int32` column.

### Which builders each kind gets

`PrimaryKey`, `Unique`, `NotNull`, `Index`, `ServerDefault`, and
`GeneratedByClient` are meaningful for any kind, so every typed column has
them. The rest are restricted:

| Builder | Available on |
| --- | --- |
| `MaxLen` | `StringColumn`, `ArrayColumn[T]` |
| `Numeric` | `DecimalColumn`, `ArrayColumn[T]` |
| `Enum` | `EnumColumn` |
| `JSON`, `JSONB`, `Serialize` | `JSONColumn[T]` |

`References`, `ReferencesTable`, `OnDelete`, and `OnUpdate` are narrowed the
same way, to the kinds that make sense as keys: `IntColumn`, `Int32Column`,
`BigIntColumn`, `StringColumn`, `UUIDColumn`, and their nullable forms. See
[Foreign Keys](/docs/2.fundamentals/3.foreign-keys).

Splitting enums into their own type is what makes `MaxLen` and `Enum` mutually
exclusive: combining them is rejected when a migration is generated, and a type
that offers only one of the two can't reach that state.

```go
orm.NewEnumColumn("status", "order_status", "pending", "done").MaxLen(5)
// compile error: no method MaxLen
```

On `ArrayColumn[T]`, `MaxLen` and `Numeric` apply to the *element*:
`ArrayColumn[string]` with `MaxLen(20)` is `VARCHAR(20)[]`. Being generic over
`T`, it can't narrow which of the two applies the way the scalar types do, so
using the wrong one is still caught when a migration is generated. It's the one
place the typed columns don't fully close that gap.

### JSON, enum, and array columns

`JSONColumn[T]` stores a `T` as a document. It's `JSONB` unless you say
otherwise, matching Postgres's own recommendation:

```go
type Preferences struct {
	Theme string `json:"theme"`
}

Prefs: orm.NewJSONColumn[Preferences]("prefs"),          // JSONB
Raw:   orm.NewJSONColumn[Preferences]("raw").JSON(),     // JSON
```

It offers no comparison operations. Postgres's `json` type has no equality
operator at all, and a `JSONColumn` may be either kind, so an `Eq` here would
be a method that fails at query time for half its uses. Containment and path
operators need query-compiler support that doesn't exist yet.

`ArrayColumn[T]` takes its element type from `T`. Arrays of arrays are rejected
when a migration is generated; Postgres's multi-dimensional arrays are a
different feature and aren't modelled.

A nullable array is backed by `*[]T`, not `[]T`. A nil slice would be the
natural way to spell NULL, but nullability here is decided by whether the Go
type is a pointer, and a slice isn't one, so the entity field has to be a
`*[]T` too.

### Building them inside `DefineTable`

Most kinds have a `TableBuilder` method:

```go
Status: t.Enum("status", "order_status", "pending", "done"),
Score:  t.Double("score"),
Count:  t.Int32("count"),
```

JSON and array columns don't. Their document or element type is a type
parameter, and Go doesn't allow a method to declare one, so they're built with
the package-level constructors inside the same function, exactly as foreign
keys are:

```go
var Users = orm.DefineTable[User]("users", func(t *orm.TableBuilder[User]) *UserModel {
	return &UserModel{
		Table: t.Table(),
		ID:    t.Int("id").PrimaryKey(),
		Prefs: orm.NewJSONColumn[Preferences]("prefs"),
		Tags:  orm.NewArrayColumn[string]("tags"),
	}
})
```

`DefineTable` binds every column it finds on the returned model, however that
column was constructed, so nothing is lost by not going through the builder.

### Query operations

Declaring a column also gives you the operations for building predicates. What
you get depends on the kind, on the same principle:

```go
Users.ID.Gt(100)                  // ordered kinds
Users.Username.StartsWith("a")    // string kinds only
Users.Username.In("alice", "bob") // every kind
Users.CreatedAt.Between(from, to) // ordered kinds
Users.ID.Desc()                   // every kind, for ORDER BY
```

```go
Users.ID.Contains("1")     // compile error: no method Contains
Users.Active.Gt(true)      // compile error: no method Gt
```

`Contains`, `StartsWith`, and `EndsWith` match literally: wildcards in the
argument are escaped, so `Contains("50%")` finds the text `50%` rather than
everything beginning `50`. Use `Like` or `ILike` when you want to write the
pattern yourself.

### Nullable columns take the underlying type

A `NullableStringColumn` wraps a `Column[*string]`, but its operations take a
plain `string`. Comparing against a value is the common case, and needing a
pointer helper to express it would be a papercut on every call:

```go
Users.Email.Eq("alice@example.com") // not Eq(ptr("alice@example.com"))
Users.Email.Set("new@example.com")

Users.Email.IsNull()
Users.Email.IsNotNull()
Users.Email.SetNull()
```

The pointer forms are there when you actually hold one:

```go
Users.Email.EqPtr(emailPtr)  // IS NULL when emailPtr is nil
Users.Email.SetPtr(emailPtr) // SQL NULL when emailPtr is nil
```

`EqPtr(nil)` becomes `IS NULL` deliberately. `col = NULL` is never true in SQL,
so compiling it literally would turn a nil pointer into a query that silently
matches nothing.

`IsNull` and `IsNotNull` exist only on the nullable types, so
`Users.Username.IsNull()` on a non-nullable column doesn't compile.
