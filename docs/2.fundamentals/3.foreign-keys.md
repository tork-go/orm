# Foreign Keys

A foreign key isn't a special kind of column. It's an ordinary column that
references another one:

```go
AuthorID *orm.IntColumn

AuthorID: t.Int("author_id").NotNull().References(Users.ID),
```

That's the whole declaration. The referenced table comes from the column you
point at, so you never repeat `"users"`, and the constraint name is derived
rather than spelled out.

## What gets generated

```sql
CONSTRAINT "fk_posts_author_id" FOREIGN KEY ("author_id") REFERENCES "users" ("id")
```

The name follows `fk_<table>_<columns>`, the same deterministic scheme every
other constraint uses. You never write it, and two people declaring the same
key get the same name. See
[Constraints](/docs/2.fundamentals/4.constraints) for the full set of naming
rules.

## Types have to match

`References` won't accept a column of a different type:

```go
t.String("author_id").References(Users.ID)
// cannot use Users.ID (variable of type *orm.IntColumn) as orm.Ref[string]:
//     have Base() *orm.Column[int]
//     want Base() *orm.Column[string]
```

A **nullable** key onto a non-nullable primary key is the ordinary shape of an
optional relationship, and is accepted — the key is `*int`, the target is
`int`:

```go
AuthorID: t.NullableInt("author_id").References(Users.ID),
```

Only kinds that make sense as keys have `References` at all: `Int`, `Int32`,
`BigInt`, `String`, and `UUID`, plus their nullable forms. `t.Bool(...)` and
`t.StringArray(...)` simply don't have the method.

## Keys over more than one column

`References` attaches a key to one column, so a key spanning several is
declared at the table level, the same way a compound index is:

```go
func (m *OrderLineModel) ForeignKeys() []orm.ForeignKeyDef {
	return []orm.ForeignKeyDef{
		orm.NewForeignKeyDef(m.OrgID, m.OrderID).
			References(Orders.OrgID, Orders.ID).
			OnDelete(orm.ActionCascade),
	}
}
```

Column order carries the pairing on both sides: the first column references the
first referenced column, and so on. A mismatched count is rejected rather than
left to pair the wrong columns silently.

`ReferencesTable("orders", "org_id", "id")` names the target directly, for a
table `References` can't name.

The name is derived as usual, `fk_order_lines_org_id_order_id`.

## Referencing a table you can't name

`References` reads the target's table from the target column, which means the
target has to belong to a table — it has to come from a model declared with
`DefineTable`. For anything else, spell the reference out:

```go
AuthorID: t.Int("author_id").ReferencesTable("legacy_authors", "author_pk"),
```

Use this for a table managed outside your program, or for a model built with
`NewTable` (whose columns are never bound to a table and so can't report one).

::alert{type="warning"}
If you pass `References` a column that was never bound to a table, it reports
an empty referenced table — which is indistinguishable from no reference at
all, so no constraint is generated and nothing errors. Reach for
`ReferencesTable` when the target isn't from a `DefineTable` model.
::

## Self-references

A column can reference one declared beside it in the same table. Hold the
target in a local variable:

```go
var Categories = orm.DefineTable[Category]("categories",
	func(t *orm.TableBuilder[Category]) *CategoryModel {
		id := t.Int("id").PrimaryKey()
		return &CategoryModel{
			Table:    t.Table(),
			ID:       id,
			ParentID: t.NullableInt("parent_id").References(id),
		}
	})
```

This works because the target is resolved when it's read, not when
`References` is called — at the point the literal is evaluated, neither column
has been bound to the table yet.

## Referential actions

`.OnDelete` and `.OnUpdate` declare what the database should do to this row
when the referenced row is deleted or updated:

```go
AuthorID: t.Int("author_id").References(Users.ID).OnDelete(orm.ActionCascade),
```

`ForeignKeyAction` values: `ActionNoAction` (the zero value, the database's own
default, renders no clause at all), `ActionCascade`, `ActionSetNull`,
`ActionSetDefault`, `ActionRestrict`.

## Reading it back

```go
Posts.AuthorID.ReferencedTable()  // "users"
Posts.AuthorID.ReferencedColumn() // "id"
Posts.AuthorID.OnDeleteAction()   // orm.ActionCascade
```

A column that references nothing reports `""` from both, which is exactly what
distinguishes a foreign key from an ordinary column. `orm.ForeignKeys(model)`
filters on it:

```go
orm.Columns(Posts)      // every column, including author_id
orm.ForeignKeys(Posts)  // only author_id
```

Every column satisfies `ForeignKeyMeta`, since any column *may* carry a
reference. Satisfying the interface isn't what makes a field a foreign key —
having a referenced table is.

## Relationships are a separate thing

Declaring a foreign key doesn't load related rows. That's what `HasMany` /
`BelongsTo` are for, and loading them is not built yet — a relationship reports
which columns a join would match on, but nothing fetches them. See
[Relationships](/docs/2.fundamentals/5.relationships).
