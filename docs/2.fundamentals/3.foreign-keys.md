# Foreign Keys

A `*ForeignKey[T]` field is a column that references a column in another table.
It embeds `Column[T]`, so everything from
[Columns](/docs/2.fundamentals/2.columns), `PrimaryKey`, `Unique`, `NotNull`,
`MaxLen`, and the read accessors, applies to it too.

```go
AuthorID *orm.ForeignKey[int]

AuthorID: orm.NewForeignKey("author_id", User.TableName(), User.ID),
```

## Type inference

`NewForeignKey`'s signature is:

```go
func NewForeignKey[T any](name, refTableName string, refColumn *Column[T]) *ForeignKey[T]
```

`T` is inferred from `refColumn`, the column being referenced, not written
explicitly. `User.ID` is `*Column[int]`, so `NewForeignKey("author_id",
User.TableName(), User.ID)` produces a `*ForeignKey[int]`, matching the
`AuthorID *orm.ForeignKey[int]` field it's assigned to. If the referenced
column's type doesn't match the field's declared type, that's a compile error,
not a runtime surprise.

## Referenced table and column

```go
ReferencedTable() string  // "users"
ReferencedColumn() string // "id"
```

Only the table and column *names* are stored, as strings, not a live pointer
back to the `User` model. A foreign key never holds a reference into another
model's object graph.

## Chaining constraints on a foreign key

Because `ForeignKey[T]` embeds `Column[T]`, you'd expect `fk.NotNull()` to work,
and it does, but not by simple embedding. Go's method promotion keeps the
embedded type's return type, so a naively promoted `NotNull()` would return
`*Column[int]`, not `*ForeignKey[int]`, breaking further chaining. Tork
re-declares all four builder methods on `ForeignKey[T]` itself so the chain
stays a `*ForeignKey[T]`:

```go
AuthorID: orm.NewForeignKey("author_id", User.TableName(), User.ID).NotNull(),
```

You don't need to think about this while using the library; it's mentioned here
because if you ever wonder why `foreign_key.go` re-declares methods that look
identical to `Column`'s, that's why.

## What this doesn't do yet

Declaring a `ForeignKey` doesn't load related rows. That's what
`HasMany`/`BelongsTo` and a query API are for, and the query API isn't built
yet; see [Relationships](/docs/2.fundamentals/5.relationships). What a
`ForeignKey` does today is participate in migrations: generating the `ADD
FOREIGN KEY` constraint, and letting the diff engine detect when a reference
changes.
