# Constraints

`PrimaryKey`, `Unique`, `Index`, `NotNull`, and `MaxLen` are set per column
(see [Columns](/docs/2.fundamentals/2.columns) for the basics). This page
covers what happens when Tork turns those calls into a migration: what's
supported, what's compound, how names get generated, and what gets rejected.

## Primary keys, including composite ones

Calling `.PrimaryKey()` on more than one column in a model produces a composite
primary key, columns collected in the order they're declared on the struct:

```go
type OrderItemModel struct {
	orm.Table
	OrderID   *orm.Column[int]
	ProductID *orm.Column[int]
	Quantity  *orm.Column[int]
}

var OrderItem = &OrderItemModel{
	Table:     orm.NewTable("order_items"),
	OrderID:   orm.NewColumn[int]("order_id").PrimaryKey(),
	ProductID: orm.NewColumn[int]("product_id").PrimaryKey(),
	Quantity:  orm.NewColumn[int]("quantity").NotNull(),
}
```

This generates `CONSTRAINT "pk_order_items" PRIMARY KEY ("order_id",
"product_id")`.

A **single**-column integer or bigint primary key is treated differently: it
becomes an auto-incrementing identity column (`GENERATED ALWAYS AS IDENTITY`),
inlined into the column definition rather than a separate constraint. A
composite key, or a single-column key on a non-integer type, is always a
separate `PRIMARY KEY` constraint with no identity behavior. There's currently
no way to declare a single-column integer primary key that *isn't*
auto-incrementing; if you need that, it's not supported yet.

## Names are generated for you, but never mandatory

Every constraint and index Tork creates gets a deterministic name, so
regenerating a migration against an already-migrated database never proposes
renaming something that already matches: `pk_<table>` for primary keys,
`uq_<table>_<columns>` for unique constraints, `ix_<table>_<columns>` for
plain indexes, `fk_<table>_<columns>` for foreign keys. You've already seen
this: `.PrimaryKey()`/`.Unique()`/`.Index()` on a column never take a name
argument, one is always generated. For the compound case below, a name is
still generated by default, and you can override it if you want something
else.

## Compound unique constraints and plain indexes

A single column's `.Unique()` or `.Index()` only covers that one column. For
a constraint or index spanning more than one column, implement `Indexes()` on
your model, returning `[]orm.IndexDef`:

```go
type OrgMemberModel struct {
	orm.Table
	OrgID     *orm.Column[int]
	UserID    *orm.Column[int]
	CreatedAt *orm.Column[string]
}

func (m *OrgMemberModel) Indexes() []orm.IndexDef {
	return []orm.IndexDef{
		orm.NewIndexDef(m.OrgID, m.UserID).Unique(), // a member can't join an org twice
		orm.NewIndexDef(m.OrgID, m.CreatedAt),       // fast "members of this org, newest first"
	}
}
```

`NewIndexDef` takes the columns in order and produces a plain index by
default; chain `.Unique()` for a compound unique constraint instead. Naming
follows the same convention as everything else, auto-generated
(`uq_org_members_org_id_user_id`, `ix_org_members_org_id_created_at`) unless
you override it with `.Named(...)`:

```go
orm.NewIndexDef(m.OrgID, m.UserID).Unique().Named("uq_one_membership_per_org"),
```

Implementing `Indexes()` is entirely optional, most models won't need it. See
[Model Reflection](/docs/2.fundamentals/6.model-reflection) for how Tork
detects and calls it.

## CHECK constraints

For a raw SQL boolean condition a column-level builder can't express,
implement `Checks()` on your model, returning `[]orm.CheckDef`:

```go
type AccountModel struct {
	orm.Table
	Balance *orm.Column[int]
}

func (m *AccountModel) Checks() []orm.CheckDef {
	return []orm.CheckDef{
		orm.NewCheckDef("balance >= 0"),
	}
}
```

`NewCheckDef` takes a raw SQL expression, not parsed or validated by Tork,
the same spirit as `.ServerDefault`. Naming is different from every other
constraint here: since an expression has no natural column list to derive a
name from, an unnamed check gets a **positional** name, `ck_<table>_<n>`,
the 1-based ordinal among only the unnamed entries in `Checks()`, in
declaration order. Reordering `Checks()` therefore changes an unnamed
check's auto-generated name on the next `makemigrations` run, unlike
`ix_`/`uq_` names, which are derived from a column set and stay stable
under reordering. Name a check explicitly with `.Named(...)` if that
matters to you:

```go
orm.NewCheckDef("balance >= 0").Named("ck_accounts_balance_non_negative"),
```

Implementing `Checks()` is entirely optional, like `Indexes()`.

**Diffing risk.** Postgres can reformat a CHECK expression when storing it,
most simply by adding a wrapping layer of parentheses, but sometimes more
aggressively (rewriting `IN (...)` to `= ANY (ARRAY[...])`, or `LIKE` to the
`~~` operator, or injecting an explicit type cast for a comparison against a
differently-typed column). `makemigrations` compares the introspected
expression against your declared one after a light normalization
(whitespace collapsed, one layer of wrapping parens stripped), which is
enough for a simple single-comparison expression, but not for compound
(`AND`/`OR`) or operator-rewritten expressions, those can show up as a
spurious "changed" constraint on every subsequent run. This is a documented,
tested limitation (see
[The Postgres Driver](/docs/4.architecture/2.postgres-driver)), not silently
accepted; if you hit it, the workaround is the same as for `ServerDefault`,
hand-edit the migration.

## `MaxLen` validation

`MaxLen` is checked when a migration is generated (in `schema.ExtractSchema`,
called by `makemigrations`), not when you call it. Two rules:

- **Used on a non-string column.** `NewColumn[int]("n").MaxLen(10)` is rejected.
  `MaxLen` only makes sense for text.
- **Not a positive number, on a string column.** `MaxLen(0)` or a negative
  value is rejected. If you didn't mean to bound the length at all, don't call
  `MaxLen`, the column becomes `TEXT`.

Both surface as an error from `MakeMigrations`/`makemigrations`, naming the
table and column, before any SQL is generated:

```
cli: extracting desired schema: table "users": column "age": MaxLen is only valid on string columns
```

## `ServerDefault` versus auto-incrementing primary keys

A single-column integer or bigint primary key is always rendered as an
identity column (see above), and Postgres doesn't allow an identity column to
also have an explicit `DEFAULT`. Calling `.ServerDefault(...)` on one is
rejected the same way an invalid `MaxLen` is:

```
cli: extracting desired schema: table "orders": column "id": ServerDefault cannot be combined with a single-column integer primary key, Postgres renders it as GENERATED ALWAYS AS IDENTITY, which does not allow an explicit DEFAULT
```

This only applies to a *single-column integer or bigint* primary key. A
composite primary key, or a primary key of any other type (a UUID primary key
with `.ServerDefault("gen_random_uuid()")`, for instance), is unaffected. See
[Columns](/docs/2.fundamentals/2.columns) for more on `ServerDefault`.

## What's not validated yet

Tork doesn't currently check for every conceivable conflicting constraint
within a model. Postgres itself will reject genuinely invalid DDL when a
migration runs; not every mistake is caught earlier than that yet.
