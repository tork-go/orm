# Relationships

`HasMany[T]` and `BelongsTo[T]` mark the two directions of a one-to-many
relationship. `T` is the related model:

```go
type UserModel struct {
	orm.Table
	ID    *orm.Column[int]
	Posts orm.HasMany[PostModel]
}

type PostModel struct {
	orm.Table
	ID       *orm.Column[int]
	AuthorID *orm.ForeignKey[int]
	Author   orm.BelongsTo[UserModel]
}
```

## They're markers, not loaders

`HasMany[T]` and `BelongsTo[T]` are empty structs:

```go
type HasMany[T any] struct{}
type BelongsTo[T any] struct{}
```

They carry no data and have no methods. Notice `Posts` and `Author` are left
uninitialized in the composite literals above, that's intentional; a
zero-valued `HasMany`/`BelongsTo` is exactly what you want. Nothing about
declaring one causes a query, a join, or any database access. What actually
loads related rows is a query API, which is planned but not built yet.

## Why declare them now, then

Two reasons. First, they document the relationship for anyone reading the
model, the same way a comment would, but checked by the compiler (`T` has to
be an actual model type). Second, when the query API arrives, it will extend
these types to do real work using `HasMany[T]`/`BelongsTo[T]`'s own type
parameter plus the `ForeignKey` that already exists on the other side. Declaring
the field now means you won't need to touch these structs again to start using
relationship loading later.

## The actual link is the `ForeignKey`

The relationship that migrations and (eventually) queries act on is the
`ForeignKey` field, covered in
[Foreign Keys](/docs/2.fundamentals/3.foreign-keys), not the `HasMany`/
`BelongsTo` markers. `PostModel.AuthorID` is what generates the `FOREIGN KEY`
constraint; `PostModel.Author` and `UserModel.Posts` are purely descriptive
today.
