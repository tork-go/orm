# Relationships

A relationship field says that rows of one table relate to rows of another:

```go
type UserModel struct {
	orm.Table[User]
	ID    *orm.IntColumn
	Posts orm.HasMany[Post]
}

type PostModel struct {
	orm.Table[Post]
	ID       *orm.IntColumn
	AuthorID *orm.IntColumn
	Author   orm.BelongsTo[User]
}
```

The type parameter is the **row type** on the other side, not the model.

| Marker | Meaning | Where the key lives |
| --- | --- | --- |
| `HasMany[E]` | many related rows | on the related table |
| `HasOne[E]` | at most one related row | on the related table |
| `BelongsTo[E]` | this row points at one | on this table |
| `ManyToMany[E]` | through a join table | on the join table, named with `Through` |

## The key is worked out from the schema

You don't say which foreign key joins the two tables. Tork finds the one
column on the far side that references this table:

```go
info, err := Users.Posts.Relation()

info.LocalTable    // "users"
info.LocalColumn   // users.id
info.ForeignTable  // "posts"
info.ForeignColumn // posts.author_id
```

`BelongsTo` is the mirror, since it owns the key:

```go
info, _ := Posts.Author.Relation()
info.LocalColumn   // posts.author_id
info.ForeignColumn // users.id
```

Resolution happens the first time a relationship is used, and the answer is
cached. If no column joins the two tables, that's reported:

```
orm: users.HasMany -> posts: no column on posts references users;
    declare one with References, or name the key with a Relations method
```

## Leave them uninitialized

Relationship fields are left out of the model literal on purpose:

```go
var Users = orm.DefineTable[User]("users", func(t *orm.TableBuilder[User]) *UserModel {
	return &UserModel{
		Table: t.Table(),
		ID:    t.Int("id").PrimaryKey(),
		// Posts is not set here.
	}
})
```

That's what lets `UserModel` mention `Post` even though `Posts` is declared
after it. There is deliberately no way to bind a marker in the literal,
because doing so would mean naming the other table there:

```go
var Users = orm.DefineTable[User]("users", func(t *orm.TableBuilder[User]) *UserModel {
	return &UserModel{
		Table: t.Table(),
		ID:    t.Int("id").PrimaryKey(),
		// Anything referring to Posts here would not compile. Posts refers
		// to Users.ID for its foreign key, so the two package-level
		// variables would depend on each other, and Go rejects an
		// initialization cycle outright.
	}
})
```

This is also why resolution is deferred to first use rather than done while
the tables are being declared, and why `Relations` is a method: at
declaration time the other table may not exist yet.

## When two keys join the same tables

Inference needs exactly one candidate. Two columns referencing the same table
give it nothing to choose on:

```
orm: authors.HasMany -> articles: articles has 2 columns referencing authors
    ("author_id", "editor_id"); name the one this relationship uses with a
    Relations method
```

Name it with a `Relations` method:

```go
func (m *ArticleModel) Relations() []orm.RelationDef {
	return []orm.RelationDef{
		orm.Via(&m.Author, m.AuthorID),
		orm.Via(&m.Editor, m.EditorID),
	}
}
```

A model implementing `orm.Relater` has its relationships named this way; any
it doesn't name still fall through to inference.

It's a **method** rather than something declared beside the columns for the
same reason relationships resolve late: a method body isn't part of any
variable's initializer, so it can mention another table without closing an
initialization cycle. `Relations` is read the first time a relationship is
used, not while the model is being declared.

## `ManyToMany` needs its join table named

Every other shape is inferred from the schema. A many-to-many can't be: a
`ManyToMany[Role]` names the far side but not the table joining the two, and no
two tables imply a particular join table.

Name the join table's two keys with `Through`:

```go
func (m *UserModel) Relations() []orm.RelationDef {
	return []orm.RelationDef{
		orm.Through(&m.Roles, UserRoles.UserID, UserRoles.RoleID),
	}
}
```

That settles the join table as well, since a column knows the table it belongs
to. Both keys must come from the same one, and each must reference the side it
stands for.

The result carries both hops:

```go
info, _ := Users.Roles.Relation()

info.LocalTable        // "users"
info.LocalColumn       // users.id
info.LocalJoinColumn   // user_roles.user_id
info.JoinTable         // "user_roles"
info.ForeignJoinColumn // user_roles.role_id
info.ForeignColumn     // roles.id
info.ForeignTable      // "roles"
```

Leaving one unnamed reports that rather than guessing:

```
orm: users.ManyToMany -> models.Role: no join table named; nothing in the
    declaration says which table joins the two, so name the join table's two
    keys with Through in a Relations method
```

## Worked examples

Each of these is a whole schema, small enough to read at once. The comment
under each shows what `Relation()` resolves to.

### One to many: an author and their posts

The commonest shape. `posts.author_id` is the only key between the two tables,
so neither side needs a `Relations` method.

```go
type Author struct {
	ID   int
	Name string
}

type AuthorModel struct {
	orm.Table[Author]
	ID    *orm.IntColumn
	Name  *orm.StringColumn
	Posts orm.HasMany[Post]
}

var Authors = orm.DefineTable[Author]("authors", func(t *orm.TableBuilder[Author]) *AuthorModel {
	return &AuthorModel{
		Table: t.Table(),
		ID:    t.Int("id").PrimaryKey(),
		Name:  t.String("name").NotNull(),
	}
})

type Post struct {
	ID        int
	AuthorID  int
	Title     string
	CreatedAt time.Time
}

type PostModel struct {
	orm.Table[Post]
	ID        *orm.IntColumn
	AuthorID  *orm.IntColumn
	Title     *orm.StringColumn
	CreatedAt *orm.TimeColumn
	Author    orm.BelongsTo[Author]
}

var Posts = orm.DefineTable[Post]("posts", func(t *orm.TableBuilder[Post]) *PostModel {
	return &PostModel{
		Table:     t.Table(),
		ID:        t.Int("id").PrimaryKey(),
		AuthorID:  t.Int("author_id").NotNull().References(Authors.ID).OnDelete(orm.ActionCascade),
		Title:     t.String("title").NotNull().MaxLen(200),
		CreatedAt: t.Time("created_at").NotNull().ServerDefault("now()"),
	}
})

// Authors.Posts   authors.id       -> posts.author_id
// Posts.Author    posts.author_id  -> authors.id
```

### One to one: a user and their profile

The same shape as one-to-many, with the key made `Unique()`. That uniqueness is
what makes it one-to-one; `HasOne` says the same thing to a reader.

```go
type User struct {
	ID    int
	Email string
}

type UserModel struct {
	orm.Table[User]
	ID      *orm.IntColumn
	Email   *orm.StringColumn
	Profile orm.HasOne[Profile]
}

var Users = orm.DefineTable[User]("users", func(t *orm.TableBuilder[User]) *UserModel {
	return &UserModel{
		Table: t.Table(),
		ID:    t.Int("id").PrimaryKey(),
		Email: t.String("email").Unique().NotNull(),
	}
})

type Profile struct {
	ID     int
	UserID int
	Bio    *string
}

type ProfileModel struct {
	orm.Table[Profile]
	ID     *orm.IntColumn
	UserID *orm.IntColumn
	Bio    *orm.NullableStringColumn
	User   orm.BelongsTo[User]
}

var Profiles = orm.DefineTable[Profile]("profiles", func(t *orm.TableBuilder[Profile]) *ProfileModel {
	return &ProfileModel{
		Table:  t.Table(),
		ID:     t.Int("id").PrimaryKey(),
		UserID: t.Int("user_id").Unique().NotNull().References(Users.ID).OnDelete(orm.ActionCascade),
		Bio:    t.NullableString("bio"),
	}
})

// Users.Profile   users.id         -> profiles.user_id
// Profiles.User   profiles.user_id -> users.id
```

### Many to many: posts and tags

Three tables. The join table is an ordinary model with two keys, and needs no
relationship markers of its own. Each side names the two keys with `Through`,
in its own order: local key first, then the one pointing at the far side.

```go
type Tag struct {
	ID   int
	Name string
}

type TagModel struct {
	orm.Table[Tag]
	ID    *orm.IntColumn
	Name  *orm.StringColumn
	Posts orm.ManyToMany[Post]
}

func (m *TagModel) Relations() []orm.RelationDef {
	return []orm.RelationDef{orm.Through(&m.Posts, PostTags.TagID, PostTags.PostID)}
}

var Tags = orm.DefineTable[Tag]("tags", func(t *orm.TableBuilder[Tag]) *TagModel {
	return &TagModel{
		Table: t.Table(),
		ID:    t.Int("id").PrimaryKey(),
		Name:  t.String("name").Unique().NotNull(),
	}
})

type PostTag struct {
	PostID int
	TagID  int
}

type PostTagModel struct {
	orm.Table[PostTag]
	PostID *orm.IntColumn
	TagID  *orm.IntColumn
}

var PostTags = orm.DefineTable[PostTag]("post_tags", func(t *orm.TableBuilder[PostTag]) *PostTagModel {
	return &PostTagModel{
		Table:  t.Table(),
		PostID: t.Int("post_id").NotNull().References(Posts.ID).OnDelete(orm.ActionCascade),
		TagID:  t.Int("tag_id").NotNull().References(Tags.ID).OnDelete(orm.ActionCascade),
	}
})
```

The other side goes on `PostModel` from the first example: one more field, and
a `Relations` method naming the same two keys the other way round.

```go
type PostModel struct {
	orm.Table[Post]
	ID        *orm.IntColumn
	AuthorID  *orm.IntColumn
	Title     *orm.StringColumn
	CreatedAt *orm.TimeColumn
	Author    orm.BelongsTo[Author]
	Tags      orm.ManyToMany[Tag] // added
}

func (m *PostModel) Relations() []orm.RelationDef {
	return []orm.RelationDef{orm.Through(&m.Tags, PostTags.PostID, PostTags.TagID)}
}

// Posts.Tags   posts.id -> post_tags.post_id ... post_tags.tag_id -> tags.id
// Tags.Posts   tags.id  -> post_tags.tag_id  ... post_tags.post_id -> posts.id
```

Note the two `Through` calls list the same two columns in opposite orders. The
first argument is always the key pointing back at the table declaring the
relationship.

### Self-reference: a category tree

A table can reference itself. Hold the key column in a local variable, since
the model doesn't exist yet inside its own builder:

```go
type Category struct {
	ID       int
	Name     string
	ParentID *int
}

type CategoryModel struct {
	orm.Table[Category]
	ID       *orm.IntColumn
	Name     *orm.StringColumn
	ParentID *orm.NullableIntColumn
	Parent   orm.BelongsTo[Category]
	Children orm.HasMany[Category]
}

var Categories = orm.DefineTable[Category]("categories", func(t *orm.TableBuilder[Category]) *CategoryModel {
	id := t.Int("id").PrimaryKey()
	return &CategoryModel{
		Table:    t.Table(),
		ID:       id,
		Name:     t.String("name").NotNull(),
		ParentID: t.NullableInt("parent_id").References(id),
	}
})

// Categories.Parent     categories.parent_id -> categories.id
// Categories.Children   categories.id        -> categories.parent_id
```

Both directions resolve without a `Relations` method, because `parent_id` is
the only key from `categories` to itself. A second self-reference (an
`owner_id` alongside `parent_id`, say) would make both ambiguous and need one.

`ParentID` is nullable, which is what lets a root category have no parent.

### Two keys to the same table

A message has a sender and a recipient, both users. Inference has two
candidates and no way to choose, so name them:

```go
type Message struct {
	ID          int
	SenderID    int
	RecipientID int
	Body        string
}

type MessageModel struct {
	orm.Table[Message]
	ID          *orm.IntColumn
	SenderID    *orm.IntColumn
	RecipientID *orm.IntColumn
	Body        *orm.StringColumn
	Sender      orm.BelongsTo[User]
	Recipient   orm.BelongsTo[User]
}

func (m *MessageModel) Relations() []orm.RelationDef {
	return []orm.RelationDef{
		orm.Via(&m.Sender, m.SenderID),
		orm.Via(&m.Recipient, m.RecipientID),
	}
}

var Messages = orm.DefineTable[Message]("messages", func(t *orm.TableBuilder[Message]) *MessageModel {
	return &MessageModel{
		Table:       t.Table(),
		ID:          t.Int("id").PrimaryKey(),
		SenderID:    t.Int("sender_id").NotNull().References(Users.ID),
		RecipientID: t.Int("recipient_id").NotNull().References(Users.ID),
		Body:        t.String("body").NotNull(),
	}
})

// Messages.Sender      messages.sender_id    -> users.id
// Messages.Recipient   messages.recipient_id -> users.id
```

Without the `Relations` method both would report the ambiguity rather than
picking one:

```
orm: messages.BelongsTo -> users: messages has 2 columns referencing users
    ("sender_id", "recipient_id"); name the one this relationship uses with a
    Relations method
```

Note `Via` takes the model's own column, `m.SenderID`, not the referenced one.
The key is the column that holds the reference.

## Loading related rows

`Load` fetches them, into the field of the row type the marker is named after:

```go
authors, err := Authors.With(db).Load(Authors.Posts).All(ctx)
// authors[0].Posts is filled
```

That needs the row type to carry the field:

```go
type Author struct {
    ID    int
    Name  string
    Posts []Post   // matches no column, so the entity mapping ignores it
}
```

See [Querying](/docs/2.fundamentals/8.querying) for narrowing, ordering and
nesting a load, and for why it is a second query rather than a join.

Declaring a relationship today costs nothing, has no effect on migrations
(relationship fields are not columns and are skipped by
[schema extraction](/docs/2.fundamentals/6.model-reflection)), and means you
won't need to revisit these structs when loading lands.
