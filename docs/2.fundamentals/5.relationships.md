# Relationships

Four types mark the shape of a relationship between models. `T` is always the
related model:

| Type | Marks |
|---|---|
| `HasMany[T]` | The "many" side of a one-to-many, doesn't own the foreign key. |
| `BelongsTo[T]` | The "one" side of a one-to-many or one-to-one, owns the foreign key. |
| `HasOne[T]` | The non-owning side of a one-to-one: at most one related row. |
| `ManyToMany[T]` | Either side of a many-to-many, through a join table. |

```go
type UserModel struct {
	orm.Table
	ID      *orm.Column[int]
	Posts   orm.HasMany[PostModel]
	Profile orm.HasOne[ProfileModel]
	Roles   orm.ManyToMany[RoleModel]
}

type PostModel struct {
	orm.Table
	ID       *orm.Column[int]
	AuthorID *orm.ForeignKey[int]
	Author   orm.BelongsTo[UserModel]
}

type ProfileModel struct {
	orm.Table
	ID     *orm.Column[int]
	UserID *orm.ForeignKey[int]
	User   orm.BelongsTo[UserModel]
}
```

`HasOne` exists as its own type, rather than reusing `HasMany`, specifically
so a model documents "at most one related row" instead of "many": the
distinction between `UserModel.Profile` (one) and `UserModel.Posts` (many) is
meaningful even though nothing enforces it yet.

## They're markers, not loaders

All four are empty structs:

```go
type HasMany[T any] struct{}
type BelongsTo[T any] struct{}
type HasOne[T any] struct{}
type ManyToMany[T any] struct{}
```

They carry no data and have no methods. Notice `Posts`, `Profile`, `Roles`,
and `Author` are left uninitialized in the composite literals above, that's
intentional; a zero-valued marker is exactly what you want. Nothing about
declaring one causes a query, a join, or any database access. What actually
loads related rows is a query API, which is planned but not built yet.

## Why declare them now, then

Two reasons. First, they document the relationship for anyone reading the
model, the same way a comment would, but checked by the compiler (`T` has to
be an actual model type). Second, when the query API arrives, it will extend
these types to do real work. Declaring the field now means you won't need to
touch these structs again to start using relationship loading later.

## `ManyToMany` doesn't carry a join table yet

Unlike the other three, a real many-to-many relationship needs more than a
marker to actually load: a join table name and the two foreign key columns on
it. `ManyToMany[T]` deliberately doesn't carry any of that yet. Adding it now
would mean guessing at a shape before the query API that will actually consume
it exists, and likely redesigning it once real requirements are known. For
now, model the join table itself as an ordinary model with two `ForeignKey`
fields:

```go
type UserRoleModel struct {
	orm.Table
	UserID *orm.ForeignKey[int]
	RoleID *orm.ForeignKey[int]
}

func (m *UserRoleModel) Indexes() []orm.IndexDef {
	return []orm.IndexDef{orm.NewIndexDef(m.UserID, m.RoleID).Unique()}
}
```

See [Constraints](/docs/2.fundamentals/4.constraints) for `Indexes()` and
compound unique constraints.

## The actual link is the `ForeignKey`

The relationship that migrations and (eventually) queries act on is the
`ForeignKey` field, covered in
[Foreign Keys](/docs/2.fundamentals/3.foreign-keys), not the relationship
markers. `PostModel.AuthorID` is what generates the `FOREIGN KEY` constraint;
`PostModel.Author` and `UserModel.Posts` are purely descriptive today.
