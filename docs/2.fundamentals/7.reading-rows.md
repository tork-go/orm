# Reading Rows

`DefineTable` works out which field of the row type each column scans into.
That mapping is what turns a result row back into a struct.

The query API isn't built yet, so there's no `Users.Query(db)` to hand you
rows. What exists is the piece underneath it, which you can also use directly
against SQL you wrote yourself.

## Column order is the mapping

```go
Users.Columns() // the columns, in the order they were declared on the model
```

That order isn't cosmetic. `Rows` has no way to report the result's column
names, so scanning is **positional**: the only thing tying a value in the
result back to a field is that both sides walk the same list. A `SELECT` Tork
generates lists columns in this order, which is why `ScanRow` can fill in the
right fields without being told anything about the result.

## Scanning a row

```go
rows, err := conn.Query(ctx, `SELECT id, username, email FROM users`)
if err != nil {
	return err
}
defer rows.Close()

for rows.Next() {
	user, err := Users.ScanRow(rows)
	if err != nil {
		return err
	}
	fmt.Println(user.Username)
}
return rows.Err()
```

You advance `rows`; `ScanRow` reads whatever row `Next` last moved to. The
loop stays yours to write.

The `SELECT` list has to match `Columns()` in order and count, since that's
what the destinations are built from. Getting it wrong is reported rather
than silently filling in the wrong fields.

## What it handles

**Fields promoted from an embedded struct** are filled in exactly like direct
ones:

```go
type Timestamps struct{ CreatedAt, UpdatedAt time.Time }

type Article struct {
	Timestamps
	ID    int
	Title string
}
```

`created_at` finds `Timestamps.CreatedAt` without a tag.

An embedded **pointer** works too, and is allocated on the way in if it's nil:

```go
type Article struct {
	*Timestamps
	ID int
}
```

The embedded type has to be exported for that, since filling it in means
assigning to the pointer field and reflect refuses that for an unexported one.
An embedded *value* only needs its leaf field exported, because nothing along
the way is assigned to.

**Nullable columns** leave the field at its zero value for a `NULL`, which for
a nullable column is the nil pointer `NULL` means.

**Document columns** can't be scanned into their Go type directly, since the
database hands back encoded bytes. They're staged through a `[]byte` and then
decoded with the column's own codec, so a custom `Serialize` pair applies on
the way in as well as on the way out:

```go
Prefs: orm.NewJSONColumn[Preferences]("prefs").Serialize(marshal, unmarshal),
```

## Models without a row type

A model built with `NewTable` has no entity mapping, so there's nothing to
scan into:

```
orm: table "legacy": no entity mapping, so rows cannot be scanned;
    declare the model with DefineTable rather than NewTable
```

See [Tables](/docs/2.fundamentals/1.tables) for when that's the right choice.

## `ValueCodec`

Encoding and decoding go through the column, not through `encoding/json`
directly:

```go
type ValueCodec interface {
	GenerateAny() (any, bool)
	MarshalAny(v any) ([]byte, error)
	UnmarshalAny(b []byte) (any, error)
}
```

Every column satisfies it. It exists because `Generator` and `Serializer` both
mention the column's `T`, so neither can be part of `ColumnMeta`, which is the
interface everything generic works through. These three say the same things
without naming `T`.

`MarshalAny` and `UnmarshalAny` fall back to `encoding/json` when no
`Serialize` pair was set, which is what a `JSONB` column declared without one
uses. `GenerateAny` returns a value from `GeneratedByClient`, or `false` if the
column has no generator.

You rarely call these yourself; they're how the ORM handles values when all it
holds is a `ColumnMeta`.
