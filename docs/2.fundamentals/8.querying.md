# Querying

Every query starts from a model and a database handle:

```go
db, err := orm.Connect(ctx, os.Getenv("DATABASE_URL"))

users, err := Users.With(db).Where(Users.Age.Gte(18)).All(ctx)
```

The examples on this page use one model throughout:

```go
type User struct {
    ID        int
    Username  string
    Email     *string
    Age       int
    Active    bool
    CreatedAt time.Time
}

type UserModel struct {
    orm.Table[User]
    ID        *orm.IntColumn
    Username  *orm.StringColumn
    Email     *orm.NullableStringColumn
    Age       *orm.IntColumn
    Active    *orm.BoolColumn
    CreatedAt *orm.TimeColumn
}
```

## Two types, one chain

`With(db)` gives you a `*Query[User]`. Naming a condition, an order, or a limit
turns it into a `*Filtered[User]`:

```go
Users.With(db)                          // *Query[User]
Users.With(db).Where(Users.Age.Gt(18))  // *Filtered[User]
```

They carry different operations, and that is the point:

| | `Query[E]` | `Filtered[E]` |
| --- | --- | --- |
| Reading | `All` `First` `Count` `Exists` `Find` | `All` `First` `Count` `Exists` |
| One row you hold | `Insert` `Update` `Save` `Delete` | — |
| Many rows you hold | `InsertMany` `UpdateMany` `DeleteMany` | — |
| Every matching row | `UpdateAll` `DeleteAll` | `UpdateAll` `DeleteAll` |

A filter cannot change which row a `*User` identifies, so combining the two never
meant anything. Writing one before the other is a compile error rather than
something reported at run time:

```go
Users.With(db).Where(Users.Age.Gt(18)).Insert(ctx, user)
// compile error: Insert is not a method on *orm.Filtered[User]
```

## Reading

### `All`

```go
users, err := Users.With(db).Where(Users.Active.Eq(true)).All(ctx)
```

Returns `[]*User`. An empty result is an empty slice, not an error.

### `First`

```go
user, err := Users.With(db).Where(Users.Username.Eq("alice")).First(ctx)
if errors.Is(err, orm.ErrNoRows) {
    // nothing matched
}
```

Adds `LIMIT 1` to a copy of the query, so the query you called it on keeps its own
limit.

### `Find`

```go
user, err := Users.With(db).Find(ctx, 42)   // by primary key
```

The key is type-checked against the primary key column before the statement is
built, so a mismatch reports itself rather than arriving as a database error:

```
orm: table "users": Find was given a string but "id" is int
```

A composite primary key has no single value to look up by, and says so; use `Where`.

### `Count` and `Exists`

```go
n, err := Users.With(db).Where(Users.Age.Lt(18)).Count(ctx)      // int64
any, err := Users.With(db).Where(Users.Email.IsNull()).Exists(ctx) // bool
```

`Count` drops ordering and paging: they change which rows come back, not how many
match.

## Filtering

`Where` takes any number of conditions, joined with `AND`:

```go
Users.With(db).Where(Users.Age.Gte(18), Users.Active.Eq(true))
// WHERE ("age" >= $1 AND "active" = $2)
```

Conditions accumulate across calls, so building a query up in steps works:

```go
q := Users.With(db).Where(Users.Active.Eq(true))
if minAge > 0 {
    q = q.Where(Users.Age.Gte(minAge))
}
```

### Every operator, by column kind

Which operations a column has is decided by its type, at compile time. Calling one
that does not apply does not compile.

**Equality — every kind**

```go
Users.ID.Eq(1)                    // "id" = $1
Users.ID.NotEq(1)                 // "id" <> $1
Users.ID.In(1, 2, 3)              // "id" IN ($1, $2, $3)
Users.ID.NotIn(1, 2)              // "id" NOT IN ($1, $2)
```

An empty list is defined rather than a syntax error: `In()` matches nothing,
`NotIn()` matches everything.

`InQuery` and `NotInQuery` take another query instead of a list of values — see
[Matching against another query](#matching-against-another-query).

**Ordering — numbers, strings, timestamps, decimals**

```go
Users.Age.Gt(18)                  // "age" > $1
Users.Age.Gte(18)                 // "age" >= $1
Users.Age.Lt(65)                  // "age" < $1
Users.Age.Lte(65)                 // "age" <= $1
Users.Age.Between(18, 65)         // "age" BETWEEN $1 AND $2   (inclusive)
```

```go
Users.Active.Gt(false)            // compile error: no method Gt on *BoolColumn
```

Bool, UUID, enum, JSON and array columns have no ordering: `active > false` is legal
SQL but never something you mean.

**Text — string columns only**

```go
Users.Username.Contains("ali")    // LIKE '%ali%'
Users.Username.StartsWith("a")    // LIKE 'a%'
Users.Username.EndsWith("e")      // LIKE '%e'
Users.Username.Like("a%e")        // your pattern, untouched
Users.Username.ILike("A%E")       // case-insensitively
```

`Contains`, `StartsWith` and `EndsWith` match **literally**: wildcards in the
argument are escaped, so `Contains("50%")` finds the text `50%` rather than
everything beginning `50`. Use `Like` when the wildcards are the point.

**Full-text search — string columns only**

```go
Articles.Body.Matches("golang orm")
// to_tsvector("body") @@ websearch_to_tsquery($1)
```

`Matches` searches the words of a column rather than matching characters the way
`Like` does. The query is parsed the way a search box's contents are —
`"quoted phrases"`, `-exclusions`, and `or` are understood — and malformed input
matches nothing rather than erroring, so it is safe to pass straight from a user.

For the full tsquery vocabulary (`&`, `|`, `:*` prefix matching), reach for
[`orm.Raw`](#an-escape-hatch-raw-fragments) with `to_tsquery`. `Matches` is one
the dialect writes for itself, so a database with no full-text facility returns
an error naming it. It is the one operator that error can reach without a column
kind to gate it first: JSON and array operators only exist on columns a
non-supporting dialect could not declare, but every dialect has string columns.

**JSON documents — JSON columns only**

A JSON column is written and read as a whole, and also queried inside:

```go
Users.Prefs.HasKey("theme")                 // "prefs" ? $1          — has the key
Users.Prefs.Contains(Prefs{Theme: "dark"})  // "prefs" @> $1::jsonb  — contains this
Users.Prefs.Key("theme").Eq("dark")         // ("prefs" ->> $1) = $2 — the text at a key
```

`Contains` asks whether the stored document holds the given one as a subtree, so
a partial value matches: `Contains(Prefs{Theme: "dark"})` finds every row whose
`theme` is `dark`, whatever else its document holds. `Key(k)` extracts the text
at a top-level key, which `Eq` and `NotEq` then compare; a number or a nested
object is what `Contains` is for, or [`orm.Raw`](#an-escape-hatch-raw-fragments)
where a cast is needed.

There is still no `Eq` over the whole document. Postgres's `json` type has no
equality operator at all, so it would fail at query time for half a JSON
column's instantiations. And these three are where databases diverge most —
Postgres spells them as operators, MySQL as functions — so the dialect writes
each, and a driver that cannot express one returns an error naming it rather
than wrong SQL.

**Arrays — array columns only**

An array column compares whole arrays for equality, and also tests membership
and length:

```go
Users.Tags.Has("go")             // "tags" @> $1          — holds the element
Users.Tags.HasAll("go", "sql")   // "tags" @> $1          — holds all of these
Users.Tags.HasAny("go", "sql")   // "tags" && $1          — holds any of these
Users.Tags.Len().Gt(3)           // cardinality("tags") > $1
```

The element type is the array's, so a `StringArrayColumn`'s `Has` takes a string
and an `IntArrayColumn`'s takes an int. The elements are bound as one array
parameter rather than a placeholder each, which is what lets the driver send
them as the column's own array type — an `ARRAY[$1]` constructor would leave the
element type for the database to infer, and it infers text.

`HasAll()` and `HasAny()` with no arguments are defined rather than errors, the
same way an empty `In()` is: holding all of nothing is true of every array, so
it drops out of the condition, and holding any of nothing is false. `Len` counts
elements, zero for an empty array, so `Len().Eq(0)` means empty.

Arrays are a native type in Postgres and absent from MySQL, so here too the
dialect writes each test and a driver without them returns an error naming the
operation.

**Null tests — nullable columns only**

```go
Users.Email.IsNull()              // "email" IS NULL
Users.Email.IsNotNull()           // "email" IS NOT NULL

Users.Username.IsNull()           // compile error: not a nullable column
```

**Nullable columns take the underlying type**

```go
Users.Email.Eq("alice@example.com")   // not Eq(ptr("alice@example.com"))
Users.Email.EqPtr(maybeNil)           // IS NULL when maybeNil is nil
```

`EqPtr(nil)` becomes `IS NULL` deliberately: `col = NULL` is never true in SQL, so
compiling it literally would turn a nil pointer into a query that silently matches
nothing.

### Combining conditions

```go
orm.Or(Users.Age.Lt(18), Users.Age.Gt(65))
// ("age" < $1 OR "age" > $2)

orm.And(Users.Active.Eq(true), Users.Email.IsNotNull())
// ("active" = $1 AND "email" IS NOT NULL)

orm.Not(Users.Username.StartsWith("test"))
// NOT ("username" LIKE $1)
```

They nest freely:

```go
Users.With(db).Where(
    Users.Active.Eq(true),
    orm.Or(
        Users.Age.Between(18, 25),
        orm.And(Users.Age.Gt(65), Users.Email.IsNotNull()),
    ),
)
```

`And()` with no arguments is true, `Or()` with none is false. Both are defined
because a caller that filtered a slice down to nothing usually has a bug, and a
defined answer is easier to find than an accidental one.

### An escape hatch: raw fragments

When a condition is not in the vocabulary above — a function call, an operator
no column exposes, something a particular database spells its own way —
`orm.Raw` drops to SQL without dropping the safety:

```go
Users.With(db).Where(
    Users.Active.Eq(true),
    orm.Raw("lower(username) = ?", name),
).All(ctx)
// WHERE ("active" = $1 AND lower(username) = $2)
```

Each `?` binds its argument as a parameter, rewritten to the dialect's own
marker and numbered in with the rest of the statement — so a value in a fragment
is exactly as injection-safe as one in `Eq`. Only the SQL you wrote is written
literally; `name` never is, whatever it contains.

It is an ordinary predicate, so it composes like any other — beside typed
conditions, inside `orm.Or` and `orm.Not`, and as a condition on `UpdateIf`:

```go
Users.With(db).Where(orm.Or(Users.Age.Lt(18), orm.Raw("age % 2 = ?", 0)))
```

Write `??` for a literal question mark that binds nothing, which is how
Postgres's JSON operators are spelled inside a fragment whose other `?` are
placeholders:

```go
Users.With(db).Where(orm.Raw(`"prefs" ?? ?`, "theme"))
// WHERE "prefs" ? $1     — the jsonb "has key" operator
```

The number of `?` must match the number of arguments; a mismatch is reported
when the statement compiles, naming the fragment.

## Reading fewer columns

### One column, typed

```go
names, err := orm.Select(Users.With(db), Users.Username).All(ctx)   // []string
```

`orm.Select` reads a single column and gives you its values, not rows. It's a
function rather than a method because the result's type is the *column's*, not
the row's, and a Go method can't introduce a type parameter of its own.

It carries the whole query — conditions, ordering, paging:

```go
countries, err := orm.Select(
    Users.With(db).Where(Users.Active.Eq(true)).OrderBy(Users.Country.Asc()),
    Users.Country,
).Distinct().All(ctx)
```

A **nullable** column gives pointers, because `T` is the column's Go type: a
`NullableStringColumn` is a `Column[*string]`, so a `NULL` is a `nil` rather
than an empty string.

```go
emails, err := orm.Select(Users.With(db), Users.Email).All(ctx)   // []*string
```

Also available: `First(ctx)` (returns `ErrNoRows` when nothing matched) and
`Count(ctx)`, which counts the *column* and so skips `NULL`s — the answer to
"how many values are there", which differs from "how many rows".

```go
n, err := orm.Select(Users.With(db), Users.Email).Count(ctx)
// COUNT("email") — rows with no email are not counted

n, err := orm.Select(Users.With(db), Users.Country).Distinct().Count(ctx)
// COUNT(DISTINCT "country")
```

### Several columns

```go
users, err := Users.With(db).Select(Users.ID, Users.Username).All(ctx)
// SELECT "id", "username" FROM "users"
```

The rows still come back as `*User`.

::alert{type="warning"}
**A field whose column wasn't selected keeps its zero value**, and nothing
distinguishes that from a value the database genuinely holds as zero. Prisma
returns a narrower type for a projection; Go can't without code generation. If
one column is what you want, `orm.Select` has no such ambiguity.
::

Calls accumulate, so a projection built up in branches means what it reads as:

```go
q := Users.With(db).Select(Users.ID)
if wantNames {
    q = q.Select(Users.Username)
}
```

### Why bother

Reading 100 rows, measured with `-benchmem`:

| | ns/op | B/op | allocs/op |
|---|---|---|---|
| whole rows | 45,167 | 32,384 | 523 |
| one column into `*User` | 15,177 | 17,057 | 313 |
| `orm.Select`, typed | 8,018 | 8,229 | 215 |

## Distinct

```go
Users.With(db).Distinct().All(ctx)
Users.With(db).Select(Users.Country).Distinct().All(ctx)
```

Over a whole row it rarely says much — a table with a primary key has no
duplicate rows to drop. It earns its place on a projection, where the columns
you left out are exactly what made the rows differ.

`Count` on a distinct query counts the rows that query returns, which means
counting over the read rather than over the table:

```sql
SELECT COUNT(*) FROM (SELECT DISTINCT "country" FROM "users") AS "t"
```

## Aggregates

```go
total,   err := orm.Sum(ctx, Users.With(db), Users.Age)         // int
mean,    err := orm.Avg(ctx, Users.With(db), Users.Age)         // float64, always
youngest, err := orm.Min(ctx, Users.With(db), Users.Age)        // int
newest,  err := orm.Max(ctx, Users.With(db), Users.CreatedAt)   // time.Time
```

Each returns the *column's* type, inferred from the column — so like
`orm.Select`, these are functions rather than methods.

They carry the whole query, and honour `Distinct`:

```go
orm.Sum(ctx, Users.With(db).Where(Users.Active.Eq(true)), Users.Age)
// SELECT SUM("age") FROM "users" WHERE "active" = $1

orm.Sum(ctx, Users.With(db).Distinct(), Users.Age)
// SELECT SUM(DISTINCT "age") FROM "users"
```

Ordering and paging are dropped, as `Count` drops them: they change which rows
come back, not what the set adds up to.

### Over no rows

SQL answers `NULL`, and what that means differs by aggregate:

| | over no rows |
|---|---|
| `Sum` | the zero value — a total of nothing is nothing |
| `Min`, `Max`, `Avg` | `ErrNoRows` — there is no such value, and zero would read as one |

```go
if _, err := orm.Max(ctx, q, Users.Age); errors.Is(err, orm.ErrNoRows) {
    // nothing matched
}
```

**A nullable column is the exception**, and deliberately: it's a
`Column[*string]`, so `nil` is a value meaning `NULL`, not the absence of one.

```go
largest, err := orm.Max(ctx, Users.With(db), Users.Email)   // *string
// err is nil and largest is nil when every email is NULL
```

`Avg` returns a float whatever the column holds, because a mean of integers
isn't an integer and rounding it silently would be a lie.

### Grouping

```go
byCountry, err := orm.CountBy(Users.With(db), Users.Country).All(ctx)
// []orm.Bucket[string, int64]{{Key: "TR", Value: 42}, ...}

orm.SumBy(Users.With(db), Users.Country, Users.Age).All(ctx)   // []Bucket[string, int]
orm.AvgBy(Users.With(db), Users.Country, Users.Age).All(ctx)   // []Bucket[string, float64]
orm.MinBy(...)   orm.MaxBy(...)
```

One function per aggregate rather than a `GroupBy` followed by a `Sum`, because
that second call can't exist: the value's type comes from the column being
aggregated, and a Go method can't introduce a type parameter of its own. Naming
both halves at once means nothing has to be spelled out.

The top ten, which is what a grouped query usually wants:

```go
top, err := orm.CountBy(Users.With(db).Where(Users.Active.Eq(true)), Users.Country).
    Having(orm.OpGte, 10).
    OrderByValueDesc().
    Limit(10).
    All(ctx)
```
```sql
SELECT "country", COUNT(*) FROM "users" WHERE "active" = $1
GROUP BY "country" HAVING COUNT(*) >= $2
ORDER BY COUNT(*) DESC LIMIT 10
```

`Having` takes the same `Operator` a column comparison does — the thing being
compared is a single value and the six spellings are the same six. Conditions
accumulate, joined with `AND`.

`OrderBy` sorts by the **key**; `OrderByValue` / `OrderByValueDesc` sort by the
aggregate. Ordering by anything else is rejected, since every other column has
been collapsed into the group:

```
orm: table "users": a grouped query can only be ordered by its key, "country",
    or by the aggregate with OrderByValue
```

`Keys(ctx)` returns just the keys, for when the aggregate was only there to
drive a `HAVING`.

A **nullable** key collects its `NULL`s into a group of their own, and `K` is a
pointer, so it holds one:

```go
byEmail, err := orm.CountBy(Users.With(db), Users.Email).All(ctx)
// []orm.Bucket[*string, int64] — one bucket has Key == nil
```

::alert{type="info"}
**One key, not several.** Two would need a `Bucket` with two key types and a
constructor per arity. It's `Bucket` rather than `Group`, incidentally, because
`orm.Group` is already a predicate.
::

## Ordering and paging

```go
Users.With(db).OrderBy(Users.CreatedAt.Desc(), Users.Username.Asc())
// ORDER BY "created_at" DESC, "username" ASC

Users.With(db).Limit(20).Offset(40)
// LIMIT 20 OFFSET 40
```

Terms accumulate in the order given. A negative `Limit` or `Offset` is an error,
reported from whichever terminal you call.

Ordering is available on every column kind except JSON documents.

A read can also hold the rows it returns until the transaction ends, with
`ForUpdate`, `ForShare`, `SkipLocked` and `NoWait` — see
[Transactions](/docs/2.fundamentals/10.transactions#locking-the-rows-you-read).

## Streaming

`All` builds a slice with every matching row in it. For a query that matches a
handful, that is exactly what you want. For one that matches a million, it is a
million `*User` in memory before you have looked at the first. `Each` reads them
one at a time instead:

```go
for user, err := range Users.With(db).Where(Users.Active.Eq(true)).Each(ctx) {
    if err != nil {
        return err
    }
    process(user)
}
```

It is a range-over-func iterator, so it reads like a `for range` over a slice
and needs no `Close`. The difference is that the next row is fetched from the
database only when the loop asks for it, so the whole result set is never held
at once.

The error is the second range value rather than something returned, because a
range-over-func has nowhere to return one to. It is yielded once, with a nil
row, and iteration then stops — so a loop that checks it each step sees every
failure `All` would: a row that would not decode, an `AfterLoad` that refused
it, or a connection lost partway through. Check it every iteration; it is not
an afterthought at the end.

Breaking out of the loop closes the result set, so an early exit leaks nothing:

```go
for user, err := range Users.With(db).OrderBy(Users.CreatedAt.Desc()).Each(ctx) {
    if err != nil {
        return err
    }
    if seenEnough(user) {
        break   // the cursor is closed here
    }
}
```

`orm.Select` streams its one column the same way, for when a whole row is more
than you need:

```go
for name, err := range orm.Select(Users.With(db), Users.Username).Each(ctx) {
    ...
}
```

### Streaming and locks go together

Because the cursor stays open across the loop, a row lock the read took is held
for as long as you are iterating — which is exactly the shape a work queue
wants. Inside a transaction, `ForUpdate().SkipLocked()` hands each worker a
different batch to stream through, each row locked until the transaction ends:

```go
err := db.Transaction(ctx, func(tx *orm.DB) error {
    for job, err := range Jobs.With(tx).Where(Jobs.State.Eq("queued")).
        ForUpdate().SkipLocked().Each(ctx) {
        if err != nil {
            return err
        }
        // handle job; its row stays locked until this transaction ends
    }
    return nil
})
```

::alert{type="info"}
**One statement at a time on a connection.** While a stream is open that
connection is busy with it, so you cannot run another query on the same handle
until the loop finishes or breaks. Reach for a second handle if you must write
as you read, or collect the rows you want and act on them after the loop.
::

### What it will not stream

A query with `Load` is refused rather than streamed:

```go
Users.With(db).Load(Users.Posts).Each(ctx)
// error: Each cannot stream a query with Load, since eager loading needs
//        every row in hand at once; use All
```

Eager loading collects every parent key first and then runs a statement per
relationship over the whole set, which is the result set in memory that
streaming exists to avoid. Load with `All`, or stream without it.

## Matching against another query

`orm.Select` gives back a value. Run it, and it's a list of values; hand it to
`InQuery` instead, and it becomes part of the statement:

```go
authors := orm.Select(Posts.With(db).Where(Posts.Published.Eq(true)), Posts.AuthorID)

ids, err := authors.All(ctx)                                  // run it
users, err := Users.With(db).Where(Users.ID.InQuery(authors)). // or embed it
    All(ctx)
```

```sql
SELECT "id", "username", ... FROM "users"
WHERE "id" IN (SELECT "posts"."author_id" FROM "posts" WHERE "posts"."published" = $1)
```

One statement, not two: nothing is fetched and sent back. That also makes it the
right shape for a set too large to bind — an `In` list is capped by the dialect's
parameter ceiling, and a subquery isn't.

The subquery carries its own conditions, ordering, paging and `Distinct`, and its
columns are qualified inside the parentheses so nothing is resolved by how the
two happen to be nested. Placeholders keep counting across the boundary.

It's an ordinary predicate, so it goes anywhere one goes — beside other
conditions, inside `orm.Or` and `orm.Not`, inside an `orm.Has`, in front of a
write, and inside another subquery:

```go
Users.With(db).Where(Users.ID.NotInQuery(authors)).DeleteAll(ctx)

tagged := orm.Select(BookTags.With(db).Where(BookTags.TagID.Eq(7)), BookTags.BookID)
written := orm.Select(Books.With(db).Where(Books.ID.InQuery(tagged)), Books.AuthorID)
Authors.With(db).Where(Authors.ID.InQuery(written)).All(ctx)
```

The type is part of the check. `InQuery` on an int column takes a subquery
yielding ints, so pointing it at a column of usernames doesn't compile.

### `NOT IN` and `NULL`

This is SQL's sharpest edge, and Tork makes you walk around it.

`NOT IN` is **never true** once the subquery yields a single `NULL`. Not false —
unknown, which a `WHERE` treats the same way. So this, in raw SQL, returns
nothing at all, however many rows plainly qualify:

```sql
SELECT * FROM authors WHERE id NOT IN (SELECT editor_id FROM books)
-- one book with no editor, and the whole query is empty
```

Selecting a nullable column gives a `*Scalars[*int]`, and `NotInQuery` on an int
column wants a `SubqueryOf[int]`, so that mistake is a **compile error**:

```go
editors := orm.Select(Books.With(db), Books.EditorID)   // *Scalars[*int]
Authors.With(db).Where(Authors.ID.NotInQuery(editors))  // does not compile
```

`orm.NonNull` is the fix, and it fixes both halves at once — it retypes the
subquery *and* adds the condition that makes the answer right:

```go
Authors.With(db).Where(Authors.ID.NotInQuery(orm.NonNull(editors))).All(ctx)
// ... NOT IN (SELECT "books"."editor_id" FROM "books"
//             WHERE "books"."editor_id" IS NOT NULL)
```

It narrows a copy, so the query you passed in is unchanged and still runnable on
its own terms.

::alert{type="info"}
A **nullable column on the outside** needs no ceremony: `Reviews.BookID.InQuery(...)`
takes a subquery of `int`, exactly as its `Eq` takes an `int`. A row whose own
value is `NULL` matches neither `InQuery` nor `NotInQuery`, which is ordinary SQL
and what `IsNull` is for.
::

## Filtering by related rows

```go
// users who have posted at all
Users.With(db).Where(orm.Has(Users.Posts)).All(ctx)

// users who have a published post
Users.With(db).Where(orm.Has(Users.Posts, Posts.Published.Eq(true))).All(ctx)

// users who have never posted
Users.With(db).Where(orm.HasNone(Users.Posts)).All(ctx)

// users with no published post — a user with only drafts matches
Users.With(db).Where(orm.HasNone(Users.Posts, Posts.Published.Eq(true))).All(ctx)
```

```sql
WHERE EXISTS (
    SELECT 1 FROM "posts"
    WHERE "posts"."author_id" = "users"."id" AND "posts"."published" = $1
)
```

`EXISTS` rather than a join, because the question is *whether* a related row is
there, not what is in it. A join would multiply the rows a user appears in by
the number of posts they have, leaving you to collapse them back down.

### It's an ordinary predicate

Which is the point — it goes wherever one goes:

```go
Users.With(db).Where(
    Users.Active.Eq(true),
    orm.Has(Users.Posts, Posts.Published.Eq(true)),
    orm.HasNone(Users.Bans),
).OrderBy(Users.Name.Asc()).Limit(20).All(ctx)

orm.Or(orm.Has(Users.Posts), orm.Has(Users.Comments))
orm.Not(orm.Has(Users.Posts))

n, err := Users.With(db).Where(orm.HasNone(Users.Posts)).DeleteAll(ctx)
```

And it nests — the conditions may themselves ask about a relationship of the
related rows:

```go
// authors who have a book that has a tag
Authors.With(db).Where(orm.Has(Authors.Books, orm.Has(Books.Tags))).All(ctx)
```

A **many-to-many** becomes two `EXISTS` rather than a join:

```sql
WHERE EXISTS (
    SELECT 1 FROM "book_tags"
    WHERE "book_tags"."book_id" = "books"."id"
      AND EXISTS (
        SELECT 1 FROM "tags"
        WHERE "tags"."id" = "book_tags"."tag_id" AND "tags"."name" = $1
      )
)
```

Unfiltered, the inner one is dropped: a row in the join table already answers
the question, since a foreign key means the row it names is there.

::alert{type="info"}
**Coming from Prisma**, `Has` is `some` and `HasNone` is `none`. There is no
`every`, which would be `HasNone` with the condition negated —
`orm.HasNone(Users.Posts, orm.Not(Posts.Published.Eq(true)))` matches users all
of whose posts are published, and, like Prisma's `every`, also those with no
posts at all.
::

### Filtering and loading are separate

```go
Users.With(db).
    Where(orm.Has(Users.Posts, Posts.Published.Eq(true))).
    Load(Users.Posts.Where(Posts.Published.Eq(true))).
    All(ctx)
```

The condition appears twice, and that is deliberate rather than an oversight.
They answer different questions — *which users* versus *which of their posts* —
and coupling them would surprise anyone wanting users who have a published post
together with all of their posts. Prisma has exactly the same shape.

## Loading related rows

```go
users, err := Users.With(db).Load(Users.Posts).All(ctx)
// users[0].Posts is filled
```

The field is found by the marker's own name on the model, the same convention
that matches a column to a field: `UserModel.Posts` fills `User.Posts`. That
field matches no column, which is exactly why it can exist — the entity mapping
ignores a field no column is named after.

```go
type User struct {
    ID    int
    Name  string
    Posts []Post   // filled by Load(Users.Posts)
}
```

A `HasMany` or `ManyToMany` wants a slice (`[]Post` or `[]*Post`); a `HasOne` or
`BelongsTo` wants one value (`Post` or `*Post`).

### Narrowing what gets loaded

```go
Users.With(db).Load(
    Users.Posts.
        Where(Posts.Published.Eq(true)).
        OrderBy(Posts.CreatedAt.Desc()).
        Limit(5),
).All(ctx)
```

Several at once, and nested:

```go
Users.With(db).Load(Users.Posts, Users.Profile).All(ctx)
Users.With(db).Load(Users.Posts.Load(Posts.Comments)).All(ctx)
```

### How many statements

One per relationship, whatever the number of rows:

```sql
SELECT "id", "name" FROM "users"
SELECT "id", "author_id", "title" FROM "posts" WHERE "author_id" IN ($1, $2, $3)
```

This is a second query, not a join — SQLAlchemy's `selectinload` rather than its
`joinedload`. A join would multiply the parent's columns by the number of
related rows, so a user with fifty posts arrives fifty times; and your `Limit`
would stop meaning what it says, since it would cap joined rows rather than
users.

A **many-to-many** takes two statements rather than a join: the join table's
pairs, then the rows those pairs name. A row two parents share is fetched once
and given to both.

::alert{type="info"}
**`Limit` is per parent**, which is the only reading that means anything — the
five most recent posts of *each* user. No portable SQL caps rows per group, so
a limited load costs one statement per parent. Without it, the whole load is
one statement however many parents there are.
::

### What it won't let you do

A projection that leaves out the matching column is rejected rather than
silently loading nothing:

```go
Users.With(db).Select(Users.Name).Load(Users.Posts).All(ctx)
```
```
orm: table "users": loading Posts needs column "id", which this query does not
    select; add it to the Select, or drop the Select
```

A row type with no field for the relationship is fine until you ask for the
rows — the marker is still worth declaring for what `Relation()` reports:

```
orm: users.Posts: models.User has no exported field named "Posts" to load into;
    a relationship fills the field of the row type it is named after
```

## Writing one row

```go
user := &User{Username: "alice", Age: 30}
err := Users.With(db).Insert(ctx, user)
// user.ID is now the key the database generated
```

```go
user.Age = 31
err := Users.With(db).Update(ctx, user)   // by primary key
```

```go
err := Users.With(db).Delete(ctx, user)   // by primary key
```

```go
err := Users.With(db).Save(ctx, user)     // Insert if new, Update if not
```

`Save` tells a new row from a stored one by whether the generated key is still zero.
A table whose key you supply yourself has no such signal, and `Save` says so rather
than guessing.

`Update` writes **every** writable column, not only the ones that changed: Go cannot
intercept a field assignment, so nothing can know which those were. The primary key
and the generated key are excluded, since they say which row is being written rather
than what to write.

`Update` and `Delete` return `orm.ErrNoRows` when the row was not there.

### Writing only if the row has not changed

```go
seen := user.Version
user.Version++
user.Name = newName

err := Users.With(db).UpdateIf(ctx, user, Users.Version.Eq(seen))
if errors.Is(err, orm.ErrNoRows) {
    // somebody else got there first — re-read and try again
}
```

```sql
UPDATE "users" SET ... WHERE ("id" = $9 AND "version" = $10)
```

This is **optimistic locking**, and the other answer to the problem
[`ForUpdate`](/docs/2.fundamentals/10.transactions#locking-the-rows-you-read)
solves pessimistically. Instead of holding a lock between reading a row and
writing it, the write states what the row looked like when it was read and fails
if that is no longer true. It costs nothing while nothing contends, which is the
case it is for; a lock is the better answer when contention is the norm.

Note that the condition uses the version that was **read**, not the one now in the
struct — the row in hand has already been changed by the time it is written.

`DeleteIf` is the same for removal:

```go
err := Posts.With(db).DeleteIf(ctx, post, Posts.Draft.Eq(true))
```

The key and the conditions become one `WHERE` joined with `AND`, and the conditions
are ordinary predicates, so the whole vocabulary of
[filtering](#every-operator-by-column-kind) is available — not just equality on a
version column.

Both return `orm.ErrNoRows` when nothing was written, which means **either** that the
row is gone **or** that the conditions no longer hold. Those are not distinguished:
telling them apart would take a second statement, whose answer would be about a
moment later than the one you asked about.

### What `Insert` leaves to the database

A column is left out of the statement when the database supplies its value: the
generated key always, and any column with a `ServerDefault` whose field is still
zero. Both are read back into your struct afterwards.

That has one consequence worth knowing: a zero cannot be inserted explicitly into a
defaulted column, because nothing distinguishes "left alone" from "set to zero" in a
Go struct. Declare the field as a pointer if the difference matters.

```go
err := Users.With(db).InsertMany(ctx, users...)              // rows you hold
n, err := Users.With(db).Where(...).UpdateAll(ctx, sets...)  // rows a filter matches

// an insert that expects the row may already be there
n, err := Users.With(db).OnConflict(Users.Email).DoUpdateAll().InsertMany(ctx, users...)
```

## Seeing the SQL

```go
sql, args, err := Users.With(db).Where(Users.Age.Gt(18)).Limit(5).SQL()

// SELECT "id", "username", "email", "age", "active", "created_at" FROM "users"
//     WHERE "age" > $1 LIMIT 5
// [18]
```

Nothing runs. Useful in tests, and for reading what an unfamiliar chain produces.

## Queries are values

Every builder method returns a **new** query rather than narrowing the one it was
called on, so a query is safe to hold, to branch from, and to use from several
goroutines:

```go
adults := Users.With(db).Where(Users.Age.Gte(18))

alice := adults.Where(Users.Username.Eq("alice"))
bob   := adults.Where(Users.Username.Eq("bob"))
```

`adults` still means "everyone over 18". Were they to share state, each branch would
carry the other's condition and both would match nothing, with no error anywhere to
say why.

## Values are always parameters

Every value you supply is bound as a parameter; none is ever written into the
statement as text. That makes the query API injection-safe by construction:

```go
Users.With(db).Where(Users.Username.Eq(userInput))
// WHERE "username" = $1     — userInput is a parameter, whatever it contains
```

The only SQL written literally is SQL you wrote yourself: a `ServerDefault`
expression, a `CHECK` expression, an index predicate, or an
[`orm.Raw`](#an-escape-hatch-raw-fragments) fragment — and even there the `?`
values are bound as parameters, only the fragment's text being yours to write.

## Hooks

A row type can implement any of these, and they run around the matching operation:

```go
func (p *Post) BeforeCreate(ctx context.Context) error {
    p.Slug = slugify(p.Title)
    return nil
}
```

`BeforeCreate` `AfterCreate` `BeforeUpdate` `AfterUpdate` `BeforeDelete`
`AfterDelete` `AfterLoad`.

A `Before` hook returning an error stops the operation before any SQL runs. `Save`
has no hook of its own; it fires the pair belonging to whichever operation it chose.

`UpdateAll` and `DeleteAll` fire nothing: they never load a row, so there is no
value to call a method on.

The receiver must be a pointer, or the hook mutates a copy that is then thrown away.
`DefineTable` rejects a value receiver at startup rather than let that happen
silently.

## Errors

```go
var ErrNoRows = errors.New("orm: no rows")
```

Returned by `First`, `Find`, `Update` and `Delete`, and wrapped by the batch
operations when they write fewer rows than they were given. Test it with
`errors.Is`.

Errors from builder methods are held until a terminal runs, so a chain never breaks
mid-way:

```go
users, err := Users.With(db).Limit(-1).All(ctx)
// err: orm: table "users": Limit(-1) is negative
```
