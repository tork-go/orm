# Querying

Every query starts from a model and a database handle:

```go
db, err := orm.Connect(ctx, os.Getenv("DATABASE_URL"))

users, err := Users.With(db).Where(Users.Age.Gte(18)).All(ctx)
```

The examples on this page use one model throughout:

```go
type User struct {
    ID        int
    Username  string
    Email     *string
    Age       int
    Active    bool
    CreatedAt time.Time
}

type UserModel struct {
    orm.Table[User]
    ID        *orm.IntColumn
    Username  *orm.StringColumn
    Email     *orm.NullableStringColumn
    Age       *orm.IntColumn
    Active    *orm.BoolColumn
    CreatedAt *orm.TimeColumn
}
```

## Two types, one chain

`With(db)` gives you a `*Query[User]`. Naming a condition, an order, or a limit
turns it into a `*Filtered[User]`:

```go
Users.With(db)                          // *Query[User]
Users.With(db).Where(Users.Age.Gt(18))  // *Filtered[User]
```

They carry different operations, and that is the point:

| | `Query[E]` | `Filtered[E]` |
| --- | --- | --- |
| Reading | `All` `First` `Count` `Exists` `Find` | `All` `First` `Count` `Exists` |
| One row you hold | `Insert` `Update` `Save` `Delete` | — |
| Many rows you hold | `InsertMany` `UpdateMany` `DeleteMany` | — |
| Every matching row | `UpdateAll` `DeleteAll` | `UpdateAll` `DeleteAll` |

A filter cannot change which row a `*User` identifies, so combining the two never
meant anything. Writing one before the other is a compile error rather than
something reported at run time:

```go
Users.With(db).Where(Users.Age.Gt(18)).Insert(ctx, user)
// compile error: Insert is not a method on *orm.Filtered[User]
```

## Reading

### `All`

```go
users, err := Users.With(db).Where(Users.Active.Eq(true)).All(ctx)
```

Returns `[]*User`. An empty result is an empty slice, not an error.

### `First`

```go
user, err := Users.With(db).Where(Users.Username.Eq("alice")).First(ctx)
if errors.Is(err, orm.ErrNoRows) {
    // nothing matched
}
```

Adds `LIMIT 1` to a copy of the query, so the query you called it on keeps its own
limit.

### `Find`

```go
user, err := Users.With(db).Find(ctx, 42)   // by primary key
```

The key is type-checked against the primary key column before the statement is
built, so a mismatch reports itself rather than arriving as a database error:

```
orm: table "users": Find was given a string but "id" is int
```

A composite primary key has no single value to look up by, and says so; use `Where`.

### `Count` and `Exists`

```go
n, err := Users.With(db).Where(Users.Age.Lt(18)).Count(ctx)      // int64
any, err := Users.With(db).Where(Users.Email.IsNull()).Exists(ctx) // bool
```

`Count` drops ordering and paging: they change which rows come back, not how many
match.

## Filtering

`Where` takes any number of conditions, joined with `AND`:

```go
Users.With(db).Where(Users.Age.Gte(18), Users.Active.Eq(true))
// WHERE ("age" >= $1 AND "active" = $2)
```

Conditions accumulate across calls, so building a query up in steps works:

```go
q := Users.With(db).Where(Users.Active.Eq(true))
if minAge > 0 {
    q = q.Where(Users.Age.Gte(minAge))
}
```

### Every operator, by column kind

Which operations a column has is decided by its type, at compile time. Calling one
that does not apply does not compile.

**Equality — every kind**

```go
Users.ID.Eq(1)                    // "id" = $1
Users.ID.NotEq(1)                 // "id" <> $1
Users.ID.In(1, 2, 3)              // "id" IN ($1, $2, $3)
Users.ID.NotIn(1, 2)              // "id" NOT IN ($1, $2)
```

An empty list is defined rather than a syntax error: `In()` matches nothing,
`NotIn()` matches everything.

**Ordering — numbers, strings, timestamps, decimals**

```go
Users.Age.Gt(18)                  // "age" > $1
Users.Age.Gte(18)                 // "age" >= $1
Users.Age.Lt(65)                  // "age" < $1
Users.Age.Lte(65)                 // "age" <= $1
Users.Age.Between(18, 65)         // "age" BETWEEN $1 AND $2   (inclusive)
```

```go
Users.Active.Gt(false)            // compile error: no method Gt on *BoolColumn
```

Bool, UUID, enum, JSON and array columns have no ordering: `active > false` is legal
SQL but never something you mean.

**Text — string columns only**

```go
Users.Username.Contains("ali")    // LIKE '%ali%'
Users.Username.StartsWith("a")    // LIKE 'a%'
Users.Username.EndsWith("e")      // LIKE '%e'
Users.Username.Like("a%e")        // your pattern, untouched
Users.Username.ILike("A%E")       // case-insensitively
```

`Contains`, `StartsWith` and `EndsWith` match **literally**: wildcards in the
argument are escaped, so `Contains("50%")` finds the text `50%` rather than
everything beginning `50`. Use `Like` when the wildcards are the point.

**Null tests — nullable columns only**

```go
Users.Email.IsNull()              // "email" IS NULL
Users.Email.IsNotNull()           // "email" IS NOT NULL

Users.Username.IsNull()           // compile error: not a nullable column
```

**Nullable columns take the underlying type**

```go
Users.Email.Eq("alice@example.com")   // not Eq(ptr("alice@example.com"))
Users.Email.EqPtr(maybeNil)           // IS NULL when maybeNil is nil
```

`EqPtr(nil)` becomes `IS NULL` deliberately: `col = NULL` is never true in SQL, so
compiling it literally would turn a nil pointer into a query that silently matches
nothing.

### Combining conditions

```go
orm.Or(Users.Age.Lt(18), Users.Age.Gt(65))
// ("age" < $1 OR "age" > $2)

orm.And(Users.Active.Eq(true), Users.Email.IsNotNull())
// ("active" = $1 AND "email" IS NOT NULL)

orm.Not(Users.Username.StartsWith("test"))
// NOT ("username" LIKE $1)
```

They nest freely:

```go
Users.With(db).Where(
    Users.Active.Eq(true),
    orm.Or(
        Users.Age.Between(18, 25),
        orm.And(Users.Age.Gt(65), Users.Email.IsNotNull()),
    ),
)
```

`And()` with no arguments is true, `Or()` with none is false. Both are defined
because a caller that filtered a slice down to nothing usually has a bug, and a
defined answer is easier to find than an accidental one.

## Ordering and paging

```go
Users.With(db).OrderBy(Users.CreatedAt.Desc(), Users.Username.Asc())
// ORDER BY "created_at" DESC, "username" ASC

Users.With(db).Limit(20).Offset(40)
// LIMIT 20 OFFSET 40
```

Terms accumulate in the order given. A negative `Limit` or `Offset` is an error,
reported from whichever terminal you call.

Ordering is available on every column kind except JSON documents.

## Writing one row

```go
user := &User{Username: "alice", Age: 30}
err := Users.With(db).Insert(ctx, user)
// user.ID is now the key the database generated
```

```go
user.Age = 31
err := Users.With(db).Update(ctx, user)   // by primary key
```

```go
err := Users.With(db).Delete(ctx, user)   // by primary key
```

```go
err := Users.With(db).Save(ctx, user)     // Insert if new, Update if not
```

`Save` tells a new row from a stored one by whether the generated key is still zero.
A table whose key you supply yourself has no such signal, and `Save` says so rather
than guessing.

`Update` writes **every** writable column, not only the ones that changed: Go cannot
intercept a field assignment, so nothing can know which those were. The primary key
and the generated key are excluded, since they say which row is being written rather
than what to write.

`Update` and `Delete` return `orm.ErrNoRows` when the row was not there.

### What `Insert` leaves to the database

A column is left out of the statement when the database supplies its value: the
generated key always, and any column with a `ServerDefault` whose field is still
zero. Both are read back into your struct afterwards.

That has one consequence worth knowing: a zero cannot be inserted explicitly into a
defaulted column, because nothing distinguishes "left alone" from "set to zero" in a
Go struct. Declare the field as a pointer if the difference matters.

## Writing many rows

```go
err := Users.With(db).InsertMany(ctx, users...)              // rows you hold
n, err := Users.With(db).Where(...).UpdateAll(ctx, sets...)  // rows a filter matches
```

## Seeing the SQL

```go
sql, args, err := Users.With(db).Where(Users.Age.Gt(18)).Limit(5).SQL()

// SELECT "id", "username", "email", "age", "active", "created_at" FROM "users"
//     WHERE "age" > $1 LIMIT 5
// [18]
```

Nothing runs. Useful in tests, and for reading what an unfamiliar chain produces.

## Queries are values

Every builder method returns a **new** query rather than narrowing the one it was
called on, so a query is safe to hold, to branch from, and to use from several
goroutines:

```go
adults := Users.With(db).Where(Users.Age.Gte(18))

alice := adults.Where(Users.Username.Eq("alice"))
bob   := adults.Where(Users.Username.Eq("bob"))
```

`adults` still means "everyone over 18". Were they to share state, each branch would
carry the other's condition and both would match nothing, with no error anywhere to
say why.

## Values are always parameters

Every value you supply is bound as a parameter; none is ever written into the
statement as text. That makes the query API injection-safe by construction:

```go
Users.With(db).Where(Users.Username.Eq(userInput))
// WHERE "username" = $1     — userInput is a parameter, whatever it contains
```

The only SQL written literally is SQL you wrote yourself: a `ServerDefault`
expression, a `CHECK` expression, or an index predicate.

## Hooks

A row type can implement any of these, and they run around the matching operation:

```go
func (p *Post) BeforeCreate(ctx context.Context) error {
    p.Slug = slugify(p.Title)
    return nil
}
```

`BeforeCreate` `AfterCreate` `BeforeUpdate` `AfterUpdate` `BeforeDelete`
`AfterDelete` `AfterLoad`.

A `Before` hook returning an error stops the operation before any SQL runs. `Save`
has no hook of its own; it fires the pair belonging to whichever operation it chose.

`UpdateAll` and `DeleteAll` fire nothing: they never load a row, so there is no
value to call a method on.

The receiver must be a pointer, or the hook mutates a copy that is then thrown away.
`DefineTable` rejects a value receiver at startup rather than let that happen
silently.

## Errors

```go
var ErrNoRows = errors.New("orm: no rows")
```

Returned by `First`, `Find`, `Update` and `Delete`, and wrapped by the batch
operations when they write fewer rows than they were given. Test it with
`errors.Is`.

Errors from builder methods are held until a terminal runs, so a chain never breaks
mid-way:

```go
users, err := Users.With(db).Limit(-1).All(ctx)
// err: orm: table "users": Limit(-1) is negative
```

## What is not built yet

- **Loading related rows.** `HasMany`, `BelongsTo` and friends resolve which columns
  a join would match on (see [Relationships](/docs/2.fundamentals/5.relationships)),
  but nothing fetches them yet.
- **Joins**, and selecting a subset of columns.
- **Aggregates** beyond `Count`.
- **Upsert** (`ON CONFLICT`).
