# Bulk Operations

There are two different things "write many rows" can mean, and Tork spells them
differently because they behave differently.

```go
// many rows you are holding
err := Users.With(db).InsertMany(ctx, users...)

// every row a filter matches
n, err := Users.With(db).Where(Users.Age.Lt(18)).UpdateAll(ctx, Users.Active.Set(false))
```

The first has rows, so hooks fire on each one and generated values are read back
into them. The second has only a condition; the rows it writes may not be
represented in Go at all, so neither is possible.

## One rule for three suffixes

| Form | Means | Example |
| --- | --- | --- |
| bare | one row you hold | `Insert(ctx, user)` |
| `Many` | many rows you hold | `InsertMany(ctx, users...)` |
| `All` | every row the filter matches | `DeleteAll(ctx)` |

`All` already means "every row this query matches" on the read side: `All(ctx)`
returns those rows, `UpdateAll` and `DeleteAll` write them.

::alert{type="info"}
**Coming from Prisma?** `updateMany` and `deleteMany` there are the *set*
operations, which here are `UpdateAll` and `DeleteAll`. Tork's `UpdateMany` and
`DeleteMany` are the entity batch, which Prisma has no name for.
::

## `InsertMany`

```go
users := []*User{
    {Username: "alice", Age: 30},
    {Username: "bob", Age: 41},
}
err := Users.With(db).InsertMany(ctx, users...)
```

```sql
INSERT INTO "users" ("username", "age") VALUES ($1, $2), ($3, $4) RETURNING "id"
```

Values the database produces are read back into each row, exactly as `Insert` does
for one: a generated key, and any column left to a server default. After the call,
`users[0].ID` is the key that row was stored under.

### Rows need not write the same columns

A column with a `ServerDefault` is left out while its field is still zero (see
[Columns](/docs/2.fundamentals/2.columns)), so two rows of the same type can want
different column lists:

```go
rows := []*Article{
    {Title: "first"},                      // created_at left to the database
    {Title: "second", CreatedAt: someTime}, // supplied explicitly
    {Title: "third"},                      // left to the database
}
```

Rather than forcing a value on the rows that had none, the batch is split into one
statement per column list:

```sql
INSERT INTO "articles" ("title") VALUES ($1), ($2) RETURNING "id", "created_at";
INSERT INTO "articles" ("title", "created_at") VALUES ($1, $2) RETURNING "id";
```

Each row still gets its own values back, across the split. This is the same
approach SQLAlchemy takes for heterogeneous parameter dictionaries.

### Batches are split to fit the driver

Every database caps how many parameters one statement may bind: PostgreSQL allows
65535, a limit of the wire protocol rather than a setting. A ten-column insert
therefore hits it at about 6,500 rows, and the failure is a driver error naming a
number nobody chose.

Tork asks the dialect for that ceiling and splits the batch to stay under it. There
is no batch-size argument to pass and no error to handle; a hundred thousand rows
is a valid call.

### The order rows come back in

Correlating the *n*-th returned row with the *n*-th written row assumes the database
returns a plain multi-row `VALUES` insert in insertion order. PostgreSQL does, and
it is the only correlation available: the values being read back are exactly the
ones you did not supply, so nothing in a returned row identifies which input row it
belongs to.

This is worth knowing rather than worrying about. It holds for the statements Tork
writes; it stops holding for a statement whose rows the database is free to skip or
reorder, which is exactly why an [upsert](#upsert) with anything to read back drops
to one row per statement.

### Databases without `RETURNING`

A dialect that reports no `RETURNING` support has only one signal for a generated
key, a single "last insert id", which cannot be attributed to a row of a statement
covering several. Such a batch falls back to one statement per row so that key
belongs to a known row.

Where there is nothing to read back, the batch is still one statement. PostgreSQL
supports `RETURNING`, so this only affects future drivers.

## Upsert

An insert that expects the row may already be there says so with `OnConflict`,
then says what to do about it:

```go
// skip a row already there
n, err := Users.With(db).OnConflict(Users.Email).DoNothing().InsertMany(ctx, users...)

// overwrite the columns you name
Users.With(db).OnConflict(Users.Email).DoUpdate(Users.Username, Users.Age).
    InsertMany(ctx, users...)

// overwrite everything this insert was writing
Users.With(db).OnConflict(Users.Email).DoUpdateAll().Insert(ctx, user)
```

```sql
INSERT INTO "users" ("email", "username", "age") VALUES ($1, $2, $3)
ON CONFLICT ("email") DO UPDATE SET "username" = EXCLUDED."username", "age" = EXCLUDED."age"
RETURNING "id"
```

It is two steps because those are two decisions, and each step hands back a type
carrying only what can follow it — so what comes next is a completion list rather
than something to look up.

**The columns are the ones that conflict**, which is to say the ones carrying the
unique constraint the insert would violate. That is usually a natural key like an
email, not the primary key: a row identified by a generated key is new by
construction and could never collide.

`DoUpdate` names columns rather than taking assignments, because the values are
not yours to choose here — they are the ones already in the rows being inserted.
An upsert whose update wrote something else would be two statements wearing one
name.

### `DoUpdateAll` and which columns that is

Every column *this statement writes*, less the ones it conflicted on and the
primary key. Decided per statement, not once: a column with a server default is
left out while its field is still zero, so a batch whose rows write different
columns overwrites different columns too.

The primary key is excluded because it says which row is being written rather than
what to write, the same rule `Update` follows.

### Insert and InsertMany return a count

```go
n, err := Users.With(db).OnConflict(Users.Email).DoNothing().InsertMany(ctx, users...)
// n is how many rows were actually written
```

This is the only report of a skip there is — nothing in a statement that wrote
nothing says which row it declined to write. `Insert` returns the same count, `1`
or `0`, for the same reason.

A shortfall is **not** an error here, unlike the one [`UpdateMany`
reports](#partial-writes-are-reported): rows that were already there is what an
upsert is for, and the count answers how many were new rather than signalling a
failure.

### One statement per row when reading values back

An upsert breaks the [correlation](#the-order-rows-come-back-in) a batched
`RETURNING` relies on: a row the database skipped returns nothing, so the *n*-th
row back is no longer the *n*-th row written. So an upsert with anything to read
back — a generated key, a server default — runs one row per statement, inside one
transaction. That is the price of learning each row's fate, and it buys exactly
that: after a batch, the rows that were new carry a key and the skipped ones do
not.

Where there is nothing to read back, the batch is one statement as usual, and the
driver's row count is what says how many were written.

### `BeforeCreate` and `AfterCreate` still run

For every row you gave, including one the database went on to skip. Which rows
those were is not something an upsert reports, so the alternative would be hooks
that fire or not depending on what a statement covering several rows happened to
do.

### What each database can express

`OnConflict()` with **no columns** means any conflict at all. PostgreSQL allows
that for `DO NOTHING` and only there — overwriting has to know which row it is
overwriting — so `DoUpdate` and `DoUpdateAll` reject it, and the error comes from
the dialect that said so.

A driver whose SQL cannot express an upsert returns an error naming the operation
rather than emitting something close. See
[Multi-driver design](/docs/4.architecture/1.multi-driver-design).

## `UpdateMany`

```go
for _, u := range users {
    u.Age++
}
n, err := Users.With(db).UpdateMany(ctx, users...)
```

Every writable column of each row is written to the row its primary key identifies,
exactly as `Update` does for one.

This runs **one statement per row**, inside one transaction. Rows carry different
values, and no statement portable across the databases Tork targets writes different
values to different rows. PostgreSQL could with an `UPDATE ... FROM (VALUES ...)`,
which every other driver would then need its own answer to; Ent refuses the
operation entirely for the same reason. Doing it as many statements and saying so is
the honest version.

If you are writing *one* set of values to many rows, that is `UpdateAll`, which is
one statement however many rows it matches.

## `DeleteMany`

```go
n, err := Users.With(db).DeleteMany(ctx, users...)
```

```sql
DELETE FROM "users" WHERE "id" IN ($1, $2, $3)
```

A composite primary key has no list form, so it becomes a comparison per row:

```sql
DELETE FROM "memberships"
    WHERE (("org_id" = $1 AND "user_id" = $2) OR ("org_id" = $3 AND "user_id" = $4))
```

Both are chunked under the same parameter ceiling. A table with no primary key has
nothing to identify its rows by, and says so:

```
orm: table "events": DeleteMany needs a primary key to say which rows it means,
    and the table declares none; use Where(...).DeleteAll instead
```

## `UpdateAll` and `DeleteAll`

```go
n, err := Users.With(db).
    Where(Users.Age.Lt(18)).
    UpdateAll(ctx, Users.Active.Set(false), Users.Note.SetNull())

n, err := Users.With(db).
    Where(Users.Email.IsNull(), Users.CreatedAt.Lt(cutoff)).
    DeleteAll(ctx)
```

```sql
UPDATE "users" SET "active" = $1, "note" = $2 WHERE "age" < $3
DELETE FROM "users" WHERE ("email" IS NULL AND "created_at" < $1)
```

Both return how many rows they changed. One statement, however many rows match.

Every column type carries `Set`, and the nullable ones also carry `SetPtr` and
`SetNull`, so what may be assigned is decided the same way what may be compared is,
at compile time:

```go
Users.Email.Set("new@example.com")
Users.Email.SetPtr(maybeNil)   // SQL NULL when maybeNil is nil
Users.Email.SetNull()

Users.Username.SetNull()       // compile error: not a nullable column
```

Matching no rows is not an error. Unlike `Delete`, which is given a row and can say
that row was not there, these are given a condition, and a condition matching
nothing is an ordinary answer.

### Getting the rows back instead of a count

```go
users, err := Users.With(db).
    Where(Users.Age.Lt(18)).
    UpdateAllReturning(ctx, Users.Active.Set(false))   // []*User

gone, err := Users.With(db).
    Where(Users.Draft.Eq(true)).
    DeleteAllReturning(ctx)                            // []*User
```

```sql
UPDATE "users" SET "active" = $1 WHERE "age" < $2
    RETURNING "id", "username", "email", "age"
```

The rows come back **as they are after the write**, which is what makes this worth
having over updating and then selecting: a column a trigger changed, a default
filled in, or another transaction had a hand in is readable without a second
statement and without the gap between the two.

For a delete it is the only way to see the rows at all. Selecting them first leaves
a window in which another transaction can change what the delete matches, so the
two lists need not be the same list — here they are the same statement. That makes
a returning delete a way to *claim* work:

```go
// two workers cannot both take the same task
batch, err := Tasks.With(db).Where(Tasks.State.Eq("queued")).DeleteAllReturning(ctx)
```

Every column comes back; a `Select` is not something a write takes. `AfterLoad` runs
on the returned rows, since a `*E` handed to you has been through it however it was
read — but no *write* hook fires, for the reason [below](#hooks).

A driver reporting no `RETURNING` support says so and points you at `UpdateAll` or
`DeleteAll`. PostgreSQL supports it.

### No `Where` means every row

```go
n, err := Users.With(db).DeleteAll(ctx)   // DELETE FROM "users"
```

That reads as what it does, so it is allowed. What is *not* allowed is a `Where`
that ended up contributing no condition:

```go
conds := buildFilters(req)                       // came back empty
Users.With(db).Where(conds...).DeleteAll(ctx)
```

```
orm: table "users": Where added no condition, so DeleteAll would delete every row;
    if that is what you meant, call DeleteAll without a Where
```

Calling `Where` and getting nothing back is a bug; not calling it is a choice. Only
the first is rejected, so there is no escape-hatch method to learn.

### No ordering or paging

```go
Users.With(db).Where(...).Limit(10).DeleteAll(ctx)   // error
```

SQL has no portable way to order or count off the rows a write touches. Dropping
the clause silently would run a statement over every matching row when you wrote one
that appeared to touch ten, so it is reported instead.

## Hooks

The `Many` operations fire the same hooks their single-row equivalents do, once per
row:

```go
func (p *Post) BeforeCreate(ctx context.Context) error {
    p.Slug = slugify(p.Title)
    return nil
}
```

Every `BeforeCreate` runs before any column list is worked out, because a hook that
fills in a field changes whether that field is still zero, and so whether its column
is written at all or left to the database.

`UpdateAll` and `DeleteAll` fire nothing. They are given a condition rather than a
row, so there is no value to call a method on.

`UpdateAllReturning` and `DeleteAllReturning` fire `AfterLoad` and nothing else.
They really do read rows, and a `*E` handed to you has been through `AfterLoad`
however it was read. A *write* hook still cannot fire: the row it would be called
on did not exist until the statement returning it had already run.

See [Reading Rows](/docs/2.fundamentals/7.reading-rows) and the hook interfaces in
the package documentation.

## Partial writes are reported

`UpdateMany` and `DeleteMany` return the rows they actually affected. When that is
fewer than the rows you gave them, they *also* return an error:

```
orm: table "users": UpdateMany updated 8 of 10 rows; the rest matched no row: orm: no rows
```

The error wraps `orm.ErrNoRows`, so `errors.Is(err, orm.ErrNoRows)` works, and the
count comes back either way:

```go
n, err := Users.With(db).UpdateMany(ctx, users...)
if errors.Is(err, orm.ErrNoRows) {
    log.Printf("only %d of %d rows were still there", n, len(users))
}
```

Single-row `Update` and `Delete` already return `ErrNoRows` for a row that was not
there. A batch quietly writing 8 of 10 would be the same situation with the
diagnostic removed.

## Everything is atomic

A bulk operation that takes more than one statement wraps itself in a transaction,
so a failure partway through leaves nothing behind. An operation that takes one
statement does not, since a statement is already atomic.

If you already opened a transaction, it joins yours rather than nesting:

```go
err := db.Transaction(ctx, func(tx *orm.DB) error {
    if err := Users.With(tx).InsertMany(ctx, users...); err != nil {
        return err
    }
    _, err := Posts.With(tx).Where(Posts.Draft.Eq(true)).DeleteAll(ctx)
    return err
})
```

See [Transactions](/docs/2.fundamentals/10.transactions).
