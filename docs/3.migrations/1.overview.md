# Migrations

A Tork migration is a plain `.sql` file with an `up` section and a `down`
section. You don't write it by hand (though you can edit it): you run
`makemigrations`, and Tork writes it for you by comparing your live database
against your Go models.

This is the same model SQLAlchemy and Alembic use, and it's different from
tools where migrations are hand-authored SQL, or where a `db.AutoMigrate(&User{})`
call mutates the schema directly with no file, no history, and no way to
review the diff before it runs.

## The three things being compared

1. **The live database.** Tork connects and reads the current schema
   (`information_schema`) for exactly the tables your models declare, nothing
   else in the database is touched or even looked at.
2. **Your models.** `User`, `Post`, whatever you pass to `makemigrations`, read
   through the reflection described in
   [Model Reflection](/docs/2.fundamentals/6.model-reflection).
3. **The diff.** Both sides get normalized into the same internal shape (the
   `schema` package's `Schema`/`Table`/`Column` types), so they can be compared
   structurally, not as SQL text.

The result of a diff is an ordered list of operations: create this table, add
that column, add this foreign key. Rendering them into real SQL is a job for a
[Dialect](/docs/4.architecture/1.multi-driver-design), PostgreSQL today.

## What gets detected

Table and column existence, column type changes, nullability changes, and
primary key, unique, and foreign key constraints. Constraints are matched by
their *column set*, not their name, so a constraint Postgres auto-named (like
`users_pkey`) is recognized as already satisfying your model, rather than being
proposed for a pointless rename.

## What doesn't get detected

Renaming a column (or a table) looks like a drop plus an add, there's no rename
detection. This is a known, deliberate limitation, and it's the same one every
other autogenerate-style tool documents: distinguishing "renamed" from
"dropped one thing and added an unrelated thing" from a snapshot in time is a
genuinely hard problem, not one Tork attempts to solve. If you rename a column,
either accept the drop and re-add, or hand-edit the generated migration to an
`ALTER TABLE ... RENAME COLUMN` instead.

## The revision chain

Migrations form a linked list, not a numbered sequence. Each file's header
names its own revision and its parent:

```
-- revision: 3321ee512566
-- down_revision: 2773adc3c811
```

The one revision that's nobody's parent is the *head*, the latest migration.
`migrate up head` walks forward from what's applied to the head; `migrate down
base` walks backward to nothing. This is covered in detail in
[Migration Files](/docs/3.migrations/2.migration-files) and
[Applying Migrations](/docs/3.migrations/4.applying-migrations).
