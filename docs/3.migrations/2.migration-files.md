# Migration Files

A migration file has four parts, in this exact order: a revision header, a
down-revision header, an `up` section, and a `down` section.

```sql
-- revision: 3321ee512566
-- down_revision: 2773adc3c811
-- migrate:up
ALTER TABLE "users" ADD COLUMN "bio" TEXT;

-- migrate:down
ALTER TABLE "users" DROP COLUMN "bio";
```

## The header

`-- revision:` and `-- down_revision:` are always the first two lines, in that
order, and both are always present. A migration with no parent (the first one
in a project) writes the literal value `none`, it's never omitted:

```
-- revision: 2773adc3c811
-- down_revision: none
```

This keeps the parser's job fixed: read exactly two header lines, then look for
the two section markers. A file missing either header line, missing a marker,
or with `-- migrate:down` appearing before `-- migrate:up`, fails to parse with
an error naming the problem, rather than being silently misread.

## Revision ids

A revision id is 12 lowercase hex characters, from 6 random bytes
(`crypto/rand`), the same width Alembic uses by default. Ids are not
sequential and carry no meaning beyond identifying one migration; ordering
comes entirely from the `down_revision` chain, not from the id or the
filename.

## The `up` and `down` sections

Everything between `-- migrate:up` and `-- migrate:down` is the up SQL;
everything after `-- migrate:down` is the down SQL. Each section is executed as
a single call, not split into individual statements, Tork relies on
PostgreSQL's simple query protocol accepting multiple `;`-separated statements
in one call, rather than writing a SQL statement splitter (genuinely hard to
get right around dollar-quoting and string literals containing `;`).

## Filenames

`makemigrations` names files `<revision>_<slug>.sql`, for example
`2773adc3c811_create_users_and_posts.sql`. The slug comes from the `-m` flag
(lowercased, non-alphanumeric characters collapsed to underscores, capped at 40
characters), or `auto` if you don't pass one. The filename is for humans
browsing the `migrations/` directory; nothing in Tork parses it for ordering.
Files are loaded sorted by filename, which is a stable, reproducible read
order, but not the *application* order, that's the `down_revision` chain,
covered next.

## The chain, and finding the head

Every migration but the first has a `down_revision` pointing at its parent.
Walking that chain from `none` gives you every migration in the order it
should apply. The one revision nobody points to as *their* parent is the
*head*, the tip of the chain:

```
none -> 2773adc3c811 -> 3321ee512566   (head: 3321ee512566)
```

If two files claim the same `down_revision` (a branch), or the chain loops
back on itself (a cycle), that's an error, both are out of scope for Tork to
resolve automatically today. In practice this only happens if migration files
are hand-edited incorrectly or two people generate a migration from the same
starting point without rebasing; keep `migrations/` in version control and
treat conflicts there the way you would any other merge conflict.
