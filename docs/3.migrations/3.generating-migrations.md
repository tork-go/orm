# Generating Migrations

```sh
go run ./cmd/migrate makemigrations -m add_users_bio
```

```
Wrote revision 3321ee512566
```

`-m` sets the filename slug (see
[Migration Files](/docs/3.migrations/2.migration-files)); it's optional. If
there's nothing to do, `makemigrations` prints `No changes detected` and writes
no file, the same behavior as Alembic.

## What happens, step by step

1. Connect, and read the current schema for exactly the tables your models
   declare (`Introspect`, scoped to those table names, never the whole
   database).
2. Read your models' desired schema (`schema.ExtractSchema`).
3. Diff current against desired for the up SQL, and desired against current
   (arguments swapped) for the down SQL.
4. Render both diffs into SQL for your dialect.
5. Find the current head of the revision chain (see
   [Migration Files](/docs/3.migrations/2.migration-files)), generate a new
   revision id, and write the file.

This is also available as a plain function, if you want to call it from your
own code instead of through the CLI:

```go
func MakeMigrations(ctx context.Context, dialect driver.Dialect, dsn, dir, message string, models ...orm.Model) (*migrate.Migration, error)
```

It returns `(nil, nil)` when there's nothing to do.

## The DDL Tork writes

Go type to SQL type, via each column's `T`:

| Go type | SQL type |
|---|---|
| `bool` | `BOOLEAN` |
| `int`, `int32` | `INTEGER` |
| `int64` | `BIGINT` |
| `float32` | `REAL` |
| `float64` | `DOUBLE PRECISION` |
| `string`, `MaxLen` set | `VARCHAR(n)` |
| `string`, no `MaxLen` | `TEXT` |
| `time.Time` | `TIMESTAMP WITHOUT TIME ZONE` |
| `uuid.UUID` (`github.com/google/uuid`) | `UUID` |

A single-column integer or bigint primary key is written as an
auto-incrementing identity column, inline in the `CREATE TABLE`:

```sql
"id" INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY
```

A composite primary key, or a single-column key on any other type, is a
separate table-level constraint instead:

```sql
CONSTRAINT "pk_order_items" PRIMARY KEY ("order_id", "product_id")
```

A column with `.ServerDefault(...)` set (see
[Columns](/docs/2.fundamentals/2.columns)) gets a `DEFAULT` clause, before
`NOT NULL` if both apply:

```sql
"id" UUID DEFAULT gen_random_uuid() NOT NULL
```

Constraint and index names follow a fixed convention, so they're predictable
and so running `makemigrations` again against a hand-built database doesn't
propose renaming everything to match: `pk_<table>` for primary keys,
`uq_<table>_<columns>` for unique constraints, `ix_<table>_<columns>` for
plain indexes, `fk_<table>_<columns>` for foreign keys.

## Order of operations

Within one migration, operations run in a fixed sequence: drop foreign keys,
drop unique constraints, drop indexes, drop primary keys, drop columns, drop
tables, create tables, add columns, alter column types, alter column
nullability, add primary keys, add indexes, add unique constraints, add
foreign keys.

Two consequences worth knowing about. First, dropping a table that something
else still references is safe without `CASCADE`, because the referencing
foreign key was already dropped in an earlier phase. Second, foreign keys
(and plain indexes) are never written inline in a `CREATE TABLE`, even for a
brand new table, they're always their own statement, foreign keys at the very
end, indexes just before them. That's what makes two new, mutually-referencing
tables work in a single migration: both tables exist (phase six) before either
foreign key is added (the last phase). For a plain index this isn't a design
choice, Postgres doesn't allow `CREATE INDEX` inside a `CREATE TABLE` at all.

## Validation before any SQL is written

Extracting your models' desired schema can fail, most commonly `MaxLen` used
somewhere it shouldn't be; see
[Constraints](/docs/2.fundamentals/4.constraints). These errors are reported
before introspection or diffing happens, with the table and column named.
