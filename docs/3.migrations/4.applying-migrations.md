# Applying Migrations

```sh
go run ./cmd/migrate migrate up head
go run ./cmd/migrate migrate down base
```

Both subcommands take a target as their next argument. There's no default;
you always say how far to go.

## Targets

| Target | Meaning | Valid for |
|---|---|---|
| `head` | Every pending migration. | `up` only |
| `base` | Roll back everything applied. | `down` only |
| `+N` | Apply the next `N` pending migrations. | `up` |
| `-N` | Roll back the last `N` applied migrations. | `down` |
| a revision id | Apply up to and including it, or roll back everything after it. | both |

```sh
go run ./cmd/migrate migrate up +2
go run ./cmd/migrate migrate down -1
go run ./cmd/migrate migrate up 2773adc3c811
```

`up` rejects `base`, and `down` rejects `head`, with an error, rather than
guessing what you meant.

## What "applying" actually does

Migrations run in chain order (following `down_revision` links, not filesystem
order), and each one runs in its own transaction: execute its SQL, record it in
the migrations history table, then commit. If a migration fails, everything
before it in that run stays committed, and the error names the first one that
failed:

```
migrate up: applying revision 3321ee512566: ERROR: column "bio" already exists
```

There's no automatic rollback of earlier, successfully-applied migrations in
the same run, that's what `migrate down` is for, applied deliberately. This
mirrors how DDL transactions work in PostgreSQL: a failed migration's own
changes are rolled back (they were never committed), but migrations that
already committed stay committed.

## The history table

Tork creates and maintains `tork_migrations` in your database the first time
you run `migrate up`, `migrate down`, or `history` (or call `migrate.Apply`):

```sql
CREATE TABLE IF NOT EXISTS tork_migrations (
    revision      TEXT PRIMARY KEY,
    down_revision TEXT,
    applied_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

It's created with plain, fixed DDL, not through the same diff engine your own
tables go through: it's a single permanent bootstrap table, not something
`makemigrations` would ever need to change.

## Checking status

```sh
go run ./cmd/migrate history
```

```
2773adc3c811  applied 2026-07-20 22:05:17
3321ee512566  pending
```

Every migration in chain order, each one's applied or pending status, and the
timestamp if applied.
