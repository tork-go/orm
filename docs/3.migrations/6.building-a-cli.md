# Building a CLI

There's no `tork` executable you install globally. Go can't import an
arbitrary package at runtime the way a scripting language can, so there's no
way for a prebuilt binary to discover your models on its own. Instead,
`migrate/cli` is a library: a small `main.go` in your own project imports your
models and this package, and compiling it gives you a private, fully-working
CLI.

```go
// cmd/migrate/main.go
package main

import (
	"os"

	"github.com/tork-go/orm/driver/postgres"
	"github.com/tork-go/orm/migrate/cli"

	"yourmodule/models"
)

func main() {
	os.Exit(cli.Run(os.Getenv("DATABASE_URL"), "migrations",
		models.Users, models.Posts))
}
```

```sh
go run ./cmd/migrate makemigrations -m add_users
go run ./cmd/migrate migrate up head
go run ./cmd/migrate history
```

Every subcommand (`makemigrations`, `migrate up`, `migrate down`, `history`)
is fully implemented in the library. Your `main.go` only supplies the four
things that are yours: the connection string, where migrations
live, and the models.

## `Run` and `RunWithArgs`

```go
func Run(dsn, migrationsDir string, models ...orm.Model) int
```

`Run` reads `os.Args[1:]` and writes to `os.Stdout`/`os.Stderr`, which is what
you want in a real `main.go`. It returns a process exit code, meant to be
passed straight to `os.Exit`.

```go
func RunWithArgs(args []string, out, errOut io.Writer, dsn, migrationsDir string, models ...orm.Model) int
```

`RunWithArgs` is the same dispatcher with its arguments and output streams
explicit. `Run` is a thin wrapper around it. Reach for `RunWithArgs` if you're
embedding the dispatcher somewhere that isn't a plain CLI invocation, for
example, driving it from a test, or from another tool that captures its
output instead of printing directly.

## Exit codes

`0` on success, `1` for a runtime error (connection failed, a migration
failed, an unknown revision target), `2` for a usage error (missing or
malformed arguments). If you're wiring this into a script, `2` is worth
distinguishing from `1`: it means the command itself was wrong, not that
anything touched the database.

## `migrationsDir` defaults to `"migrations"`

Passing an empty string for `migrationsDir` is the same as passing
`"migrations"`. Anything else is used as given, relative to the process's
working directory.
