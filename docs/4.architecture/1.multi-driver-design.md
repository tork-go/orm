# Multi-Driver Design

Tork targets PostgreSQL, SQLite, and MySQL; only PostgreSQL is built so far.
This page covers the boundary that's meant to keep adding a driver from
requiring changes anywhere else, in case you're implementing one, or just
want to know how the pieces fit together.

## The dependency direction

```
orm              no driver import, ever
schema           imports orm
driver           imports schema and orm; no driver client import
driver/postgres  imports driver, schema, pgx
migrate          imports schema, driver; no orm import
migrate/cli      imports migrate, driver, schema, orm
```

`driver` imports `orm` because the execution interfaces are declared there
and re-exported here as aliases. See below.

`orm` (your model definitions) never imports anything driver-specific, and
neither does `schema` (the diffable schema representation) or `migrate` (the
diff engine and runner). Only a `driver/*` package imports its database
client. Verify this for yourself against the real dependency graph:

```sh
go list -deps github.com/tork-go/orm | grep -i pgx
```

That prints nothing, `orm` has zero pgx dependency, or any other driver's,
by construction.

## Why a Tork-native `Conn`, not `database/sql`

The connection contract is deliberately narrower than Go's standard
`database/sql`:

```go
type Execer interface {
	Query(ctx context.Context, sql string, args ...any) (Rows, error)
	QueryRow(ctx context.Context, sql string, args ...any) Row
	Exec(ctx context.Context, sql string, args ...any) (Result, error)
}

type Tx interface {
	Execer
	Commit(ctx context.Context) error
	Rollback(ctx context.Context) error
}

type Conn interface {
	Execer
	Begin(ctx context.Context) (Tx, error)
	Close(ctx context.Context) error
}
```

`database/sql` would give broader driver compatibility for less code, since
nearly every Go database client has a `database/sql` adapter. Tork uses pgx
directly for PostgreSQL instead, specifically to keep pgx's native
capabilities available rather than routing everything through a
lowest-common-denominator interface. Each driver package adapts its own best
client to this interface; `driver/postgres`'s adapter is two small wrapper
types (`conn`, `tx`) that hold a `*pgx.Conn`/`pgx.Tx` and turn pgx's command
tag into a `Result`. `pgx.Rows` and `pgx.Row` already satisfy `Rows`/`Row`
directly, no wrapper needed.

`Exec` returns a `Result`, currently carrying only `RowsAffected`. It's a
struct rather than an interface, so a driver whose database reports a last
inserted id can gain a field later without breaking any caller. Postgres has
no need of one: it returns generated values through `RETURNING`.

## Where the execution interfaces live

`Row`, `Rows`, `Result`, `Execer`, `Tx`, and `Conn` are declared in `orm`,
and `driver` re-exports them:

```go
type (
	Result = orm.Result
	Row    = orm.Row
	Rows   = orm.Rows
	Execer = orm.Execer
	Tx     = orm.Tx
	Conn   = orm.Conn
)
```

They're aliases, not redeclarations, so a `driver.Conn` **is** an `orm.Conn`
and every reference to either name means the same type.

They live in `orm` because query building hangs off `Table[E]`, which is
there, and a query has to reach a connection to run. `orm` can't import
`driver` without closing a cycle through `schema`. Declaring a parallel set
in `orm` wouldn't help either: Go interface satisfaction needs identical
signatures, and a method returning `driver.Rows` doesn't satisfy one
returning `orm.Rows`, however alike they look. Named types from different
packages are different types.

This costs `orm` one stdlib import, `context`, and nothing else. The
invariant still holds and is still mechanically checkable:

```sh
go list -deps github.com/tork-go/orm | grep -i pgx
```

## `QueryDialect`: what a query compiler can't write for itself

Most of a `SELECT` is the same everywhere, so the compiler builds it once and
asks the dialect only about the parts that genuinely differ:

```go
type QueryDialect interface {
	QuoteIdent(name string) string
	Placeholder(n int) string
	RenderLike(quotedColumn, placeholder string, caseInsensitive bool) string
	SupportsReturning() bool
}
```

`driver.Dialect` embeds it, so a driver implements one interface and callers
have one to pass around.

The split is deliberate in both directions. Writing the whole statement in
the compiler would bake Postgres's `$1` and double quotes into the model
layer; handing the whole tree to each dialect would make every future driver
reimplement a `SELECT` compiler to change two tokens.

Postgres answers with `"quoted"`, `$1`, `LIKE`/`ILIKE`, and `true`. The fake
dialect used in tests answers with `[quoted]` and a repeated `?` on purpose,
so a compiler test can't pass by hard-coding Postgres's spelling, and it
reports no `RETURNING` support so that path stays reachable without a second
real driver.

## `Dialect`: the contract a driver implements

```go
type Dialect interface {
	Name() string
	Connect(ctx context.Context, dsn string) (Conn, error)
	Introspect(ctx context.Context, conn Conn, tables []string) (schema.Schema, error)
	EnsureHistoryTable(ctx context.Context, conn Conn) error
	InsertHistoryRow(ctx context.Context, exec Execer, revision, downRevision string) error
	DeleteHistoryRow(ctx context.Context, exec Execer, revision string) error
	AppliedRevisions(ctx context.Context, exec Execer) ([]AppliedRevision, error)

	RenderCreateTable(t schema.Table) ([]string, error)
	RenderDropTable(table string) []string
	RenderAddColumn(table string, col schema.Column) ([]string, error)
	RenderDropColumn(table, column string) []string
	RenderAlterColumnType(table string, col schema.Column) ([]string, error)
	RenderAlterColumnNullability(table, column string, notNull bool) []string
	RenderAddPrimaryKey(table string, pk schema.PrimaryKey) []string
	RenderDropPrimaryKey(table, name string) []string
	RenderAddUnique(table string, u schema.UniqueConstraint) []string
	RenderDropUnique(table, name string) []string
	RenderAddIndex(table string, idx schema.Index) []string
	RenderDropIndex(table, name string) []string
	RenderAddCheck(table string, c schema.Check) []string
	RenderDropCheck(table, name string) []string
	RenderAddForeignKey(table string, fk schema.ForeignKey) []string
	RenderDropForeignKey(table, name string) []string

	RenderCreateEnumType(e schema.EnumType) []string
	RenderDropEnumType(name string) []string
	RenderAddEnumValue(name, value, before, after string) []string
}
```

The `Render*` methods correspond one-to-one with the operation kinds
`migrate.Diff` produces (see
[Generating Migrations](/docs/3.migrations/3.generating-migrations)). Each
returns `[]string`, not a single string, on purpose: a future SQLite driver,
whose `ALTER TABLE` support is famously limited, might need to implement a
column drop or type change as several statements (create a new table, copy
data, drop the old one, rename), without the shared diff and generate code
needing to know or care how many statements that took.

`Execer` is a small interface satisfied by both `Conn` and `Tx` (they share
the same `Query`/`QueryRow`/`Exec` methods), so the history-row methods can
run inside the same transaction as a migration's own SQL, or standalone
against a plain connection for read-only calls like `history`.

## Adding a new driver

A new `driver/<name>` package implements `Dialect`, the same shape
`driver/postgres` does: connect using whatever client makes sense for that
database, adapt it to `Conn`/`Tx`/`Rows`/`Row`, implement introspection using
that database's own system catalog, and implement the `Render*` methods for
its SQL dialect. Nothing in `orm`, `schema`, `migrate`, or `migrate/cli`
changes.
