# The Postgres Driver

`driver/postgres` is the `driver.Dialect` implementation for PostgreSQL, built
on `github.com/jackc/pgx/v5`. This page covers what it does and, just as
importantly, what it deliberately doesn't handle yet.

```go
import "github.com/tork-go/orm/driver/postgres"

dialect := postgres.Dialect{}
```

## Introspection

`Introspect` reads `information_schema.columns`,
`information_schema.table_constraints`, and `information_schema.key_column_usage`
for columns, primary keys, unique constraints, and foreign keys, scoped to
exactly the table names you pass it, never the whole schema. That scoping
matters: if `Introspect` looked at every table in the database, a table that
belongs to some other system, or an extension, would show up as a proposed
`DROP TABLE` the moment you ran `makemigrations`. The tables it's scoped to
always come from the `TableName()` of the models you passed to
`makemigrations` or `Run`.

Plain (non-unique) indexes have no `information_schema` view at all, so
they're read from `pg_index` directly, joined against `pg_class` and
`pg_attribute` to recover each index's column list in order. Indexes that
already back a primary key or unique constraint are excluded here (they're
already captured on the `PrimaryKey`/`Uniques` side), and so are expression
and partial indexes, see Known limitations below.

## Type mapping

Both directions, `schema.Kind` to a Postgres type string when rendering DDL,
and an introspected `information_schema.columns.data_type` string back to a
`Kind` when reading the live schema:

| `Kind` | Postgres type |
|---|---|
| `KindBoolean` | `BOOLEAN` |
| `KindInteger` | `INTEGER` |
| `KindBigInteger` | `BIGINT` |
| `KindFloat` | `REAL` |
| `KindDouble` | `DOUBLE PRECISION` |
| `KindVarchar` | `VARCHAR(n)` |
| `KindText` | `TEXT` |
| `KindTimestamp` | `TIMESTAMP WITHOUT TIME ZONE` |
| `KindUUID` | `UUID` |

`TIMESTAMP WITHOUT TIME ZONE` is written explicitly rather than left to
Postgres's default, to avoid any ambiguity with `TIMESTAMPTZ`.

## Known limitations

**Composite foreign key introspection.** For a foreign key spanning more than
one column, `information_schema` doesn't reliably preserve which of the
referencing columns lines up with which referenced column; that needs
`pg_constraint`'s `conkey`/`confkey` arrays instead, which this driver doesn't
use yet. This is documented rather than silently wrong: Tork's own model API
(see [Foreign Keys](/docs/2.fundamentals/3.foreign-keys)) only ever produces
single-column foreign keys, so this gap only affects composite foreign keys
that already exist in a hand-built database you're introspecting.

**No non-identity single-column integer primary key.** A single-column
integer or bigint primary key is always rendered as an auto-incrementing
identity column; see
[Generating Migrations](/docs/3.migrations/3.generating-migrations). There's
currently no way to opt out of that per column.

**`ServerDefault` is rendered but not diffed or introspected.** A column's
`.ServerDefault(...)` (see [Columns](/docs/2.fundamentals/2.columns)) becomes
a `DEFAULT` clause when its column is first created or added. It is not
compared when diffing an existing column, and introspection doesn't read a
live column's default back at all (that would need `pg_attrdef`). If it were
compared, every `makemigrations` run against an already-migrated table would
propose the same "default changed" operation forever, since introspection
would always report an empty default. This means changing `ServerDefault` on
an existing column and running `makemigrations` again produces no operation
for it; hand-edit the migration if you need to change an existing default.

**No `USING` clause for incompatible type changes.** `ALTER COLUMN ... TYPE`
is rendered without a `USING` expression. Postgres allows this for compatible
type changes (widening an `INTEGER` to `BIGINT`, for example) but rejects it
for changes that need an explicit conversion.

**Expression and partial indexes are excluded from introspection.**
`schema.Index` has no way to represent an expression key (e.g. `(lower(name))`)
or a partial index's `WHERE` predicate, so both are left out entirely rather
than introspected incorrectly. A plain index Tork itself creates is never one
of these, so this only affects such indexes in a hand-built database you're
introspecting.

None of these are silent: each one either produces a clear Postgres error at
apply time, or is called out above as accepted, out-of-scope behavior.
