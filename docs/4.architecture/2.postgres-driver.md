# The Postgres Driver

`driver/postgres` is the `driver.Dialect` implementation for PostgreSQL, built
on `github.com/jackc/pgx/v5`. This page covers what it does and, just as
importantly, what it deliberately doesn't handle yet.

```go
import "github.com/tork-go/orm/driver/postgres"

dialect := postgres.Dialect{}
```

## Introspection

`Introspect` reads `information_schema.columns`,
`information_schema.table_constraints`, and `information_schema.key_column_usage`,
scoped to exactly the table names you pass it, never the whole schema. That
scoping matters: if `Introspect` looked at every table in the database, a
table that belongs to some other system, or an extension, would show up as a
proposed `DROP TABLE` the moment you ran `makemigrations`. The tables it's
scoped to always come from the `TableName()` of the models you passed to
`makemigrations` or `Run`.

## Type mapping

Both directions, `schema.Kind` to a Postgres type string when rendering DDL,
and an introspected `information_schema.columns.data_type` string back to a
`Kind` when reading the live schema:

| `Kind` | Postgres type |
|---|---|
| `KindBoolean` | `BOOLEAN` |
| `KindInteger` | `INTEGER` |
| `KindBigInteger` | `BIGINT` |
| `KindFloat` | `REAL` |
| `KindDouble` | `DOUBLE PRECISION` |
| `KindVarchar` | `VARCHAR(n)` |
| `KindText` | `TEXT` |
| `KindTimestamp` | `TIMESTAMP WITHOUT TIME ZONE` |

`TIMESTAMP WITHOUT TIME ZONE` is written explicitly rather than left to
Postgres's default, to avoid any ambiguity with `TIMESTAMPTZ`.

## Known limitations

**Composite foreign key introspection.** For a foreign key spanning more than
one column, `information_schema` doesn't reliably preserve which of the
referencing columns lines up with which referenced column; that needs
`pg_constraint`'s `conkey`/`confkey` arrays instead, which this driver doesn't
use yet. This is documented rather than silently wrong: Tork's own model API
(see [Foreign Keys](/docs/2.fundamentals/3.foreign-keys)) only ever produces
single-column foreign keys, so this gap only affects composite foreign keys
that already exist in a hand-built database you're introspecting.

**No non-identity single-column integer primary key.** A single-column
integer or bigint primary key is always rendered as an auto-incrementing
identity column; see
[Generating Migrations](/docs/3.migrations/3.generating-migrations). There's
currently no way to opt out of that per column.

**No `DEFAULT` support.** Adding a `NOT NULL` column to a table that already
has rows will fail in Postgres without a default value, since existing rows
would violate the constraint. Tork doesn't have a way to declare a column
default yet, so this is a real constraint you'll hit adding a required column
to a non-empty table; for now, either make the column nullable or handle the
backfill in a hand-edited migration.

**No `USING` clause for incompatible type changes.** `ALTER COLUMN ... TYPE`
is rendered without a `USING` expression. Postgres allows this for compatible
type changes (widening an `INTEGER` to `BIGINT`, for example) but rejects it
for changes that need an explicit conversion.

None of these are silent: each one either produces a clear Postgres error at
apply time, or is called out above as accepted, out-of-scope behavior.
