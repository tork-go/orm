# The schema language

Declaring a model in Go is explicit and type safe, and it is also
about thirty lines per table: a row struct, a model struct, and a
build closure, each naming the same columns again. The `.tork` schema
language says the same thing once.

```
// schema/blog.tork
datasource db {
	provider = "postgres"
}

model User {
	id        Int      @id @default(autoincrement())
	username  String   @unique @db.VarChar(30)
	email     String?  @unique
	createdAt DateTime @default(now())
	posts     Post[]   @relation("UserPosts")

	@@index([username])
}

model Post {
	id        Int      @id @default(autoincrement())
	title     String   @db.VarChar(100)
	authorId  Int
	author    User     @relation("UserPosts", fields: [authorId], references: [id], onDelete: Cascade)
	createdAt DateTime @default(now())

	@@index([authorId, createdAt], name: "idx_posts_author_date")
}
```

```
$ go run ./cmd/generate
Parsing schema (1 file)
Generated models/post.gen.go
Generated models/user.gen.go
Generated models/registry.gen.go
```

What comes out is the same Go you would have written, in the same
idiom, and it is ordinary code from that moment on:

```go
type User struct {
	ID        int       `db:"id"`
	Username  string    `db:"username"`
	Email     *string   `db:"email"`
	CreatedAt time.Time `db:"created_at"`

	Posts []Post
}

type UserModel struct {
	orm.Table[User]
	ID        *orm.IntColumn
	Username  *orm.StringColumn
	Email     *orm.NullableStringColumn
	CreatedAt *orm.TimeColumn
	Posts     orm.HasMany[Post]
}

var Users = orm.DefineTable[User]("users", func(t *orm.TableBuilder[User]) *UserModel {
	return &UserModel{
		Table:     t.Table(),
		ID:        t.Int("id").PrimaryKey(),
		Username:  t.String("username").Unique().NotNull().MaxLen(30),
		Email:     t.NullableString("email").Unique(),
		CreatedAt: t.Time("created_at").NotNull().ServerDefault("now()"),
	}
})
```

You query it the way you always have, `models.Users.With(db)`, and
migrations do not know the difference: they still diff your Go models
against the live database, exactly as described in
[Generating migrations](/docs/3.migrations/3.generating-migrations). The DSL
generates models; it does not replace anything downstream of them.

## What it is not

There is no `tork` binary to install and no configuration file. A
generator has to know where your packages live, and the program that
already knows that is your own `main.go`:

```go
// cmd/generate/main.go
package main

import (
	"os"

	gencli "github.com/tork-go/orm/gen/cli"
)

func main() {
	os.Exit(gencli.Run(gencli.Config{SchemaDir: "schema", OutDir: "models"}))
}
```

This is the same reasoning behind the migration CLI shipping as a
library rather than a binary; see
[Building a CLI](/docs/3.migrations/6.building-a-cli).

Migrations also still compare against the **live database**, never
against your `.tork` files. Renaming a column in the schema and
regenerating does not, on its own, tell the migration engine what
happened in the database; the diff figures that out by looking.
