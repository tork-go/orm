# Writing schemas

A schema file holds three kinds of block. Comments start with `//`,
and a comment directly above a declaration or field becomes its doc
comment in the generated Go.

## datasource

Exactly one per schema, across all its files. It names the database,
which is what makes the `@db` attributes meaningful.

```
datasource db {
	provider = "postgres"
}
```

`postgres` is the only provider today, matching the only driver.

## enum

An enum generates a database enum type and a set of string constants.

```
// Publication state of a post.
enum PostStatus {
	draft
	published
}
```

```go
// Publication state of a post.
const (
	PostStatusDraft     = "draft"
	PostStatusPublished = "published"
)
```

The constants are untyped on purpose: a row struct field holding an
enum column must be exactly `string`, and an untyped constant assigns
to one without a conversion. The database type is named after the
enum, `post_status` here, unless `@@map` says otherwise.

## model

A model generates a table. Its fields become columns, except those
whose type is another model, which become relations.

### Types

| Schema | Column | Go |
|---|---|---|
| `Boolean` | boolean | `bool` |
| `Int` | integer | `int` |
| `Int32` | integer | `int32` |
| `BigInt` | bigint | `int64` |
| `Float` | real | `float32` |
| `Double` | double precision | `float64` |
| `Decimal` | numeric | `decimal.Decimal` |
| `String` | text, or varchar with `@db.VarChar` | `string` |
| `DateTime` | timestamp | `time.Time` |
| `Uuid` | uuid | `uuid.UUID` |
| `Json` | jsonb, or json with `@db.Json` | the type named by `@go.type` |
| *an enum name* | that enum type | `string` |
| *a model name* | no column; a relation | the related rows |

Add `?` for a nullable column, which makes the Go field a pointer, and
`[]` for an array column. Both together, `String[]?`, gives a nullable
array and a `*[]string`.

```
tags   String[]
scores Int[]?
email  String?
```

Enums and `Json` cannot be arrays, because the ORM has no column type
for either. A slice inside a `Json` column is a different thing and
works fine: name a slice type with `@go.type`.

### Field attributes

| Attribute | What it does |
|---|---|
| `@id` | the primary key; use `@@id` for a composite one |
| `@unique` | a unique constraint on this column |
| `@index` | an index on this column |
| `@map("name")` | overrides the column name, otherwise the field name in snake case |
| `@default(...)` | see below |
| `@softDelete` | marks the `DateTime?` column recording a soft delete |
| `@relation(...)` | see [Relations](#relations) |
| `@db.VarChar(n)` | stores a `String` as `VARCHAR(n)` rather than `TEXT` |
| `@db.Text` | stores a `String` as `TEXT`, already the default |
| `@db.Numeric(p, s)` | sizes a `Decimal` |
| `@db.Json` / `@db.JsonB` | stores a `Json` column as `json` or `jsonb`, `jsonb` being the default |
| `@go.type("Name")` | names the Go type a `Json` column scans into; required on every `Json` field |

### Defaults

```
id        Int      @id @default(autoincrement())
createdAt DateTime @default(now())
status    Status   @default(draft)
active    Boolean  @default(true)
retries   Int      @default(3)
label     String   @default("none")
slug      String   @default(dbgenerated("md5(random()::text)"))
key       Uuid     @default(uuid())
nonce     String   @default(go("myapp/ids.NewNonce"))
```

They fall into three groups by what they generate:

- **Server side.** `now()`, a literal, and `dbgenerated("...")` become
  `ServerDefault`, so the database fills the column.
- **Client side.** `uuid()` and `go("import/path.Func")` become
  `GeneratedByClient`, so your Go code fills it before the insert. The
  function must be `func() T` for the field's type; your compiler is
  what checks that.
- **Neither.** `autoincrement()` generates nothing at all. A single
  integer primary key becomes `GENERATED ALWAYS AS IDENTITY` because
  of what it is, not because of an attribute; writing
  `@default(autoincrement())` only makes that visible in the schema.
  Leave it off and you get a warning saying so, since a reader should
  not have to know the rule to read the file.

`@go.type` and `go(...)` both take either a bare name, for something
declared in the generated package next to the generated files, or an
`import/path.Name` for anything else. The import is added for you.

### Block attributes

```
model Doc {
	title    String
	authorId Int
	price    Decimal @db.Numeric(10, 2)

	@@id([a, b])
	@@unique([title, authorId], name: "u_docs_title_author")
	@@index([authorId, createdAt], name: "idx_docs_author_date")
	@@index([title], where: "title IS NOT NULL")
	@@index(on: ["lower(title)"], name: "idx_docs_lower_title")
	@@check("price >= 0", name: "ck_docs_price")
	@@map("documents")
}
```

`where:` makes a partial index, `on:` an expression index. An
expression index must be named: the ORM derives an unnamed index's
name from its column list, and an expression index has none.

## Relations

Both sides of a relation are declared. The side holding the foreign
key says so with `fields:` and `references:`; the other side just
names the relation.

```
model User {
	id    Int    @id @default(autoincrement())
	posts Post[] @relation("UserPosts")
}

model Post {
	id       Int  @id @default(autoincrement())
	authorId Int
	author   User @relation("UserPosts", fields: [authorId], references: [id], onDelete: Cascade)
}
```

The name is optional when only one relation connects a pair of models,
and required when more than one does, or when a model relates to
itself. `onDelete:` and `onUpdate:` take `Cascade`, `Restrict`,
`NoAction`, `SetNull`, or `SetDefault`, and belong on the side that
holds the key. `map:` names the foreign key constraint.

A singular field on the inverse side is a has one instead of a has
many:

```
model Author  { profile Profile @relation("AuthorProfile") }
model Profile { authorId Int @unique
                author   Author @relation("AuthorProfile", fields: [authorId], references: [id]) }
```

A composite key lists several columns on both sides, and generates a
`ForeignKeys` method rather than riding on a column:

```
pair Pair? @relation("PairRef", fields: [pa, pb], references: [a, b], map: "fk_ref_pair")
```

### Many to many

Name the join model explicitly on both sides:

```
model Post  { tags Tag[]  @relation("PostTags", through: PostTag) }
model Tag   { posts Post[] @relation("PostTags", through: PostTag) }

model PostTag {
	postId Int
	tagId  Int
	post   Post @relation(fields: [postId], references: [id])
	tag    Tag  @relation(fields: [tagId], references: [id])

	@@id([postId, tagId])
}
```

There is no implicit form. Migrations diff the models you hand them,
so a join table with no model would be a table nothing knows about; an
explicit one is also a table you can query.

The join model's own `post` and `tag` relations need no inverse on
`Post` and `Tag`, because those two already reach across it.

## What the DSL leaves to Go

Some things are Go values, not schema text, and stay handwritten in the
same package alongside the generated files:

- [Derived tables](/docs/2.fundamentals/17.derived-tables), which are query time
  constructs over other tables rather than persistent schema.
- Custom JSON serialization through `Serialize`, which takes a pair of
  functions.
- `orm.NewColumn[T]`, which exists precisely to step outside the typed
  column surface the DSL covers.

Append them to your migration wiring by hand:

```go
cli.Run(dsn, "migrations", append(models.AllModels(), models.Special)...)
```
