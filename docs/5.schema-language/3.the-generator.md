# The generator

`gen/cli` is a library, like the migration CLI. A small `main.go` in
your own project names your directories and gives you the command:

```go
// cmd/generate/main.go
package main

import (
	"os"

	gencli "github.com/tork-go/orm/gen/cli"
)

func main() {
	os.Exit(gencli.Run(gencli.Config{
		SchemaDir: "schema", // the default
		OutDir:    "models", // the default
	}))
}
```

`Package` defaults to the last element of `OutDir`, so the common
project sets nothing at all.

## Commands

```
go run ./cmd/generate            read the schema and write the models
go run ./cmd/generate check      report problems without writing
go run ./cmd/generate fmt        rewrite the schema files canonically
go run ./cmd/generate fmt -l     name the files that are not canonical
```

`generate` is the default, so the bare command does the obvious thing.
`check` is what a CI step wants: it reports everything wrong and
writes nothing. It also runs generation into memory, because a few
problems only surface once the Go is written, such as two models whose
names collide once turned into Go identifiers.

Exit codes follow the migration CLI: `0` for success, `1` for a bad
schema or a filesystem refusal, `2` for a usage mistake.

## Many files

A schema is a **directory**, not a file. Every `.tork` file in
`SchemaDir` is part of one schema sharing one namespace, so a model in
one file refers to an enum or a model in another with no import and no
ceremony. Split a hundred model application however it reads best.

Output does not depend on that layout. Models, enums, and the registry
are all emitted in name order, so moving a model between files changes
nothing in the generated code and nothing in your diff.

## What gets generated

- `<model>.gen.go` per model, holding the row struct, the model
  struct, any constraint methods, and the `DefineTable` declaration.
- `enums.gen.go`, when the schema has enums.
- `registry.gen.go`, holding `AllModels`.

`AllModels` is what makes migration wiring one line:

```go
cli.Run(os.Getenv("DATABASE_URL"), "migrations", models.AllModels()...)
```

## Living beside handwritten code

Generated files end in `.gen.go` and open with a
`// Code generated by tork. DO NOT EDIT.` header. Everything else in
the output directory is yours, and the generator never touches it.

That matters for the `@go.type` and `go(...)` escape hatches, whose
targets you write by hand in the same package:

```go
// models/support.go, written by you
package models

type Meta struct {
	Kind  string `json:"kind"`
	Score int    `json:"score"`
}
```

```
meta Json @go.type("Meta")
```

When a model leaves the schema, its file is deleted on the next run,
so a deleted model stops being a Go type. Only files that both end in
`.gen.go` **and** carry the generated header are removed; a file of
yours that happens to be named like one is read, found not to carry
the header, and left alone.

## Formatting

`fmt` writes the canonical form: fields aligned into a table by name
and type, block attributes set off below the fields, comments kept
where they were, and blank lines you left between fields preserved as
paragraph breaks.

A file with a syntax error is reported and left exactly as it is.
Reprinting a tree the parser could only half understand would delete
whatever it could not read, which is a worse outcome than an
unformatted file. The same rule applies to format on save in the
editor.
